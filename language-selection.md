# 编程语言选型

> 工具选择的本质是认知负荷管理。选语言不是选"最好的"，是选"最适合当前思维模式的"。

---

## 一、选型逻辑

两个维度交叉，四个象限，四种主力语言恰好填满：

```
                编译型                    脚本型
         ┌──────────────────┬──────────────────┐
  静态   │     Rust         │    Nushell       │
  类型   │  系统/基础设施    │  结构化脚本/管道   │
         ├──────────────────┼──────────────────┤
  动态   │     Lisp         │    Python        │
  类型   │  嵌入式/DSL      │  数据/AI/胶水     │
         └──────────────────┴──────────────────┘
```

静态类型 + 脚本型是一个罕见组合——Nushell 和 [Gluon](https://gluon-lang.github.io/) 是少数同时占据这个象限的语言。大多数静态语言选择编译（Rust/C++/Go），大多数脚本语言选择动态（Python/Ruby/JS）。Nushell 的独特之处在于：**用静态类型保证管道数据的正确性，同时保持脚本语言的交互式体验**。

四种语言形成完美对称：每种语言独占一个象限，不重叠。

核心原则：**每种语言有明确的领地，不越界**。Rust 不写胶水，Python 不碰底层，Lisp 不做 I/O 密集，Nushell 不做复杂逻辑。

## 二、Rust：静态编译型

### 定位

系统语言。需要确定性、性能、内存安全的场景——基础设施、运行时、CLI 工具、嵌入式。

### 为什么是 Rust

- **零成本抽象**：泛型、trait、模式匹配——高级语言的表达力，C 的性能
- **所有权系统**：编译时消除数据竞争和内存错误，不需要 GC
- **生态系统**：Cargo + crates.io 是目前最活跃的系统语言生态
- **工具链**：rust-analyzer、clippy、rustfmt——开箱即用的工业化工具链

### 边界

常规叙事认为 Rust 的代价是"认知负荷"——借用检查器、生命周期标注、trait bound 带来显著的心智负担，然后用"编译器替你承担运行时风险"来合理化这个代价。

这个叙事颠倒了因果。**严谨是不可省的，类型系统是严谨的最低成本载体**。

中大型长期项目，不严谨则迅速腐化。除原型之外，严谨是必然要求。问题是：严谨但不要类型系统，是否可行？技术上可行——但代价更高：

- 相关约束存储在工作记忆中，心智负担更大，存不下时需要注释
- 注释问题更多（→ 详见 [为什么不应该写注释](why-not-write-comments.md)），实际维护成本远高于类型标注
- 不是类型，所以无法依赖编译器验证。对于常见的，分布于多处的，关联的逻辑，无法形成统一的上下文。

Rust 的借用检查器和生命周期标注常被视为"额外负担"。但数据流分析是**所有语言都需要做的事情**——区别只是显式还是隐式。

编写 Python 时同样需要考虑：
- dict 传入后是否被原地修改？
- 列表排序时是否产生了意外复制？
- 中间变量是否多余？
- 变量的作用域是否过长？
- 按变量的声明和使用对逻辑分组，使用挨着声明
本质上是数据流分析，是编写优雅简洁代码的基本原则，逻辑层面的低耦合高内聚

这些问题不会因为语言的动态特性而消失。区别仅在于：编译器检查变成了人脑检查——正确时无反馈，错误时无警告，直到运行时暴露。

Rust 的所有权系统将这些分析**显式化并交给编译器验证**。代价是编写时需要多一步思考，收益是思考完成后由编译器保证正确性。其他语言的代价是省略这一步思考，运行时再追溯，成本高出数个量级。

原型阶段可以暂不考虑，但项目进入维护期后终归要面对，然后发现最终结构与 Rust 的要求基本一致。区别在于：其他语言中这些分析由人脑完成，无法由机器验证；Rust 要求你在编写时完成，编译器在完成后验证。严格了一些，但不过分。

Python 近年的渐进式类型实践（mypy/pyright）也证明了同一个方向：**类型系统不是可选装饰，是工程必然**。区别只是 Rust 天生浑然一体，Python 后天补课。

→ 详细论证见 [Zig 批判与反智主义](go-zig-anti-intellectualism.md) §二

不该用 Rust 的场景：
- 快速原型（编译时间惩罚；但类型系统本身的成本在 AI 辅助下已趋近于零 → 详见 [原型/快速开发迷思](prototype-myth.md)）
- 纯胶水/数据处理（杀鸡用牛刀）
- 需要热重载的场景（编译型天然劣势）
- 团队认知预算有限（Rust 的学习曲线是团队成本）

## 三、Lisp（Elisp / Steel）：动态编译型

### 定位

嵌入式脚本语言 / DSL 构建。Elisp 是 Emacs 的灵魂，Steel 是 Helix 的未来（如果能合并的话）。

### 为什么是 Lisp

- **代码即数据**（homoiconicity）：宏系统是 Lisp 的杀手锏，其他语言的元编程都是在模仿 Lisp
- **REPL 驱动开发**：交互式探索，即时反馈，适合需要反复试错的场景
- **极简语法**：S-expression 消除了语法歧义——解析器天然精确
- **嵌入友好**：Steel 作为 Rust 嵌入式 Lisp，启动毫秒级，与宿主零摩擦互操作

### 边界

Lisp 的括号不是问题——**语法极简是优势**。真正的门槛是心智模型：你需要习惯"代码就是列表"的思维方式。

Elisp 的特殊性：它是 Emacs 的"操作系统 API"，不是通用编程语言。用 Elisp 写 Emacs 配置和用 Elisp 写业务逻辑是两回事。

→ 嵌入式 Lisp 选型详细分析见 [嵌入式脚本语言选型](embedded-script-languages.md)

## 四、Python：动态脚本型

### 定位

数据处理 / AI / 快速原型 / 胶水语言。不是用来写"正经系统"的，是用来写"把系统粘起来"的。

更重要的：**Python 是最适合 AI 写的语言**——语法简单、语义明确、AI 生成的代码准确率最高。短期/临时任务（脚本、数据处理、一次性自动化）用 Python，让 AI 直接生成，写完即扔，不值得用更"正经"的语言。

### 为什么是 Python

- **生态垄断**：AI/ML/Data 领域的事实标准，PyTorch/HuggingFace/pandas 无可替代
- **上手成本最低**：伪代码般的语法，非程序员也能读
- **降低项目门槛**：Rust 写核心逻辑（通过 PyO3 嵌入），Python 暴露接口——低技术水平开发者不需要懂 Rust 就能使用高性能库。Polars、Ruff 都是这个模式
  → 嵌入式脚本/运行时选型详见 [嵌入式脚本语言选型](embedded-script-languages.md)
- **胶水能力**：调用 C/Rust 库、处理 JSON、写自动化脚本——快得离谱
- **NumPy/Pandas**：向量化操作在数据处理场景下碾压一切

### 边界

Python 的问题不是语言本身，是**用它做了不该做的事**：
- Web 后端（Go/Rust 更合适）
- 系统工具（性能和可靠性都不够）
- 大型项目（动态类型的诅咒在规模增长时爆发）

Python 的正确用法：**短期、数据密集、生态依赖**。超过 6 个月生命周期的项目，认真考虑是否该用 Rust/Go。

## 五、Nushell：静态脚本型

### 定位

**Linux 日常操作和系统管理的最强 shell**。核心优势一句话：**结构化数据取代了"一切都是文本"**。传统 shell 的管道传输的是无类型文本流，Nushell 的管道传输的是有类型的结构化数据（表格、数值、日期、文件大小）。数据管道是结构化数据的自然延伸。

### 为什么是 Nushell

- **系统管理最强**：日常操作（文件管理、进程监控、日志分析、系统巡检）体验远超传统 shell
- **静态类型**：管道中每一步的数据类型在运行前可推断，类型错误在交互式环境中即时反馈
- **结构化数据**：`ls | where size > 1mb | sort-by modified`——每一步都是结构化表格，不是文本解析
- **自动补全极其强大**：自定义极其方便，远超 zsh（bash 没有可比性）
- **Rust 原生**：性能远超 bash/zsh，启动快，内存占用低
- **生态无所谓**：需要调用其他程序直接调就行——bash 有生态吗？影响它到处用吗

### 数据分析能力

有 Polars 插件，很多数据分析场景可以取代 Python——但存在限制

- **工程周边不足**：数据库驱动、格式处理、算法库/框架等远不如 Python 生态成熟。
- **分发摩擦**：Nushell 脚本分享给非 Nushell 用户时，对方缺乏运行环境和知识
- **教程资源/AI 熟练度不如 Python**：遇到问题时可参考的资料和 AI 能力都弱一档

### 边界

Nushell 的静态类型 + 脚本型组合是罕见的（类似 [Gluon](https://gluon-lang.github.io/)）。它的领地是**系统交互和日常操作**——这不是"适合"，是"最强"。

## 六、被淘汰的语言

### JavaScript / TypeScript：熟练但厌恶

熟练掌握不代表值得使用。

生态问题正在被"尽量集成"的策略系统的解决：

- npm 依赖地狱、版本冲突、供应链安全: Deno/Bun
- 前端工具链：Vite+

更重要的是**工程质量**：专业组织负责筛选高质量组件，或者直接通过底层重写同时改进性能——比如 Bun 用 Zig 重写了大量组件（数据库驱动、对象存储等）。

单线程也不是问题——默认单线程限制新手，老用户有的是方法绕过。

**真正的问题是语言本身**：类 C 古典风格，烦琐冗余，再加上成吨的 bad part。类型系统和工具链再进步，也修补不了语言设计层面的根本缺陷。


### Lua / Go：反智主义

两种语言的共同特征：**刻意降低复杂度，代价是牺牲表达力**。

> "If the programmer is trying to solve a complex problem, and the language forces them to ignore the complexity, the complexity will come out sideways."
> — [Zig: Small Minded Language Design](https://ziglang.org/documentation/master/#Small-Minded-Language-Design)

Zig 的批判精准命中了这两种语言的核心问题：它们用"简单"的名义回避了类型系统应有的严谨。结果是复杂度没有被消除，只是从编译期转移到了运行期——debug 时间、线上事故、团队沟通成本。**用起来简单了，但复杂度转移到了人身上。**

- **Lua**：没有模块系统、没有命名空间、全局变量默认——"简单"的代价是大型项目的可维护性灾难
- **Go**：没有泛型（直到 1.18）、没有枚举、没有错误链——"少即是多"的极端化，结果是大量样板代码

→ 详细论证见 [Zig 批判与反智主义](go-zig-anti-intellectualism.md) §二

### Scala / Haskell：语言优秀，环境不行

语言本身的设计水准极高：
- Scala：函数式 + OOP 融合、类型系统强大、模式匹配优雅
- Haskell：纯函数式、Monad 抽象、编译时保证

但工程不仅仅是语言设计：
- **工具链**：SBT（Scala）慢到令人发指，Cabal/Stack（Haskell）碎片化严重
- **生态**：库之间版本兼容性差，升级如同拆炸弹
- **学习曲线**： Monad transformer、Typeclass derivation——心智成本与实际收益不成比例
- **部署**：编译产物巨大，运行时依赖复杂

结论：**好语言 ≠ 好工程选择**。语言只是冰山一角，水面下的工具链/生态/部署才是决定性因素。

### MoonBit：有潜力，但在缝合

MoonBit 的初始设计很锐利——面向 WASM 的系统语言，语法简洁，编译快。但正在朝着"什么都想要"的方向演化：

- 标准 WASM 环境一直有问题（WASI 兼容性、GC 支持）
- 语言特性开始堆叠——每增加一个特性都在稀释原本的设计锐度

缝合怪的终局是：**每个领域都有更好的替代品**。保持锐利比扩展广度更重要。

## 七、核心结论

| 语言 | 象限 | 领地 | 核心价值 |
|------|------|------|---------|
| Rust | 静态 + 编译 | 系统/基础设施 | 确定性 + 性能 + 安全 |
| Nushell | 静态 + 脚本 | 数据管道/系统交互 | 静态类型 + 结构化数据 |
| Lisp | 动态 + 编译 | 嵌入式/DSL | 元编程 + 交互式 + 极简 |
| Python | 动态 + 脚本 | 数据/AI/胶水 | 生态垄断 + 上手成本最低 |

**选型原则**：
1. 每种语言有明确的领地，不越界
2. 领地由问题决定，不由个人偏好决定
3. "熟练"不是使用理由，"适合"才是
4. 语言只是冰山一角——工具链/生态/部署是水面下的决定性因素
