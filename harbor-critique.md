# Harbor：企业膨胀的典型案例

**结论**：拒绝。  
**状态**：表面光鲜，架构空洞。

## 执行摘要

Harbor 是 CNCF 毕业项目，是**"企业过度工程"**的典型代表。虽然它看起来功能丰富且符合行业标准，但其底层架构对现代自动化工作流存在根本性缺陷。它试图通过引入巨大的维护开销来解决治理问题。

## CNCF 毕业幻觉

Harbor 的"毕业"状态是企业政治的胜利，而非工程卓越。
1.  **企业背书**：作为 VMware（现 Cisco）项目，它受益于 TOC 的大量投票权。
2.  **勾选工程**：毕业标准偏向"完整性"（RBAC、审计、复制）而非代码质量、简洁性或架构优雅。
3.  **"安全"选择**：它是非技术管理层的默认答案，因为它提供"可否认的合理性"——如果失败，至少我们选择了 CNCF 标准。

## 架构反模式

### 1. 伪微服务（"伪装的单体"）

Harbor 声称是微服务，但实际上是跨多个容器拆分的紧耦合单体。
*   **共享状态**：Core、Registry、Jobservice 和 Portal 都依赖单个 PostgreSQL 实例和 Redis 集群。
*   **虚假解耦**：你无法独立于 Core 或数据库扩展 Registry。如果数据库锁住，整个 registry 停止。
*   **复杂性税**：你继承了分布式系统的运维噩梦（服务发现、服务间延迟、日志聚合），却没有实际收益。

### 2. 依赖地狱

为了存储简单的 blob 和 JSON manifest，Harbor 需要：
*   PostgreSQL（用于元数据）
*   Redis（用于缓存/会话）
*   Trivy/Clair（用于扫描）
*   Core / Registry / Jobservice（应用逻辑）

**对比**：**Zot** 是单个二进制文件，使用 S3 或文件系统存储，需要**零**外部依赖。

### 3. 功能膨胀 vs. 工程价值

Harbor 包含应该属于 CI/CD 管道而非 registry 的功能：
*   **漏洞扫描**：阻止推送进行扫描会拖慢开发者反馈循环。扫描应该是异步的，属于构建管道。
*   **Web UI**：在 GitOps 和 IaC 时代，管理镜像的 UI 是反模式。它鼓励手动干预。
*   **镜像复制**：应该由 ArgoCD/Flux 或 CI 工具处理。Harbor 的内部复制只是另一个需要管理的有状态复杂性。

## 安全错位

Harbor 依赖**应用级认证**（OIDC、LDAP、Token）来执行安全。
*   **现实**：如果你运行 Kubernetes 配合 Service Mesh 或 WireGuard（Zot+WG），你的网络边界已经安全。
*   **缺陷**：Harbor 添加了冗余的安全层，引入延迟、凭证管理开销和新的攻击面。
*   **讽刺**：Harbor 经常因为数据库或认证故障而离线，而非存储故障。"安全"特性成为可用性瓶颈。

## 范式冲突：传统认知 vs. 工程现实

Harbor 代表了"行业标准"与"第一性原理工程"之间的根本分歧。

| 维度 | 传统认知（Harbor 方式）| 工程现实（你的方式）|
|:---|:---|:---|
| **信任模型** | "CNCF 毕业" = 安全且经过验证。| "CNCF 毕业" = 政治共识与供应商锁定。|
| **复杂性** | "功能丰富" = 更好的产品。| "功能丰富" = 更多故障模式与攻击面。|
| **架构** | 微服务（10+ 容器、DB、Redis）。| 单体/简单（单二进制、FS/S3）。|
| **安全** | 应用级认证（OIDC/LDAP/Token）。| 网络级认证（WireGuard/零信任）。|
| **结果** | "巧克力味的屎"：光鲜 UI，空洞核心。| 丑小鸭（CLI）：健壮、快速、可靠。|

**陷阱**：Harbor 通过复杂性销售"安心"。你在每次升级、每次数据库迁移和每次"为什么 registry 又挂了？"事故中为此付费。

**真相**：正确使用（WireGuard）的简单工具（Zot）比试图成为操作系统的复杂工具（Harbor）提供更多可靠性和安全性。

## 结论

Harbor 是包装在云原生容器中的遗留心态。它优先考虑**控制**而非**简洁**，**治理**而非**速度**。

**首选替代方案**：**[Zot Registry](https://github.com/project-zot/zot)** + WireGuard。
