# Agent 复利：持久化如何改变 AI 工具的本质

> 2026-06-26 创建

**结论**：AI 工具的价值不在单次对话的质量，而在跨会话的累积。没有持久化的 AI 是打工；有持久化的 AI 是投资。


**没有资源的人，复利是唯一可主动追求的破局路径**。运气、时机、偶然的社交碰撞可以改变处境，但不可复制、不可计划，且不构成"破局"——"破"是主动行为，彩票中奖不是破局，是被砸中。复利是唯一把时间转化为非线性回报的数学机制——初始值小，只要积累速率恒定，终值可以超越短期爆发。这是可复制的路径，也是唯一可主动追求的破局路径。运气来了要接得住——而接住运气的能力，本身也是复利的产物。
## 复利的定义

金融复利：收益产生收益，指数增长。Agent 复利：每次交互产生的知识（记忆、技能、自动化）让下次交互更快、更准、更省 token。

**初始感知**：同模型（GPT），专用工具（如 Codex）初期质量高于 Hermes。差别在于——有些工具只有初期，永远是初期；有些能走到中期、后期。这是复利效应，爱因斯坦称之为"世界第八大奇迹"。初期追赶，中期超越，后期甩开。

## 复利的基础设施

| 机制 | 作用 | 复利路径 |
|:---|:---|:---|
| **Memory** | 跨会话持久化用户偏好和纠正 | 第 1 次纠正 → 永久生效 → 节省未来 N 次重复 |
| **Skills** | 复杂任务的结晶，可被调用 | 第 1 次探索 → 写成 skill → 下次跳过探索直接执行 |
| **Cron** | 定时自动化任务 | 第 1 次配置 → 永久零成本运行 |
| **Session Search** | 历史决策可检索复用 | 过去的判断 → 站在上面继续 → 不从零开始 |

## 与其他 Agent 的对比

| 维度 | ChatGPT | Claude Projects | Cursor | Devin | **Hermes** |
|:---|:---|:---|:---|:---|:---|
| 跨会话记忆 | ❌ 每次重来 | ⚠️ 有限 | ❌ | ❌ | ✅ memory 注入每轮 |
| 技能积累 | ❌ | ❌ | ❌ | ❌ | ✅ skill 库 |
| 自动化 | ❌ | ❌ | ❌ | ⚠️ 有限 | ✅ cron 持久运行 |
| 历史检索 | ❌ | ❌ | ❌ | ❌ | ✅ session search |
| 纠错闭环 | ❌ 下次重犯 | ❌ 下次重犯 | ❌ | ❌ | ✅ 纠正 → memory → 永不重犯 |

**本质区别**：其他 Agent 是**无状态函数**（输入 → 输出 → 忘记）。Hermes 是**有状态系统**（输入 → 输出 → 记忆 → 下次更快）。复利的前提是状态持久化。

**局限**：Hermes 的复利依赖用户主动纠正和维护 skill。如果用户不纠正，memory 不会自动变好。这是"人工复利"，不是"自动复利"。具体的交互技巧和审查习惯，见 [AI Agent 使用模式](agent-usage-patterns.md)。

实际上 Agent 可以自己发现模式并更新 skill——但人工审查仍然必要。逻辑是：**AI 权重中已有的知识不需要总结，AI 总结的是权重之外的知识**（参见 [agent-principles.md](./agent-principles.md) 中"反主流护城河"的原则：AI 已经从数万亿公共 token 中消化了主流知识）。权重之外的知识出错概率更高，必须审查。自动发现 + 人工审查 = 可靠的复利循环。

### 为什么不理解复利

现实中，"代码分叉维护地狱"（每个客户物理复制一份代码）和"许愿式编程"（迷信 AI 万能，一句话搞定一切）的人群高度重叠。他们的共同特征：**用短期便利替代长期投资**。

| | 代码分叉 | 许愿式编程 |
|:---|:---|:---|
| 短期做法 | 复制一份改，快 | 问 AI 一句话，快 |
| 长期成本 | 每个副本维护成本指数增长 | 每次对话从零开始，无积累 |
| 根本问题 | 不愿意做架构（多租户、feature flag） | 不愿意做工程（prompt 设计、skill 积累） |
| 迷信对象 | "这个客户特殊" | "下一个模型更强" |

两类人都在追求**外部变量**来解决**内部问题**：代码分叉者指望"换框架/雇更多人"，许愿式编程者指望"换 GPT-5"。"模型的智能"是外部变量，不可控、不可积累。memory/skill/cron 是内部变量，可控、可积累。追求外部变量的人天然不理解复利——因为复利的本质是**内部积累产生外部结果**。

更深的讽刺：许愿式编程的人用 AI 的方式，恰恰是 AI 最不擅长的——无状态、无积累、每次从零开始。他们把最需要复利的工具，用成了最没有复利的方式。

### 理论沉淀：复利的前置成本

复利的障碍不只是工具选择，还有认知层面的"未知的未知"（unknown unknowns）——未被识别的盲区，恰恰是最危险的地基缺陷。

**理论是实战经验的高维压缩**。新手与老手的实战差距由全职时间堆砌而成，无法线性压缩。纯体力劳动不具备复利效应，本质是打工。理论是成本最低的追赶路径：吸收他人数十年踩坑结晶，无需重复其物理过程。这与 LLM 的机制同构——LLM 从海量语料中压缩出高维模式匹配能力，新手从理论中压缩出领域判断力，均属于"压缩即智能"。

**"未知的未知"是地基中的定时炸弹**。场景中不存在的流程，直到被识别才进入视野——这是 unknown unknowns 的典型暴露过程。跳过理论沉淀直接行动，遭遇未识别盲区时返工成本呈指数级增长。理论沉淀并非迟缓，而是在排除地基中的隐患。半小时的认知突破，可能节省数日的试错。

**反馈循环的意义不在于敏捷，而在于累积**。缺乏理论骨架的勤劳生产，每次均从零开始——这是纯体力劳动的本质特征：无复利、无积累。理论是骨架，实战是血肉，二者不可偏废。纯理论流于空谈，纯实战沦为重复劳动；理论与实战的闭环，才是复利的正确形态。

**理论劳动被系统性低估**。社会风气将"动手"等同于"有价值"，理论工作因其不可见、不可量化、短期无产出，被逐出"劳动"范畴。"知行合一"的本意是理论与实践不可分割，实践中被扭曲为"行优先，知可延后"——实质是将认知劳动从劳动定义中剔除。但理论并非凭空冥想，从实践中提炼理论本身即为高强度认知劳动。缺乏理论支撑的行动，本质是随机试错，寄望于概率性的偶然成功。偶然成功的案例被包装为"实践出真知"，反向强化"动手即可"的叙事——幸存者偏差与反智叙事形成正反馈闭环。更深层的博弈在于：领先者默许乃至鼓励这种反智倾向，因信奉"动手即可"者众，愿意投入理论沉淀者寡，竞争者减少，领先者的护城河反而加深。


### 数据主权与复利的关系

复利有一个隐含前提：**积累的知识必须归用户所有**。否则复利只是为平台打工。

| 工具 | 数据位置 | 能否导出 | 切换成本 | 复利归属 |
|:---|:---|:---|:---|:---|
| **Hermes** | 本地（`~/.hermes/`） | ✅ 纯文本文件，随意迁移 | 零 | 用户 |
| **OpenClaw** | 本地 | ⚠️ 部分可导出 | 中等 | 用户（但不能自主进化，复利有限） |
| **WorkBuddy 类私有工具** | 云端 | ❌ 锁定在平台 | 极高（积累全部丢失） | 平台 |

**OpenClaw 的问题**：有本地存储，但不能自主进化——memory 是被动的，不会主动学习模式、生成 skill。有复利的基础设施（存储），但没有复利的引擎（进化）。相当于有银行账户但没有利息。

**私有工具的结构性矛盾**：

如果一个 AI 工具能自主进化（积累 memory、生成 skill），它就有动力**隐藏这些积累**，增加用户切换成本。因为用户的积累越多，离开的代价越高——这是经典的平台锁定策略。

```
能自主进化 + 数据在云端 = 平台锁定
不能自主进化 + 数据在本地 = 无复利
能自主进化 + 数据在本地 = Hermes（复利归用户）
```

Hermes 的设计选择：数据在本地（`~/.hermes/`），纯文本（markdown/json），零锁定。用户可以随时迁移到另一个 Agent——这意味着 Hermes 必须**持续提供价值**才能留住用户，而不是靠锁定。这对用户是健康的约束。

## 跨项目 vs 单一项目

### 跨项目联动：全局记忆是优势

当在 60+ 项目间工作时，很多知识是跨项目通用的：

- "NixOS 模块归属铁律" → 适用于所有 NixOS 模块
- "Git 操作铁律" → 适用于所有仓库
- "编辑纪律" → 适用于所有代码修改

全局 memory 一处定义，所有项目受益。另一个优势：**跨项目的因果链**——项目 A 发现的架构判断直接应用到项目 B。

### 单一项目：全局记忆有劣势

1. **Token 污染** — memory 注入每轮对话。改一个项目的配置文件时，其他项目的记忆全是噪声
2. **模式误用** — 全局铁律可能不适用于特定项目的正当例外
3. **上下文稀释** — 项目特定决策在 wiki 里有完整记录，但 memory 里只有摘要

**折中方案**：项目特定知识放 wiki（`projects/` 目录），全局知识放 memory。wiki 需要主动 search 才加载，不自动污染。

## 直线任务 vs 支线任务

**直线任务**（同一上下文顺序执行）：每一步都看到前一步的结果，上下文持续积累。这是**会话内复利**。

**支线任务**（delegate_task 并行子代理）：每个子代理拿到隔离上下文，看不到父对话的历史。复利断裂——用并行速度换取上下文积累。

| 维度 | 直线任务 | 支线任务 |
|:---|:---|:---|
| 会话上下文 | ✅ 持续积累 | ❌ 每个子代理从零开始 |
| Memory | ✅ 继承 | ✅ 继承 |
| 前序发现 | ✅ 自动可用 | ❌ 需显式传入 context |

**最优策略**：有依赖关系 → 直线执行（复利优先）；完全独立 → delegate 并行（速度优先）。

## 多项目并行与 KV Cache

并行两个 Hermes 会话（或 Codex 实例）对 KV cache **无帮助也无损害**——cache 是会话级的，不是项目级的。真正的 cache 收益来自会话内的多轮对话（前缀稳定），和是否并行无关。

## Agent 设计优先级

以 Codex 为代表的编码 Agent，设计优先级排序：

1. **正确率** — 第一次就写对。一次写对 vs 写三次，省下的成本远超任何 cache 收益
2. **速度** — 更快的模型、更好的 tool selection
3. **成本** — cache 是成本优化，不是质量优化

Cache 是**推理层的优化**，不是 **Agent 层的设计约束**。Agent 设计者不需要考虑 cache——那是推理引擎的事。类比：CPU 缓存对编译器透明，编译器不需要为了 L1 cache 优化代码结构。

## 转折点：Hermes 何时超越专用工具

初始阶段，Hermes 不如专用工具。专用工具为特定任务优化（Cursor 写代码、git CLI 操作仓库），零开销直接执行。Hermes 有 memory 注入、上下文加载、skill 匹配的初始成本。

**数学模型**：

```math
专用工具：T(n) = C_d        （每轮固定成本，线性）
Hermes：  T(n) = C₀ + K/e^(n/k)  （初始高，随累积衰减）
```

`C_d` = 专用工具每轮固定成本，`C₀` = Hermes 初始开销（memory 注入、上下文加载），`K` = 衰减基数，`k` = 积累速率，`n` = 交互次数。

**纯 token 成本维度，Hermes 永远不比专用工具便宜**——memory 注入是额外开销。转折不在 token，在**决策质量**：

| 阶段 | 专用工具 | Hermes | 谁赢 |
|:---|:---|:---|:---|
| 第 1-5 次 | 直接执行，零开销 | 加载 memory，可能误用 | 专用工具 |
| 第 10-20 次 | 同上，不变 | memory 积累，纠正生效 | 持平 |
| 第 50+ 次 | 同上，不变 | skills 已有，自动化已配，跨项目知识贯通 | Hermes |

**转折点取决于任务特性**：

- **高重复任务**（每天改 NixOS 配置）→ 转折点早（~10 次），memory 和 skill 快速积累
- **一次性任务**（写一个脚本）→ 永远不转折，专用工具更快
- **高关联任务**（跨项目架构决策）→ 转折点最早，跨项目知识贯通的复利最大

**但全局来看，没有真正的一次性任务。** 会拿工具做比较的人，本身就是长期用户。研究对比一个月、下载用一天、彻底删除——这种场景存在吗？企业场景中有接近的（CI 环境、客服助手），这些场景初期成本高、也不是非有长期记忆不可，但做好隔离后确实有益。

**工具有复利，应尽可能地用。** 切换工具本身就有成本，而且没有必要——AI 工具能力覆盖范围广，规划和调用不同工具是 AI 的强项，人工切换只是在增加延迟。即使浪费些 token，也比人工切换优惠。全链路排查的压倒性优势就是这点——AI 快速规划调用，人工切换成本很高。
