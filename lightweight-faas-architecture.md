# 自研轻量级 FaaS（脚本执行器）架构设计

## 背景与定位

### FaaS 的经济学本质：错峰与共享

FaaS 的本质是"共享经济"和"按需计费"。

**买断制（Deployment）**：相当于租了一辆车，不管你是在路上开（高并发），还是停在车库里睡觉（低频闲置），每天都要付固定租金。

**按需制（FaaS）**：相当于网约车。只有坐在车里、车辆行驶的那些分钟才付钱。下车后，这辆车立刻去接别的乘客（机器资源给别人用）。

FaaS 存在的前提是针对**低频、零碎、不好分类、突发性**的任务。核心价值是将常驻买断制的机器闲置资源释放给其他任务，实现按秒/按次计费，避免"白花花的银子浪费在开机干等上"。

### 混合型负载策略：Serverless 溢价陷阱

**混合型负载**：服务既有稳定的基础流量（Baseline），又偶尔伴随突发的高峰流量（Spike）。在这种场景下，盲目把所有业务都改成 FaaS，在经济和技术上都不划算：

1. **单价更贵（买断 vs 散买）**：云厂商算 FaaS 账单时，单位时间内 CPU 和内存的价格通常是普通虚拟机（如 ECS）的 2~3 倍。如果混合负载里有很大一部分是持续稳定的基础流量，用 FaaS 会导致月度账单比直接买断几台机器贵得多。

2. **K8s 原生 HPA 就能完美解决**：对于混合型负载，K8s 中最优雅的做法是：
   - 用标准 Deployment 顶住常驻基础流量（如始终保持 3 个 Pod 常驻，不至于闲置）
   - 配置 K8s HPA（水平自动扩容）或搭配 KEDA（基于事件驱动的 HPA 插件）
   - 突发流量来临时（Kafka 积压增加或 CPU 飙升），K8s 自动从 3 个 Pod 扩容到 30 个；流量过去后缩容回 3 个

**严禁 FaaS 的场景**：
- 业务流量持续稳定高频（如 WebSocket 长连接、高频 Kafka 消费）
- 原因：FaaS 算力单价通常是常驻虚拟机的 2~3 倍（溢价陷阱），且频繁扩缩容会导致 Kafka 严重的 Rebalance 灾难

**最佳实践：物理隔离**
- 常规 K8s Deployment 扛住常驻基础流量
- 自研轻量脚本执行器或 K8s 原生 CronJob 容纳不好分类的零碎边缘业务

### 技术选型终极结论

| 业务场景 | 负载特征 | 选型 | 理由 |
|----------|----------|------|------|
| FastAPI + WebSocket 主服务 | 持续常驻、长连接 | 100% Deployment | 长连接无法频繁扩缩容 |
| 定时清理、数据对账 | 低频、固定时间触发 | 100% K8s CronJob | 不需要全天干等，到点开机完工关机，不占额外资源 |
| 持续稳定的 Kafka 消费 | 持续高频、高吞吐数据流 | 100% Deployment 内置异步长连接 | 频繁扩缩容导致 Kafka Rebalance 灾难 |
| 零碎边缘脚本（不好分类） | 突发性、低频 | 自研轻量 FaaS | 按需计费，避免常驻闲置 |
| 极端事件驱动（如 AI 渲染） | 几小时不触发，触发时瞬间 500 并发 | 考虑 Knative | 仅当业务真正符合极端事件驱动时才引入 |

**彻底放弃 FaaS（Knative）的决策依据**：除非未来新增"用户可能几个小时都不点一次，但只要一点，就必须瞬间拉起 500 个并发容器去处理图片/AI 渲染"这种极端事件驱动场景，否则在当前混合业务形态下，引入 Knative 只会增加运维复杂度，不会带来任何经济或技术收益。

## 架构设计

### 整体拓扑（极简 Serverless 变种）

不引入 Knative/OpenFaaS 等重型框架，采用"Web 网关控制 + 容器运行时 + 云原生持久化 + 统一包装日志"的轻量级闭环：

```
[内部通道/VPN] ──> [网关层网格拦截] ──> [Web 管理后端 (/upload & /run)]
                                            │
                                            ▼ (低权限账号执行)
                                      [Wrapper 包装脚本]
                                            │
                              ┌─────────────┴─────────────┐
                              ▼                           ▼
                    [用户 Python/Shell]           [管道 awk 格式化]
                              │                           │
                              ▼ (异步)                    ▼
                    [S3 存储大文件]              [stdout ──> Loki/ELK]
```

## 异步执行引擎设计

### 传统 CGI 的致命缺陷：HTTP 断了，任务必死

在标准 Nginx/Caddy CGI 模式下，如果浏览器/网关与服务器的 HTTP 连接断开（前端网络抖动、Nginx `proxy_read_timeout` 超时），正在运行的脚本会被操作系统强行杀死（Kill）。

**底层原理**：
1. HTTP 连接断开 → Web 服务器（Nginx）感知到连接关闭
2. Nginx 向正在执行该脚本的子进程发送 `SIGPIPE` 或 `SIGTERM` 信号
3. 如果脚本没有特殊处理这个信号，进程立即退出，任务直接夭折

### 破局方案：将"同步阻塞"改造为"异步提交"

为了让长期任务不受 HTTP 连接影响，Web 后端接收到请求时，**绝对不能**在 HTTP 响应里同步等待脚本执行完，而是要采用"异步触发、立刻返回、后台运行"的设计。

**三个极简 API 动作**：

#### 1. 提交任务接口：`/run?script=task.py`

用户或网关发起请求，后端接收到后：
1. 在后台利用 Linux 的 `nohup` 或 `setsid` 派生（Fork）一个脱离 Web 容器生命周期的独立子进程
2. 将该进程的标准输出和错误重定向到特定的日志文件（如 `/pvc/logs/{task_id}.log`）
3. 立刻向前端返回一个任务 ID（比如用刚刚生成的系统进程 PID），然后结束 HTTP 请求
4. **效果**：此时 HTTP 连接已经安全关闭，但脚本在 Linux 后台安稳地跑着

#### 2. 状态观察接口：`/status?pid=12345`

前端或者其他服务如果想知道任务跑完没有，随时可以通过这个接口来查。后端只需要在 Linux 系统里执行一行极速命令：
- **核心命令**：`ps -p 12345` 或者检查 `/proc/12345` 目录是否存在
- **返回值**：
  - 如果进程存在 → 返回 `STATUS: RUNNING`
  - 如果进程不存在 → 说明跑完了（或者崩溃了）

#### 3. 日志观测接口：`/logs?pid=12345`

怎么知道进程跑得顺不顺利？由于包装脚本已经把日志重定向到了指定文件（例如 `/pvc/logs/12345.log`）：
- 后端可以直接读取这个日志文件的最后几行（模拟 `tail -n 100`）返回给前端
- 配合 Wrapper 设计的 `|PROCESS_END|STATUS:SUCCESS|` 结束标志，一旦在日志末尾读到这一行，说明任务成功结束；如果读到 `FAILED` 则是失败退出

### 核心落地的"两行 Shell 代码"

管理后端在接收到 `/run` 请求时，在底层可以用 Python 或 Go 执行下面这行经过特殊设计的 Shell 命令：

```bash
# 异步启动包装器，脱离父进程（Web 容器）控制，将日志写到以 PID 命名的文件里
nohup ./run_wrapper.sh ./scripts/task.py "HTTP_TRIGGER" > /pvc/logs/$$_task.log 2>&1 &
```

**原理解析**：
- `nohup ... &`：让进程在后台运行，并忽略 `SIGHUP` 信号。即使 FastAPI/Nginx 容器重启了，这个子进程都不会死
- `$$_task.log`：利用当前触发时的临时进程号命名日志文件，方便后续通过 `/logs?pid=xxxx` 精准定位查看

## 日志流架构：双流模式与文件级隔离

### 彻底解耦的"双流模式"

一旦把"同步阻塞"改成"异步后台文件重定向"，整个日志流的处理难度瞬间下降了 95%。整个系统的输出被完美切分成两个互不干扰的通道：

**通道 A：HTTP 触发连接（瞬时连接）**
- 这个连接只存在几毫秒。它的输出极其单一，就是一串极简的 JSON（例如 `{"task_id": 12345, "status": "submitted"}`）
- **结果**：这里面没有任何业务日志，所以根本不需要任何过滤和清洗

**通道 B：后台独立进程（文件持久化连接）**
- 脚本在后台运行时，它的 stdout 和 stderr 被 Linux 内核通过 `>` 直接死死地绑定到了外部存储（PVC）的特定日志文件里（例如 `/pvc/logs/12345.log`）
- **结果**：脚本输出什么，文件里就存什么

### 为什么不需要"专门过滤日志输出"了？

在传统的同步 CGI 或传统的容器日志（Docker logs）中，需要疯狂做过滤，是因为所有的脚本、所有的并发请求、甚至是 Web 容器自身的错误日志，全部混杂在同一个 stdout 数据流里，必须靠正则把它们捞出来。

而现在的异步设计带来了一个天然的优势：**文件级隔离（One File Per Job）**。

因为每一个任务都有一个独一无二的 `task_id.log` 专属文件，这个文件里 100% 全是该脚本自己的亲生日志。它再也不会和核心微服务的 WebSocket 日志混在一起，也不会和其他并发运行的脚本日志打架。

既然已经是纯净的单任务日志，Wrapper 设计的 `|info/warn|` 包装就从"被迫的数据清洗"变成了"优雅的增值体验"：

1. **用户看原始日志**：当用户通过网页查看 `/logs?id=12345` 时，后端直接 `cat` 这个文件的原始内容丢给前端，原汁原味，不需要任何过滤
2. **Loki/ELK 做系统审计（可选）**：如果运维想要监控"今天有多少任务失败了"，日志收集器直接扫描 `/pvc/logs/*.log`，因为格式统一，很容易做自动化结构化提取

### 极简的"终局实现代码"结构

在这样的架构下，主 API 服务（FastAPI/Go）触发脚本的代码，甚至精简到只需要一两行：

```python
# FastAPI 伪代码：真正的输出已经跟 HTTP 毫无关系
import subprocess

@app.post("/run")
async def run_script(script_name: str):
    task_id = generate_unique_id()  # 比如生成一个 UUID 或时间戳
    log_file_path = f"/pvc/logs/{task_id}.log"
    
    # 打开日志文件（追加模式）
    log_file = open(log_file_path, "a")
    
    # 核心：使用包装器运行，并把 stdout/stderr 全权交给日志文件
    subprocess.Popen(
        ["./run_wrapper.sh", f"./scripts/{script_name}", "HTTP_API"],
        stdout=log_file,
        stderr=subprocess.STDOUT,
        start_new_session=True  # 这一步保证它脱离父进程，HTTP 断了也继续跑
    )
    
    # 立刻返回，不需要等待脚本执行
    return {"status": "running", "task_id": task_id, "log_path": log_file_path}
```

**总结**：这就叫"用架构解决代码问题"。通过将同步改为异步、将控制流（HTTP）与数据流（日志文件）物理分离，直接从根源上消灭了复杂的"并发日志清洗与过滤"逻辑。整个系统变得极其皮实（Robust），维护成本低到可以忽略不计。

## 生产级实现：Nushell + Linux 原生 tail --pid

### 核心亮点

基于 Linux 原生 `tail --pid` 机制的 Nushell 完整版轻量级 FaaS 引擎脚本。通过引入 `--pid` 参数，直接干掉了之前啰嗦的行数计算逻辑，改由 Linux 内核事件驱动。整个脚本不仅代码量骤减，而且具备极高的生产级性能与稳定性。

### 完整代码：faas_engine.nu

```nushell
# faas_engine.nu
# 基于 Nushell + Linux 原生 tail --pid 打造的工业级轻量 FaaS 引擎
# 全局共享的持久卷日志目录（建议挂载 K8s 共享 PVC）
const LOG_DIR = "/pvc/logs"

# =========================================================================
# 1. 异步提交任务 (Submit Task)
# =========================================================================
# 启动一个真正脱离父进程生命周期的后台任务。
# 成功后会将 Linux 系统的物理 PID 作为 task_id 返回，以便后续探针和 tail 精准追踪。
export def run-task [
    script_path: string  # 用户脚本的路径 (例如: ./scripts/task.py)
    trigger_src: string  # 触发来源审计标签
] {
    # 确保日志目录存在
    mkdir $LOG_DIR
    
    # 临时生成一个随机且唯一的日志标记，用于在派生瞬间精准捕捉 PID
    let tmp_tag = (random uuid)
    let log_path_tmp = $"($LOG_DIR)/($tmp_tag).log"
    
    # 执行底层派生：使用 setsid 将 Python 脚本完全托管给系统孤儿院，HTTP 断连、容器重启绝不会影响它
    # 使用 -u 强制 Python 开启无缓冲区输出，保证日志实时写入物理文件
    bash -c $"setsid python3 -u '($script_path)' >> '($log_path_tmp)' 2>&1 &"
    
    # 极速等待 10 毫秒让系统完成进程树派生，然后去系统进程表捞出刚才那个带特殊标记的进程 PID
    sleep 10ms
    let real_pid = (ps | where cmd =~ $tmp_tag | get pid | first | into string)
    
    # 得到真实的物理 PID 后，将日志文件名重命名为以 PID 命名的标准格式，方便后续解耦维护
    let final_log_path = $"($LOG_DIR)/($real_pid).log"
    mv $log_path_tmp $final_log_path
    
    # 向 Web 后端（FastAPI/Go）返回高内聚的结构化数据
    return {
        status: "submitted"
        task_id: $real_pid  # 物理 PID 直接作为全局唯一任务 ID
        log_path: $final_log_path
    }
}

# =========================================================================
# 2. 流式日志输出与自毁控制 (Stream Logs & Auto-Exit)
# =========================================================================
# 核心亮点：一行原生命令完美兼容以下两种核心业务场景：
# - 场景 A (任务运行中)：持续阻塞 HTTP 连接，呈现高流畅度的 `tail -f` 实时滚动体验。
# - 场景 B (任务已结束)：不发生任何无谓阻塞，瞬间一口气吐完历史日志，自动顺畅断开 HTTP（等同于全量下载）。
export def stream-logs [
    task_id: string  # 传入由 run-task 返回的物理 PID 字符串
] {
    let log_path = $"($LOG_DIR)/($task_id).log"
    
    # 安全边界防线：若日志文件不存在，直接抛错拦截
    if not (path exists $log_path) {
        print $"Error: Log file for task ($task_id) not found."
        return
    }
    
    # 【降维打击的核心核心】
    # -f : 开启实时追加监听
    # -n +1 : 从文件第一行（第 1 行）起开始输出。若任务已结束，此参数保证完整历史被全量吐出
    # --pid : 操作系统探针。tail 内部会紧盯这个 PID。一旦该进程销毁，tail 将在毫秒级内自动自毁退出！
    tail -f -n +1 --pid $task_id $log_path
}
```

### 架构运行轨迹全景剖析

这套方案之所以在架构上极其性感，是因为它把控制流和数据流彻底在 Linux 内核层面闭环了：

#### 场景一：用户发起 `/run`（提交任务）

1. FastAPI 收到请求，调用 `nu -c "use faas_engine.nu; faas_engine run-task './script.py'"`
2. Nushell 完美派生后台进程，立刻返回 `{"task_id": "45210"}`。HTTP 连接瞬间关闭
3. 此时，PID 为 45210 的 Python 脚本正在后台疯狂往 `/pvc/logs/45210.log` 写入数据

#### 场景二：用户刷新前端，发起 `/stream-logs/45210`（实时看日志）

1. FastAPI 启动 `StreamingResponse` 挂载 `nu -c "use faas_engine.nu; faas_engine stream-logs '45210'"`
2. 此时 `tail` 启动。由于使用了 `-n +1`，它会用极高的硬件吞吐量，瞬间把文件里已经写入的几千行历史日志全部通过 HTTP 冲刷到前端浏览器上
3. 吐完历史后，由于 `--pid 45210` 发现这个进程依然在活跃（Running），`tail` 自动切入内核级事件等待模式（inotify）。只要 Python 打印一行，前端网页就实时滚动一行

#### 场景三：后台 Python 脚本运行结束（自毁触发）

1. 后台 Python 脚本执行完毕，进程 45210 被 Linux 内核正常销毁
2. 常驻的 `tail` 进程通过系统调用瞬间捕捉到 45210 消失了，它会顺手把缓冲区里最后两行残留的日志吐给网关，然后自动向外抛出退出信号，整个 `tail` 进程安全自毁
3. 随着 `tail` 进程的死亡，FastAPI 的 `StreamingResponse` 迭代器读到了 EOF，HTTP 流式连接被极其顺畅地关闭。整个过程前端没有经历任何卡顿或多余的轮询延迟

#### 场景四：任务已经完工很久了，用户再次点进来查看

1. 前端重新发起 `/stream-logs/45210`
2. `tail` 启动，依旧通过 `-n +1` 疯狂把文件里的历史日志一口气通过 HTTP 吐给用户
3. 吐完最后一行的一瞬间，`--pid 45210` 检查发现这个进程在系统里早就没有了，`tail` 在不到一毫秒的时间内果断选择自毁退出
4. **最终体验**：用户点击的瞬间，浏览器直接下载完了一个完整的静态文本，HTTP 连接秒级关闭。代码没有任何分支判断，仅凭 Linux 原生特性就完成了完美闭环

**这就是现代云原生高级运维的魅力所在——能用系统底层解决的，绝对不要往应用层多写一行代码。**

## 关键决策点

### 1. 权限与安全

**边界防御**：
- 网关层（Envoy/Nginx/Caddy）配置黑名单，屏蔽外部对 `/upload` 和管理接口的访问
- 仅允许集群内部 Pod 通信或 VPN 授权通道接入

**爆炸半径控制**：
- Dockerfile 禁用 root 用户
- Web 容器与 CGI 脚本一律使用低权限账号（`nobody` 或 `www-data`）
- 效果：即使脚本存在命令注入漏洞或恶意代码（如 `rm -rf /`），由于权限受限，无法破坏容器根系统，更无法逃逸到 K8s 宿主机

### 2. 数据存储策略

**脚本代码层：强烈推荐 PVC**
- 动态上传的 `.py` 或 `.sh` 脚本存放在标准 K8s 持久卷（阿里云 NAS、AWS EFS 或 Ceph）
- 优势：延迟极低，完全支持并发读写与文件锁

**执行结果层：规避 s3fs 坑**
- **禁止**使用 s3fs 将对象存储挂载为本地硬盘执行代码
- 原因：s3fs 基于 FUSE 网络请求，脚本执行时产生大量 `getattr` 导致几十倍延迟，并发写入极易文件损坏

**正确做法**：
- 脚本代码读写走 PVC
- 脚本运行产生的最终大文件结果（导出的 Excel、压缩包），在代码中调用 `boto3` 接口直接上传到 S3/OSS

### 3. 可观测性（Wrapper 统一包装日志）

**痛点**：
- 传统 CGI 脚本难以监控进程状态
- 输出日志杂乱，无法被中间件精准切分

**设计方案**：
编写通用 `run_wrapper.sh` 包装器，利用 Linux 管道和 `awk` 拦截用户脚本的标准输出（stdout），动态重定向并补全为工业级标准日志格式：

```
|时间|日志级别|业务内容|PID|脚本名|触发来源|
```

**进程审计**：
- 包装器进入和退出时，自动打点 `PROCESS_START` 和 `PROCESS_END` 日志
- 记录退出码（Exit Code）

**效果**：
- 对业务脚本零侵入（写脚本的人正常 `print` 即可）
- K8s 日志采集器（Filebeat/Fluent Bit/Loki）通过简单正则完美切分日志
- Grafana 直接做脚本运行大屏

## 技术栈选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 网关 | Envoy/Nginx/Caddy | 支持黑名单配置，网格拦截 |
| 持久化 | PVC（NAS/EFS/Ceph） | POSIX 协议，低延迟，支持并发 |
| 对象存储 | S3/OSS（boto3 直接调用） | 避免 s3fs FUSE 性能陷阱 |
| 日志采集 | Filebeat/Fluent Bit/Loki | 支持正则切分，Grafana 可视化 |
| 脚本执行 | Python/Shell + Wrapper | 零侵入，统一日志格式 |

## 避坑清单

1. **不要全盘 Serverless 化**：高频常驻流量用 Deployment，零碎边缘任务用 FaaS
2. **不要用 s3fs 挂载执行代码**：FUSE 延迟高，并发写入易损坏
3. **不要 root 执行脚本**：低权限账号限制爆炸半径
4. **不要直接暴露管理接口**：网关层黑名单 + 内部通道访问
5. **不要裸跑脚本日志**：Wrapper 统一包装，便于采集和审计
