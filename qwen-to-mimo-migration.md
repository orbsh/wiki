## 一、我们为什么写这份备忘录？

团队近期在内部开发心流以及线上生产环境中，深度使用了以 Qwen 为代表的大厂"强推理（Reasoning）"模型。经过真实深水区的工程验证，我们发现纸面榜单上的高智商，在面对高频、多轮工具交互的厚 Harness 场景以及高并发、低延迟的面向客户场景时，引发了系统性问题。

团队已达成共识：**全面从 Qwen 迁移至 MiMo 2.5（标准版）**。

## 二、大模型的本质：高视野、低智力

在讨论具体模型之前，需要先对齐一个认知：**大模型是概率预测引擎，不是因果推理器**。

- **高视野**：读过全世界几乎所有的开源代码和 API 文档，知识库宽广无边
- **低智力**：缺乏真正的因果推理能力。面对跑了五年、充满陈年屎山和独特业务潜规则的特定代码库时，会因为无法理解"人类独特的业务历史包袱"而变得无用

这意味着大模型不适合做架构规划，原因有三：

1. **隐藏上下文不可传递** — 公司预算、团队技术栈熟练度、业务演进方向、部门政治博弈，不可能写进 prompt
2. **缺乏全局抽象能力** — AI 看代码是"局部微调"，人类架构师看代码是"立体拓扑"
3. **责任隔离** — 架构规划要对线上崩溃负法律责任，AI 不承担后果，方案必然马虎

**现实编码任务的真相**：大厂 80% 的日常 coding 是体力活——加字段、接接口、写 CRUD、配路由、改 CSS。不需要极高智商，人类架构师指明方向，剩下的是高重复性机械对齐。这正是 MiMo 的最佳战场。

## 三、Qwen 在生产中的问题

### 3.1 根源：外部思维链的工程代价

大厂模型为在公开榜单（MMLU、Math 等）上刷分，将指令遵循、类型安全和逻辑能力深度绑定在外部长思维链（Thinking Process）中。这导致模型在厚 Harness 场景下产生两个极端问题。

### 3.2 问题一：开着思考——响应延迟不可接受

**开发端（Zed/Hermes）**：在厚 Harness 场景下，Qwen 启动庞大的思考路由。哪怕只是 inline 补全，也要在后台推导、碎碎念很长时间。思维 Token 在多轮循环中被重新 Prefill，导致单日 Token 账单飙升。

**面向客户的生产端**：客户发来简单咨询时，模型响应极其缓慢，首字延迟（TTFT）过高，用户体验直接死亡。

### 3.3 问题二：关闭思考——指令遵循退化

关闭思考后的 Qwen，逻辑能力大幅下降。因为它的推理能力部分依赖思维链，强行砍掉思考窗口后，不仅看不懂厚 Harness 的 System Prompt，甚至在处理基础业务代码时产生低级幻觉——在实际面向客户的业务中，连简单的 API URL 拼接都无法正确完成，引发线上路由走失的生产事故。

### 3.4 核心矛盾

| 模式 | 问题 | 影响 |
|------|------|------|
| **Thinking On** | 首字延迟极高，Token 消耗巨大 | 开发心流断裂，成本失控 |
| **Thinking Off** | 指令遵循退化，格式混乱 | 生产事故，用户体验死亡 |

## 四、我们为什么选择 MiMo 2.5？

在工业级老项目的重构、接口适配、ADR 落盘这类"高视野、低智能"的任务中，我们不需要 AI 天天演内心戏搞发明创造。我们只需要它：**快、稳、听话、便宜、不断流**。

### 4.1 MiMo 2.5 的核心优势

| 维度 | MiMo 2.5 | Qwen Plus |
|------|----------|-----------|
| **思考模式** | 窄思维链（Narrow CoT）——约束在工具链拆解和边界检查，直接输出 | 依赖外部思维链，关闭后退化严重 |
| **Prefill 计算量** | 1/7（Hybrid SWA 架构） | 全量计算 |
| **流式输出** | 无停顿，全速流式喷射 | 经常断流 |
| **格式精准度** | 原生高精度 | 关思考后格式混乱 |
| **成本** | 极低（480 行 Markdown ≈ 0.2 元） | 高（300-400 元/天） |
| **首字延迟** | 毫秒级 | 分钟级 |
| **谄媚行为** | 无。训练目标是任务完成效率，不是用户满意度。不产出寒暄/铺垫/emoji，token 预算全花在干活上 | RLHF 训练显式奖励"谄媚用户"，会产出大量礼貌性废话 |

> **备注**：低谄媚比高智商更重要。交互式分析场景中，模型迎合你的观点比不够聪明更危险——你会被引导到错误的方向而不自觉。MiMo 的低谄媚不是"冷漠"，是训练目标的必然结果：优化任务完成效率，不优化用户满意度。对话场景中用户觉得"它不友好"，工程场景中开发者觉得"它不废话"——同一种特质，不同场景的评价完全相反。详见 [LLM 基础认知与心智模型 §5.4](llm-fundamentals.md#54-辅助分析快速探索思维边界)。

### 4.2 实测数据

**MiMo 2.5 表现**：
- 全速流式喷射，中间没有任何思考停顿与断流
- 2 分钟内稳稳落盘 480 行纯干货 Markdown 文档

→ 配置指南详见 [附录 C.2](#c2-生产部署配置)

### 5.2 生产环境端（面向普通客户）

- 所有高频、低延迟、强结构化响应的业务接口（客服 Agent、Webhook 路由器、自动化数据看板、API 拼装网关），全量替换为 MiMo 2.5
- 彻底杜绝线上业务出现"分钟级延迟"或"URL 都拼不对"的低级故障
- 利用 MiMo 2.5 极致的单步 TPS 吐出量，把线上客户的即时交互延迟打进毫秒级

### 5.3 Fallback 策略（5% 极端硬核场景）

对于极少数需要深度推理的场景（数学公式、算法证明）：

- **推荐**：GPT-OSS-20B + Low 模式（本地部署，可配置推理努力度）
- **备选**：网关层保留一个支持手动控制推理努力度为 Low 的轻量模型
- **严禁**：开启 Qwen 全量思考模式（无底洞）
- **原则**：宁可用慢但可控的低思考模式，不用快但不可控的全量思考

## 附录 A：思考管理技术光谱

### A.1 四大流派定位

```
[ 纯执行流 (No Pad) ] ── [ 可控思维流 (Constrained) ] ── [ 深度推理流 (Deep Pad) ]
│                       │                                │
100% 直球瞬间输出        认知载荷动态短路                  无止境脑内博弈
│                       │
│                  [ 窄思维链 (Narrow CoT) ] ← MiMo 2.5
│                  约束在窄问题域：工具链拆解 + 边界检查
│                  流式无断流，成本极低
```

| 流派 | 特征 | 代表模型 |
|------|------|----------|
| 纯执行流 | 彻底拿掉草稿本，试错权交给外部编译器纠偏 | Qwen3-Coder-Next, Gemma 4 (关思考) |
| 可控思维流 | 限定 token 预算，95% 把握即强行终止 `<think>` | GPT-OSS (Low/Mid/High), Qwen 3.7 (Low Effort) |
| 窄思维链 | MiMo 2.5：经 MOPD + Agent RL 微调，只做工具链拆解和边界检查 | MiMo 2.5 |
| 深度推理流 | 面对简单问题也推演数千 token 边缘 case，厚 Harness 下严重延迟 | DeepSeek-R1, Qwen3-Coder-Plus (全量) |

### A.2 为什么厚 Harness 不需要深度 CoT

- **外置纠偏等价于脑内推演**：LSP、编译器（`cargo check`）、测试运行器是 100% 确定性的物理评估引擎，10ms 给出最精准答案。推理模型在 `<think>` 里花 30 秒模拟"编译器怎么看这个生命周期错误"，纯属无用功。
- **过度思考摧毁心流**：推理模型在厚 Harness 长上下文污染下，极易触发无病呻吟的反思，产生极高首字延迟。
- **商业算力灾难**：低难度场景常规模型直觉准确率已 95%+，为 0.5% 提升让用户傻等数千隐藏思考 Token，ROI 严重过度工程。

→ 思辨分析详见 [LLM 过度思考批判 §1](llm-overthinking-critique.md#1-核心悖论草稿本的双重人格)

## 附录 C：部署实操

### C.1 厚 Harness 驯服铁律

**A. 物理层短路控制（Hermes / OpenClaw）**

任何支持 `extra_body` 请求体透传的智能体框架中，绝不让草稿本野蛮生长：

```json
"thinking_budget_tokens": 512
```

强制缩减草稿本面积，将使用目的从"代码编译模拟"降维到"工具链序列规划"，用完立刻关闭内心戏。

**B. 心理学解除防御提示词（Zed，无 extra_body）**

Zed 架构严禁 `extra_body`，必须通过 System Prompt 心理干预：

```
[草稿本协议：外部验证型架构]
1. 你目前运行在带有活跃 LSP 和后台 'cargo check' 自动编译流的 Zed IDE 中。
2. 严禁使用  空间去静态模拟编译或检查复杂的变量类型。
   外部的物理编译器会接管并处理这些错误。
3. 请纯粹将  块当作临时、极简的 Checklist，仅用来列出下一步要输出的 API 或重构步骤。
4. 将草稿本输出严格控制在 5 行以内，2 秒内无缝过渡到最终代码输出。
```

### C.3 实战案例：业务侧 ID 链接生成架构重构

**问题**：业务侧根据 ID 生成链接（`PROD_992147` → `https://example.com/item/992147`）。完整 CoT 太慢，关闭 CoT 后链接格式频繁损坏（`?&=%` 等敏感符号幻觉）。

**根因**：URL 包含大量敏感符号（`?`, `&`, `=`, `%`）。自回归模型直球输出时缺乏全局缓冲，极易发生符号幻觉。草稿本允许模型在脑内完成"提取前缀 → 匹配路由 → 准备拼接"的自查。

**老架构痛点**：

```
【手工两阶段管道】
阿里API ───(吐出长段预处理文本)───> 本地后端 ───(运行繁重正则解析)───> 提取URL
```

- 恶性网络 I/O 损耗：模型脑内推演在云端，被迫将数百 Token 废话通过公网全量下载再正则清洗
- 格式脆弱性：模型预处理文本稍换句式，后端正则立刻失败→偶发 500

**新架构**：MiMo 2.5 + JSON Schema 强控

```
MiMo 2.5 [窄思维链直球输出] ──> 干净 JSON (零正则、毫秒响应)
```

三大优化：
1. **窄思维链**：极短即完成，首字延迟毫秒级
2. **格式确定性**：`response_format: { "type": "json_object" }` 强控，输出 100% 完美
3. **成本断崖**：单次请求成本降低 50-100 倍
