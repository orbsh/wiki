# ADR: 重构分布式存储范式——以 Fjall + Openraft + AI 替代传统 Redis 架构

## 1. 背景与问题陈述

在传统分布式系统架构中，Redis 长期被视为解决高并发、状态共享与分布式锁等问题的"银弹"。然而，随着 Rust 等常驻内存语言的普及，以及 Fjall、Openraft 等新一代 Rust 原生中间件的成熟，叠加 AI 辅助编程带来的工程红利，许多关于 Redis 的传统认知在现代技术栈下已不再适用。继续盲目沿用 Redis 作为核心分布式协调组件，不仅带来了不必要的网络 RTT 与序列化开销，更在多写强一致性、分布式安全及工程维护成本上暴露了严重短板。

→ 详细批判见：[Redis Critique](./redis-critique.md)

## 2. 决策内容

全面摒弃将 Redis 作为核心分布式协调与持久化存储的传统架构，转向采用"Fjall（嵌入式 KV 存储） + Openraft（分布式共识） + AI（工程集成）"的原生分布式系统范式。核心决策包括：

- **存储与共识内生化**：使用纯 Rust 编写的 Fjall 替代 Redis 实现进程内持久化存储，使用 Openraft 替代 Redis 的哨兵/集群机制实现强一致性共识。
- **工程模式重构**：坚决拒绝手写底层共识协议，直接引入 Openraft 成熟库；利用 AI 专注于生成 Fjall 与 Openraft 对接的胶水代码与状态机逻辑。

## 3. 决策依据与推理链条（传统认知的颠覆）

### 认知颠覆一：从"跨网络组件"回归"进程内原生"

- **传统认知**：无论何种语言，高性能状态存储必须依赖 Redis。
- **现代现实**：对于 PHP 等短生命周期语言，Redis 是维持状态的无奈之举；但对于 Rust 等常驻内存语言，进程本身就是高效的状态容器。Fjall 作为嵌入式引擎，消除了跨进程/跨网络通信的昂贵开销（RTT、序列化），在单机与分布式场景下均能实现纳秒级响应。

### 认知颠覆二：从"伪分布式"迈向"真分布式共识"

- **传统认知**：Redis 集群或 Redlock 足以应对分布式协调与锁需求。
- **现代现实**：Redis 本质是"带网络接口的内存组件"，缺乏内置的强一致性共识机制。其分布式锁（Redlock）存在 GC 进程暂停与时钟漂移等致命破绽，无法保证绝对互斥。Openraft 提供了经过严格数学证明的 Raft 共识算法，从根本上解决了跨节点的多写协调与数据一致性问题。

### 认知颠覆三：从"高门槛造轮子"变为"AI 赋能的有手就行"

- **传统认知**：自建分布式 KV 系统复杂度极高，仅适合顶级架构师攻坚。
- **现代现实**：底层复杂度已被 Fjall 和 Openraft 等成熟库封装。AI 能够极其高效地处理组件对接的"胶水代码"与样板逻辑。在"成熟库保底，AI 做集成"的模式下，构建原生强一致性分布式系统的门槛被大幅抹平。

## 4. 预期影响与后果

- **性能与一致性跃升**：系统摆脱了网络 IO 瓶颈，获得原生语言级别的极致性能；同时具备了真正的分布式强一致性保障，彻底规避了 Redis 在分布式协调上的安全漏洞。
- **架构自主可控**：摆脱了对外部中间件（如 Redis 及其潜在的商业授权风险）的依赖，将系统控制权完全收归宿主语言内部。
- **工程效率质变**：借助 AI 集成能力，开发与维护成本显著降低，使团队能够以极小的代价构建出过去只有大厂才能驾驭的底层分布式系统。

## 5. 合规性说明

本决策严格遵循"拒绝造轮子"的工程原则。Raft 核心协议坚决使用现成开源库（Openraft），AI 仅被定位为集成工具，用于处理业务逻辑打通与接口对接，确保系统基石的稳固可靠。

## RESP 协议 vs 二进制序列化：嵌入式架构的物理优势

Redis 的 RESP（Redis Serialization Protocol）是基于行切分的纯文本协议，使用 `+`、`-`、`:` 和 `\r\n` 作为数据边界。在高吞吐场景下，这种文本协议引入三层开销：

- **字符串扫描开销**：RESP 是行分隔的，CPU 必须逐字节扫描 `\r\n` 分隔符，再将 ASCII 字符串转换为整型。CPU 时钟周期消耗在文本解析上。
- **带宽浪费**：二进制中 64 位无符号整型只需 8 字节，RESP 文本协议中 `18446744073709551615` 占 20 字节，外加类型符号和结束符。
- **跨进程内存拷贝**：Redis 是独立进程，数据从内核 Buffer 拷贝到 Redis 进程，序列化后经网络传输，再拷贝到应用进程内存。跨进程、跨网络的多次拷贝是嵌入式架构不存在的开销。

### Bincode：零元数据开销的序列化

Bincode 是 Rust 原生的纯二进制序列化格式。一个包含 `i32` 和 `bool` 的结构体序列化后恰好 5 字节——无字段名、无分隔符、无类型标记。反序列化不需要解析：CPU 从磁盘（Fjall）或网络（Openraft 的二进制 RaftLog）读出字节，在几个时钟周期内直接 cast 为 Rust 内存结构体。

### CBOR：兼顾动态扩展与硬件吞吐

CBOR 是二进制格式，保留 JSON 的 KV 动态扩展能力，但字段名和类型编码为紧凑的二进制头部。二进制头部包含字段长度，解码器可直接跳过不关心的字段，无需逐字节扫描分隔符。

### 零拷贝反序列化

嵌入式架构（Fjall + 多语言引擎在同一 Rust 进程的同一内存地址空间）实现了运行期零拷贝反序列化。Fjall 从 NVMe 读数据到共享 Buffer 后，利用 Rust 生命周期系统与 Bincode/CBOR 结合，脚本引擎的指针可直接指向 Fjall 的底层磁盘缓冲区内存，无需额外内存分配和拷贝。

```rust
#[derive(Deserialize)]
struct DistributedAgentContext<'a> {
    id: u64,           // 固定 8 字节物理映射
    status: &'a str,   // 零拷贝：指针直接指向 Fjall 磁盘缓冲区
    raw_payload: &'a [u8], // 零拷贝：无 RAM 复制
}

fn execute_zero_copy_pipeline(raw_bytes_from_fjall: &[u8]) {
    let context: DistributedAgentContext = bincode::deserialize(raw_bytes_from_fjall).unwrap();
    // context.status 的内存指针直接传给内嵌的 Steel Lisp / PyO3 执行
}
```

外部 Redis 因跨进程网络隔离，数据必须经过多次拷贝和 RESP 包装，物理上无法实现零拷贝。
