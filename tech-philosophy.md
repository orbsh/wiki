# 技术哲学：应试工程 vs. 范式转移

## "应试工程"陷阱

许多国内领先的"高并发"技术（如阿里双十一吹嘘的那些）实际上是顶级的**"应试工程"**：

1.  **制造考卷**：创造人为的极端场景（如亚秒级秒杀、春节红包）来展示技术实力。
2.  **钻营考纲**：
    *   **Dubbo**：KPI 驱动的开源项目。阿里内部自己用的是 HSF。Dubbo 曾 largely 被废弃，然后为了竞争公关而被复活。它代表了开源界的"圈子自嗨"现象。
    *   **TPC-C 跑分（OceanBase）**：与 Harbor 的 CNCF 毕业类似。传统数据库巨头（Oracle、Microsoft）不再参与，因为基准测试已过时。"获胜"通常涉及：
        *   **硬件作弊**：使用不代表标准云环境的特殊、昂贵的 RDMA 网络和 NVMe 集群。
        *   **规则漏洞**：使用"复制表"将分布式 JOIN 变成单节点本地读取，绕过基准测试本应测试的分布式开销。
3.  **结果**：在特定测试中获得顶级分数，但在底层计算机科学方面创新有限。这就像在泥路上以极快速度驾驶通用卡车底盘——底盘不是你的，驾驶技术也不能转化为制造更好的引擎。

## 范式转移（"SurrealDB 方式"）

真正的一流工程不试图赢旧游戏；它发明新游戏。

*   **第一性原理重新思考**：SurrealDB 没有修补 SQL 或给 Postgres 添加图插件，而是问："为什么我们需要为不同数据模型使用不同的数据库？"
*   **"学习孤岛"是特性**：通过创建新语法（SurQL）而非模仿 SQL，他们摆脱了 50 年的关系型包袱。他们重塑了开发者的思维模型。
*   **AI 原生优势**：在 AI 时代，新 DSL 的"学习成本"可以忽略不计。AI 助手可以瞬间写出完美的 SurQL，使现代、符合人体工程学语法的生产力收益立即可用。

## 结论

我们应该对通过"堆硬件"或"钻基准测试漏洞"获胜的技术保持怀疑（Harbor、TPC-C 领先者）。真正的技术领导力体现在**范式转移**——从根本上简化开发者体验并重新定义架构栈的项目，即使它们不在遗留基准测试中名列前茅。

## 约束链分析（逻辑链：从结果反推约束）

*一种评估事物为何如此——以及何时不再有效的思维工具。*

### 核心方法

不盲目接受传统智慧，而是**从观察到的结果向后追溯使其成立的条件**。然后问：**哪些条件不再成立？**

这类似于第一性原理思维，但更有针对性：
- **第一性原理**："基本事实是什么？"（自下而上的重构）
- **约束链**："什么条件使这个结果有效？它们仍然成立吗？"（自上而下的失效分析）

### Redis 案例研究

**观察到的结果**："Redis 很快。因此，需要速度的地方都用 Redis。"

**追溯约束**：
1. Redis 的"快"声明基于**无网络跳数的内存操作**——本地基准测试。
2. 实际上，每次 Redis 操作都涉及**网络往返**（通常 0.1–1ms）。对于细粒度操作（单键查找），网络开销*主导*了内存速度优势。
3. 所以 Redis 的"快"只在以下情况成立：(a) 操作批量处理（管道），或 (b) 替代方案是磁盘 I/O（网络开销与磁盘延迟相比可以忽略）。

**为何尽管如此仍流行？**
- **历史约束**：PHP 的无共享架构**没有持久进程内内存**。每个请求从零开始。无法在进程内缓存任何东西。Redis（或 Memcached）是跨请求缓存的*唯一*选择。
- **没有可行的替代方案**：在具有持久工作进程的现代框架（Go、Rust、Node.js）出现之前，PHP 约束是普遍的。

**哪些约束不再成立？**
- 现代软件栈（Go、Rust、Node.js、Elixir）有**持久工作进程**和进程内内存。应用级缓存（如 `sync.Map`、`Arc<RwLock<HashMap>>`）完全消除了热数据的网络跳数。
- **CDN 和边缘缓存**在网络边缘处理静态/半静态内容——比任何集中式 Redis 都快。
- **客户端缓存**（浏览器 Cache API、service worker）进一步减少后端负载。

**"分布式协调"作为有效利基呢？**

常见辩护是："Redis 作为缓存可能不理想，但对分布式锁、排行榜、消息传递等仍然有价值。"这个辩护也崩溃了：

1. **Redis 不是真正分布式的。** 它是单节点进程。Redis Cluster 是客户端分片，无强一致性。当你使用 Redis 作为"分布式协调层"时，Redis 本身成为**单点故障**。

2. **分布式锁**：简单的单线程服务（Rust `Arc<Mutex<HashMap>>`，或任何有基本 HTTP 服务器的语言）达到相同效果。Redis 是大炮打蚊子。

3. **排行榜（有序集合）**：Redis 强制所有数据在内存中——100 万用户排行榜永久消耗 GB 级 RAM。PostgreSQL 自动管理分层缓存（热条目在缓冲池，冷数据在磁盘）——相同数据少 ~1000 倍内存。

4. **地理空间**：数据很大；全部放在内存中在架构上站不住脚。PostGIS 使用 R-Tree 索引和磁盘存储——可比性能，少几个数量级的内存。

5. **HLL**：很少使用，作为库或数据库内部特性更好（PostgreSQL 用它做列统计）。

6. **消息传递**：Redis pub/sub 无持久化、无消费者组、无反压。NATS/Kafka/RabbitMQ 是专用系统；相比之下 Redis 是玩具。

**结论**：Redis 的理由在**所有用例**中都已崩溃，不仅仅是缓存。"数据结构丰富性"是设计异味——Redis 试图成为所有东西（缓存、锁服务、消息队列、数据库），但没有一样做得好。它唯一剩下的理由是**惯性**——来自 PHP 时代的僵尸约束。现代架构应该使用专用工具：进程内内存用于缓存，基于 Raft 的存储用于协调，PostgreSQL/SurrealDB 用于复杂结构，NATS/Kafka 用于消息传递，PostGIS 用于地理空间。

### 如何应用

1. **识别声明**："X 好因为 Y。"
2. **提取隐藏约束**：Y 成立需要什么条件？
3. **检查每个约束**：哪些是历史产物？哪些已被技术变革推翻？
4. **重新评估**：如果约束不再成立，结论就是僵尸——它靠惯性持续，而非逻辑。

### 其他应用

- **"微服务最好"**：当团队庞大且部署必须独立时有效。对于协调开销超过部署灵活性收益的小团队则不太有效。
- **"Kubernetes 万能"**：当你有 50+ 服务且需要自动缩放时有效。当你只有 5 个服务和 3 人团队时则不太有效。
- **"最终一致性没问题"**：对社交媒体信息流有效。对金融账本是灾难性的。约束是*业务能承受什么*，而非技术提供什么。
