# Redis："网络 RAM"陷阱

**结论**：新架构中拒绝使用。
**状态**：针对旧语言（PHP/Java）根本性设计缺陷和硬件限制的"补丁式"解决方案。

## 执行摘要

经典的"Redis + MySQL"技术栈不是工程理想，而是历史妥协。Redis 在更好的替代方案（异步运行时、本地 KV、事件溯源）不存在或不成熟的年代，解决了特定问题（PHP/Java 的无状态性、微秒级延迟）。在现代高性能环境（Rust/Go、事件溯源）中坚持使用 Redis，会引入不必要的延迟和复杂性。

*Nginx 批判见 [`nginx-critique.md`](./nginx-critique.md)。*

## 1. 设计哲学：2009 年的最优解

### 历史背景

Redis 诞生于 2009 年——一个由**机械硬盘（HDD）**和**单核 CPU**主导的时代。它的单线程、全内存架构是当时硬件现实的最优解：当磁盘 I/O 需要毫秒级时，把所有东西放在 RAM 中并顺序处理是实现微秒延迟的唯一方法。

Redis 的爆炸性流行是因为 **PHP-FPM 是无状态的**。每个请求都从零开始，迫使应用将*所有东西*（会话、计数器、缓存）卸载到外部存储。在具有持久运行时的语言（Rust、Go、Node.js、Elixir）中，应用进程可以在本地内存（HashMap、B-Tree）中高效持有状态，无需网络开销。Redis 常常是"Share Nothing"架构的拐杖——通过网络共享*太多*。

### 现代硬件的失效

我们已不在 2009 年。现代服务器拥有：
- **多核 CPU**（16-128 核）：Redis 的单线程事件循环浪费了 95% 以上的可用算力
- **NVMe SSD**（亚毫秒级 I/O）："RAM vs 磁盘"的差距已从 1000 倍缩小到 10 倍
- **云原生架构**：网络 RTT（同数据中心 0.5-2ms）主导了 Redis 的微秒级处理

Redis 的单线程事件循环意味着：
- **吞吐上限**：每实例 ~10 万 QPS（受限于单核性能）
- **无法利用多核**：128 核服务器浪费 99% 的算力
- **扩展需要分片**：Redis Cluster 是客户端分片，无强一致性，手动管理槽位，重新分片痛苦

这不是"可以修复的限制"——这是一个**根本性的架构选择**，在 2009 年合理，但在 2026 年站不住脚。现代系统（Rust/Tokio、Go、Elixir）通过利用多核并行实现**数百万 QPS**。Redis 被困在单核时代。

### 多实例 Redis：复杂性陷阱

**变通方案**：在 128 核机器上运行 64 个 Redis 实例（每实例 2 核）以利用多核。

**问题**：这引入了巨大的复杂性，却没有解决根本问题：

**1. 客户端分片复杂性**
```python
# PostgreSQL：简单直接
db.execute("SELECT * FROM users WHERE id IN (1, 1001, 2001)")

# 多实例 Redis：客户端必须实现路由
def get_users(user_ids):
    results = {}
    for uid in user_ids:
        instance = hash(uid) % 64  # 路由到正确实例
        results[uid] = get_redis_client(instance).get(f"user:{uid}")
    return results
# 64 次网络请求而不是 1 次查询
```

**2. 跨实例操作不可能**
- `MGET user:1 user:1001 user:2001` → `CROSSSLOT` 错误（键在不同实例上）
- 跨实例事务需要 2PC 或 Saga 模式（应用层补偿）
- PostgreSQL：单实例，ACID 事务，无分片逻辑

**3. 内存浪费持续**
- 100GB 数据集分布在 64 个实例 = 每实例 1.56GB
- 每个实例仍将所有数据存储在内存中（无分层缓存）
- 冷数据（数据集的 90%）占据昂贵的 RAM 但很少被访问
- PostgreSQL：20GB 缓冲池（热数据）+ 100GB SSD（所有数据）= 节省 80% 内存

**4. 运维复杂性爆炸**
- 64 个配置文件，64 个监控端点，64 个备份策略
- 重新分片需要数据迁移（迁移期间服务降级）
- PostgreSQL：1 个实例，1 个配置，1 个监控端点

**性能对比**（128 核机器，100GB 数据集）：

| 指标 | 64 个 Redis 实例 | 1 个 PostgreSQL 实例 |
|------|-----------------|---------------------|
| CPU 利用率 | 64/128 (50%) | 4-8/128 (3-6%) |
| 内存使用 | 100GB（所有数据）| 20GB（热数据缓存）|
| 吞吐量 | 6.4M QPS | 3-6M QPS |
| 跨实例操作 | ❌ 不支持 | ✅ 支持 |
| 事务 | ❌ 仅单实例 | ✅ ACID |
| 运维复杂性 | 🔴 极端（64 个实例）| 🟢 低（1 个实例）|
| 扩展成本 | 🔴 高（重新分片 + 迁移）| 🟢 低（加磁盘）|

### PostgreSQL 的多核吞吐量

现代 PostgreSQL 在吞吐量上完全可以把单线程的 Redis 显著超越。

**PostgreSQL 的现代重武器**：
- **多进程/多线程并发执行**：Parallel Query（并行查询）能力，跨核心线性扩展
- **高度优化的共享内存缓冲区**：直接映射到操作系统共享内存（Shared Buffers）
- **Unlogged Tables（不记日志表）**：跳过 WAL 日志，换取最大写入吞吐（当持久性不是关键时）
- **JSONB 重度缓存**：将 JSON 文档直接缓存在共享内存中，配合 GIN 索引查询

**基准数据**（128 核机器，全 NVMe 阵列，200-500 线程并发压测）：

| 指标 | 普通表（Logged） | UNLOGGED 表 | Redis 单线程 |
|------|-----------------|-------------|-------------|
| **纯写入 TPS** | 5-15 万（受限于 fsync/WAL） | 30-60 万+（受限于 LWLock 锁竞争） | ~10 万（单核瓶颈） |
| **只读 TPS（全内存命中）** | 150-250 万 | 150-250 万（读性能与 WAL 无关） | ~10 万（单核瓶颈） |
| **批量导入** | 30-50 万条/秒 | 60-100 万+条/秒 | 不适用 |

关键发现：**只读场景下，即使不开 UNLOGGED，PG 的吞吐量也是 Redis 的 15-25 倍**（150-250 万 vs ~10 万）。原因很简单——PG 能利用全部 128 核，Redis 被锁死在单核。Redis 的"内存速度优势"在多核并行面前毫无意义。

**架构的讽刺**：Redis 被设计来避免磁盘 I/O 和锁的"开销"。但现代 PostgreSQL 的共享缓冲池本质上就是一个高度优化、多核安全、ACID 合规的内存缓存——外加一个额外好处：冷数据自动溢出到廉价 SSD，而不是 24/7 占据昂贵的 RAM。

### 可扩展性鸿沟

当你需要扩展到单个 Redis 实例之外时，成本和复杂性会爆炸：
- **缓存**：流式计算（RisingWave/Flink）+ CDN **便宜 900 倍**，且可无限水平扩展。Redis Cluster 需要手动管理槽位，跨槽操作失败，重新分片痛苦不堪。
- **会话**：这是一个伪装成缓存问题的**分布式协调问题**。逻辑很简单（存储/检索用户状态），但 Redis Cluster 引入了单点故障风险。现代分布式 KV 存储（etcd、Consul、Fjall+Openraft）提供正确的共识保证。
- **排行榜**：在大规模（1 亿+条目）下，专用算法（Count-Min Sketch、T-Digest）或流处理引擎处理聚合更高效。在中等规模（100 万条目）下，PostgreSQL 配合适当索引可实现**100 万并发查询**——Redis 的吞吐优势消失了，因为它被单核性能（~10 万 QPS）瓶颈限制。

**混合部署概率**：在 N 台机器的集群中，应用和 Redis 在同一台机器上的概率是 1/N。即使是同机部署，预期延迟仍然是 ~1ms。

DHH 的洞察：现代硬件性能如此之好，我们应该**重新审视 2010 年代之前的每一个架构假设**。过去没有缓存层就"不可能"实现的事情，现在通过直接数据库查询就能轻松实现。Redis 的存在是一个**历史约束**，来自硬件慢 100 倍的时代。

## 2. 网络延迟：阿喀琉斯之踵

**如果数据在远程，"快速内存数据库"就是一个语义矛盾。**

内存访问延迟约 100 纳秒。网络 RTT（同数据中心典型值 0.5-2ms）比内存访问慢 5,000-20,000 倍。即使 Redis 在 0.01ms 内完成查询，网络往返（1ms）也主导了总延迟。使用 Redis 来"加速"数据访问，通常比查询索引良好的本地 SQLite/Redb 文件或使用本地进程内缓存更慢。

### RTT 计算：被忽视的数量级

这是架构设计中最常见的盲区——开发者关注 Redis 内部的微秒级延迟，却忽视网线中光速的限制。

```
Redis 内部执行时间：        ~0.1 ms（纯内存查找）
PostgreSQL 共享缓冲区执行：  ~0.5-1.0 ms（共享内存）
机房局域网真实 RTT：         ~1.0-3.0 ms（网络物理开销、TCP 握手、交换机路由）

真实总延迟 = 网络 RTT + 数据库内部处理时间
```

**正面对比**：
- **Redis 总延迟**：2.0 ms (RTT) + 0.1 ms = **2.1 ms**
- **PostgreSQL 总延迟**：2.0 ms (RTT) + 0.5 ms = **2.5 ms**

两者绝对差距仅 0.4 毫秒。在绝大多数商业 Web 服务和 API 后端场景中，0.4 毫秒的差距对终端用户不可感知。为此在架构中引入一个高风险、易丢失、内存昂贵、还需防范缓存击穿和雪崩的独立中间件，在工程上缺乏合理性。

这不是性能优化，是 cargo cult（货物崇拜）式的架构模仿。

## 3. 成本分析：内存 vs 磁盘

### 硬件资源对比

**场景**：100GB 数据集，10 万 QPS 目标

| 资源 | Redis | PostgreSQL | 成本差异 |
|------|-------|------------|---------|
| **CPU** | 2-4 核（单线程）| 16-32 核（多核）| PG 多 4-8 倍 |
| **内存** | **100GB**（所有数据）| **10-20GB**（热数据缓存）| Redis 多 5-10 倍 |
| **磁盘** | 0GB（纯内存）| 100GB SSD（所有数据）| PG 更多（但 SSD 便宜）|

**内存 vs 磁盘定价**（AWS 参考价格，2026 年）：
- 内存：$0.023/GB/小时 = **$16.56/GB/月**
- SSD：$0.08/GB/月
- **内存比 SSD 贵两个数量级**（具体倍数随云厂商定价调整，但量级差恒定）

### 内存浪费

```
100GB 数据集访问模式：
- 10% 热数据（1000 次访问/天）= 10GB
- 30% 温数据（10 次访问/天）= 30GB
- 60% 冷数据（0-1 次访问/天）= 60GB

Redis：必须将所有 100GB 存储在内存中
- 内存成本：100GB × $16.56/GB/月 = $1,656/月
- 浪费：60GB 冷数据占据 60% 内存但很少被访问

PostgreSQL：分层缓存
- 缓冲池（内存）：20GB（热数据）
- 磁盘（SSD）：100GB（所有数据）
- 内存成本：20GB × $16.56/GB/月 = $331/月
- 磁盘成本：100GB × $0.08/GB/月 = $8/月
- 总计：$339/月

节省：60GB 冷数据放在磁盘上（$0.08/GB/月 vs $16.56/GB/月）
```

### 总拥有成本

（3 年，100GB 数据集，10 万 QPS）：

| 成本项 | Redis | PostgreSQL | 差异 |
|--------|-------|------------|------|
| **硬件** | $166,320 | $21,024 | Redis 多 8 倍 |
| **运维** | $54,000（3 人年）| $18,000（1 人年）| Redis 多 3 倍 |
| **扩展** | $138,600（增长到 1TB）| $21,600（增长到 1TB）| Redis 多 6 倍 |
| **总计** | **$358,920** | **$60,624** | **Redis 多 6 倍** |

**扩展成本对比**（100GB → 1TB）：
- **Redis**：成本增加 10 倍（必须为所有数据添加内存）
- **PostgreSQL**：成本增加 2 倍（主要添加磁盘，不是内存）

Redis 的"低成本"是个神话。在真实世界的中等规模场景中，PostgreSQL 的 TCO 是 **Redis 的 1/6**。Redis 的内存浪费和 CPU 浪费使其成为**最昂贵的缓存解决方案**。

## 4. 反应式架构：CDN、流式推送与版本号接口

传统的 Redis 缓存是**请求驱动的被动模式**——用户请求来了，才去 Redis 查、去源站查。现代架构应该反过来：数据变更时**主动推送**到边缘，用户请求只命中已就绪的缓存。这一节拆解三个层面：CDN 替代 Redis 做静态缓存、流引擎主动推送预计算结果、版本号接口解决缓存失效。

### CDN vs Redis：静态数据杀手

对于典型的 Web 应用（电商、内容平台），数据自然分为**静态**和**动态**两类：

| 数据类型 | 示例 | 新鲜度要求 |
|---------|------|-----------|
| **静态** | 商品列表、分类页、博客文章 | 小时到天（可接受的陈旧度）|
| **动态** | 用户资料、购物车、实时库存 | 秒级（必须新鲜）|

**Redis 反模式**：使用 Redis 缓存*所有东西*——包括不常变化的静态数据。这在架构上是浪费的。

**CDN 的优势**：静态数据（甚至 API JSON 响应）可以缓存在 CDN 边缘。对于每天更新一次的商品目录：
- **Redis**：每个请求都打到你的源站 → Redis → 响应。你为 Redis 内存、网络跳数和源站负载付费。
- **CDN**：请求从最近的边缘节点提供服务。零源站负载，零 Redis 成本，全球亚 10ms 延迟。

**成本对比**（粗略）：
- Redis：托管 Redis 约 $0.10/GB/小时（AWS ElastiCache）
- CDN：CloudFront 约 $0.08/GB/月（每 GB 便宜 1000 倍）

对于缓存 1 天的 100GB 商品目录，Redis 成本约 $240/天；CDN 成本约 $8/月。**CDN 便宜 900 倍。**

### 流式预计算与主动推送

CDN 解决了静态数据的缓存问题，但数据更新后如何让边缘节点拿到新版本？传统方案是 TTL 过期后回源拉取——这又回到了"请求驱动"的被动模式。现代架构应该**主动推送**而非**被动拉取**。

现代架构（RisingWave、Flink）可以**预计算**静态数据并直接推送到 CDN：
1. 数据库中的原始数据发生变化（例如，商品价格更新）。
2. 流引擎（RisingWave/Flink）检测到变化，聚合/转换它。
3. 预计算的 JSON 通过 API 推送到 CDN 边缘节点（Cloudflare Workers、Fastly）。
4. 用户从 CDN 获取新鲜数据，永远不会打到你的源站。

这完全消除了静态数据的 Redis，并提供**可预测的延迟**（CDN 边缘）而不是**可变延迟**（Redis + 源站）。

**权衡**：这需要重大的架构重构。传统的"应用服务器 + Redis 缓存"模式必须重新设计为"流管道 + CDN 边缘"。对于遗留系统，这是数月的迁移。对于新项目，这是从第一天起就正确的架构。

### 索引版本号接口：最简缓存失效

CDN 缓存常被质疑"最终一致性"——数据更新后边缘可能返回旧版本。以商品目录为例：

流计算生成索引 JSON，引用一堆资源（商品详情、图片、价格），每个资源带版本号。索引本身也有版本号。索引体积大，放 CDN 边缘能显著加速。但 CDN 缓存失效是问题——你无法主动通知全球边缘节点"索引更新了"。

解法极简：**一个接口返回索引的当前版本号（或索引下载链接）**。

```
流计算生成索引 JSON
  → 每个资源引用带版本号
    → 索引本身带版本号
      → 索引推送到 CDN（大文件，CDN 只做静态文件服务）
        → 客户端调用接口获取当前版本号
          → 版本不匹配则从 CDN 拉取新索引
```

CDN 不需要任何复杂操作——它只是托管静态文件。版本检查在客户端完成：`if (remote_version > local_version) { download_from_cdn(); }`。

接口内部：版本号存在共享内存中，读取纳秒级。写入由流计算端直接赋值。无论实现语言多慢，共享内存直读都比 Redis 通过网络接口访问快几个数量级。

不需要 Redis 的 TTL 管理、缓存击穿防护、雪崩降级——极动态的数据本身没有缓存的必要，缓存引入的复杂性远大于收益。

**与现代 Lakehouse 架构思路一致**：Lakehouse 通过 Catalog 层（元数据服务）管理数据版本和快照，而不是依赖存储层的 TTL 缓存。索引版本号接口是同一思路的极简实现——把版本治理从存储层提升到控制面，用一个接口替代一整套缓存失效机制。详见 [Lakehouse 研究](lakehouse-research.md)。完整论述见 [反应式架构](reactive-architecture.md)。

## 5. 缓存层分类学：Redis 的位置（以及为什么它是错的）

现代缓存不是"Redis"和"无缓存"之间的二元选择。有一个**缓存策略谱系**，每个都有不同的权衡。Redis 占据了一个越来越过时的尴尬中间地带。

### L0：进程内内存（最快，最小范围）

**示例**：Caffeine（Java）、cachepot（Rust）、带 TTL 的 `HashMap`、LRU 缓存。

- **延迟**：~10-100 纳秒（直接内存访问，无序列化）。
- **范围**：单个应用实例。不跨进程/节点共享。
- **失效**：简单（进程本地状态，无分布式共识）。
- **用例**：适合 RAM 的热数据，每请求缓存，有状态应用中的会话数据。

**Redis 为何在这里输**：即使 Redis 在 0.01ms 内执行，网络 RTT（20-500µs）也主导了。进程内**快 200-50,000 倍**。对于具有持久运行时的语言（Rust、Go、Elixir），没有理由将缓存外部化到网络服务。

### L1：数据库透明缓存（自动，查询级）

**示例**：**ReadySet**（基于 Noria 论文）、**ProxySQL 查询缓存**（已弃用）、**MySQL Query Cache**（8.0 中移除）。

- **机制**：位于应用和数据库之间。拦截 SQL 查询，返回缓存结果，数据变化时自动失效（通过 binlog 复制或基于触发器的跟踪）。
- **延迟**：~100-500 微秒（仍是网络跳数，但少于 Redis + DB）。
- **范围**：数据库查询结果。对应用透明（无需代码更改）。
- **失效**：自动（基于数据流，无手动 TTL 或 cache-aside 逻辑）。

**ReadySet** 是最复杂的：它维护 SQL 查询的**增量物化视图**。当基表变化时，ReadySet 增量更新缓存（不是完全重新计算）。这在架构上优于 Redis cache-aside：
- **无缓存击穿**：ReadySet 优雅地处理并发缓存未命中（合并）。
- **无陈旧数据**：失效是自动且即时的（binlog 驱动）。
- **无应用更改**：即插即用的 MySQL/PostgreSQL 代理。

**Redis 为何在这里输**：Redis 需要手动缓存管理（TTL、cache-aside 模式、失效逻辑）。这容易出错（缓存击穿、陈旧数据、惊群效应）。ReadySet 自动化了 Redis 让你手动做的一切。

**历史失败**：MySQL Query Cache（8.0 中移除）失败是因为它使用全局互斥锁（整个缓存的单锁）。ProxySQL 查询缓存被弃用是因为它无法处理复杂查询（连接、子查询）且没有自动失效。ReadySet 用**数据流图**（Noria 论文）解决了这些问题——每个查询是一个算子 DAG，更新增量流动。

### L2：CDN 边缘缓存（最便宜，最大规模）

**示例**：CloudFront、Cloudflare、Fastly、Akamai。

- **延迟**：~10-50 毫秒（地理分布，但到边缘有网络跳数）。
- **范围**：静态或半静态数据（HTML、JSON、图片、API 响应）。
- **失效**：基于 TTL（最终一致性）或手动清除 API。
- **成本**：~$0.08/GB/月（每 GB 比 Redis 便宜 1000 倍）。

**Redis 为何在这里输**：对于静态数据，CDN **便宜 900 倍**且**更快**（边缘节点比 Redis 集群更靠近用户）。Redis 强制每个请求都打到你的源站 + Redis 集群，为两者付费。

### L3：分布式 KV / 协调（专用，非缓存）

**示例**：etcd、Consul、基于 Raft 的存储。

- **目的**：分布式协调、服务发现、配置管理。
- **不是缓存**：这些是分布式状态的**真实来源（source of truth）**，不是其他数据的缓存。
- **Redis 为何在这里输**：Redis Cluster 不是强一致的（无 Raft/Paxos）。使用 Redis 做分布式锁或协调是单点故障。etcd/Consul 提供正确的共识保证。

### Redis 陷阱：卡在 L0 和 L2 之间

| 层级 | 延迟 | 成本 | 自动化 | Redis 对比 |
|------|------|------|--------|-----------|
| L0（进程内）| 10-100 ns | 免费（你已有的 RAM）| 手动（但简单）| Redis **慢 200-50,000 倍** |
| L1（ReadySet）| 100-500 µs | 中等（代理开销）| **自动**（binlog 驱动）| Redis 需要**手动 cache-aside** |
| L2（CDN）| 10-50 ms | $0.08/GB/月 | 基于 TTL | Redis **贵 900 倍** |
| L3（etcd/Consul）| 1-10 ms | 中等 | N/A（真实来源）| Redis **无共识** |

**判决**：Redis 比 L0 慢，比 L1 自动化程度低，比 L2 贵，比 L3 不可靠。它是**万金油，样样不精**——与 MySQL 的可插拔存储引擎或 Nginx 的 Lua 脚本是同一类反模式。

**Redis *可能*有意义的场景**（罕见）：
- **遗留迁移**：你有一个 PHP/Java 应用，有数千个 cache-aside 调用。重写为使用 ReadySet 或进程内缓存是数月的项目。Redis 是"最不坏"的增量步骤。
- **多语言微服务**：多个服务（Python、Go、Node.js）需要共享缓存。进程内缓存无法跨语言边界工作。但是：ReadySet 是语言无关的（SQL 代理），CDN 是通用的。
- **Pub/Sub 玩具**：你需要简单的发布/订阅，不想部署 NATS/Kafka。但是：Redis pub/sub 无持久化、无消费者组、无反压。NATS 更轻更好。

**对于新项目**：完全跳过 Redis。使用：
- **L0**（进程内）用于热数据。
- **L1**（ReadySet）用于数据库查询缓存（如果你需要）。
- **L2**（CDN）用于静态/半静态数据。
- **L3**（etcd/Consul）用于分布式协调。
- **SurrealDB/PostgreSQL** 用于复杂数据结构（Redis 的有序集合、GEO、HLL 均有更优替代方案）。

## 6. Redis 不是分布式的

Redis 的常见辩护是："即使作为缓存不理想，它作为分布式协调/共享状态层仍然有价值。"这种辩护在审视下崩溃了。

### 核心问题：单点故障

Redis 本质上是一个**单节点进程**。Redis Cluster 不是真正的分布式计算——它是客户端分片，在分区期间没有强一致性保证。当你使用 Redis 作为"分布式协调层"时，Redis 本身就成了你分布式系统中的**最薄弱环节**。

**客户端缓存（Redis 7.0）**：终极讽刺。Redis 引入客户端缓存是因为它终于承认：*远程查询太慢，你无论如何都需要本地数据。* 结果：你最终要管理*两个*缓存（客户端 + 服务端），处理失效逻辑，为边际收益增加复杂性。

### 逐案击破

#### 分布式锁
- **声称**："Redis 非常适合分布式锁（Redlock）。"
- **现实**：Redis 是单节点的，所以锁服务本身就是单点故障。如果 Redis 宕机，所有锁都丢失。
- **替代方案**：一个简单的单线程 Rust 服务，使用 `Arc<Mutex<HashMap>>` — 数据在内存中，无异步，无并发 — 实现相同效果。锁服务是简单的逻辑，Redis 在此场景属于过度设计。

##### Redlock 论战：Kleppmann vs Antirez（2016）

Redlock 是 Redis 作者 Antirez 提出的分布式锁算法——基于多个 Redis 节点实现互斥。当时很多人将其视为生产环境分布式锁的标准方案。

Martin Kleppmann（《Designing Data-Intensive Applications》作者）撰文 **《How to do distributed locking》** 提出系统性质疑：

**核心论点：分布式锁有两个用途——效率（efficiency）和正确性（correctness）。**

| 用途 | 含义 | 正确方案 |
|------|------|---------|
| **效率** | 避免重复工作，锁丢失只是多做一次 | 单机 Redis 就够了，不需要 Redlock |
| **正确性** | 保护共享资源不被并发破坏，锁丢失导致数据损坏 | Redlock 不安全，需要共识算法 |

**Kleppmann 提出的攻击场景：**

1. **GC 停顿问题**：客户端 A 获取锁后发生 GC 暂停，锁在暂停期间过期，客户端 B 获取到同一把锁——两个客户端同时持有锁，互斥失效
2. **时钟跳变**：Redlock 依赖节点间时钟同步，如果某个节点时钟跳变（NTP 校正等），锁的安全性崩溃
3. **网络延迟**：客户端请求锁的响应在路上延迟很久，但锁已过期——客户端以为拿到了锁，实际已失效

**Kleppmann 的替代方案：fencing token（栅栏令牌）**

每次获取锁时，锁服务返回一个单调递增的 token。客户端携带 token 访问资源，资源端拒绝旧 token。这把安全性从锁服务转移到了资源端，不依赖时钟和网络时序假设。

**Antirez 的回应：《Is Redlock safe?》**

- 承认 GC 停顿确实有问题，但认为实际发生概率极低
- 对时钟跳变场景做了反驳
- 认为 Redlock 在工程上是 practical 的

**后续共识：**

这场论战没有明确胜负，但社区形成了清晰结论：

> **Redlock 不能用于对正确性有要求的场景。**

- 需要严格互斥（保护共享资源）→ **ZooKeeper / etcd**（基于 Raft/Zab 共识算法）
- 只为效率优化 → 单机 Redis 够了，不需要 Redlock
- Redlock 处于尴尬中间地带——**比单机 Redis 重，但又没安全到能当正确性保障**

**本质**：这场论战揭示的不只是 Redlock 的缺陷，而是 Redis 作为「代词」的典型模式——很多人用 Redlock，其实没想清楚自己需要的是效率还是正确性。就像架构文档里无脑加 Redis，背后是一个你没想清楚的问题。

#### 有序集合 / 排行榜
- **声称**："Redis 有序集合非常适合排行榜。"
- **现实**：Redis 将所有东西存储在内存中。一个 100 万用户的排行榜永久消耗数 GB 的 RAM。
- **替代方案**：PostgreSQL（或任何真正的数据库）自动管理**分层缓存**。前 1000 名是热的（通过缓冲池在内存中）。其余的在磁盘上，很少被访问。成本差异：相同数据少 ~1000 倍内存。

#### 地理空间
- **声称**："Redis GEO 命令很方便。"
- **现实**：地理空间数据量大（坐标 + 元数据 + 空间索引）。全部存储在内存中成本过高。
- **替代方案**：PostGIS 使用 R-Tree 索引和磁盘存储。查询性能相当；内存占用小几个数量级。对于大型地理空间数据集，Redis 由于内存成本实际上不可用。

#### HyperLogLog (HLL)
- **声称**："Redis 内置 HLL 用于基数估计。"
- **现实**：HLL 不是通用功能。它很少使用，远不如 Bloom 过滤器知名，也不是大多数应用需要的。
- **替代方案**：HLL 更适合作为**库**（例如 Rust 的 `hyperloglog` crate）或数据库内部功能（PostgreSQL 在 `pg_stats` 中内部使用 HLL 做列统计）。它不足以证明专用网络服务的合理性。

#### Pub/Sub、Streams、队列
- **声称**："Redis 可以做消息传递。"
- **现实**：Redis pub/sub 无持久化、无消费者组、无反压。Streams 是对真正消息队列的拙劣模仿。
- **替代方案**：NATS（pub/sub）、Kafka（streams）、RabbitMQ（queues）。这些是为正确持久性、排序保证和消费者管理而构建的专用系统。

#### Lua 脚本
- **声称**："Redis Lua 脚本提供原子多步操作，很方便。"
- **现实**：Lua 作为嵌入式脚本语言有显著的工程摩擦：
  - **从 1 开始的索引**：与 C、Rust、Go、Python、Java、JavaScript 等主流语言的约定相反，每个数组操作都需要心智转换，增加出错概率。
  - **标准库匮乏**：没有内置 JSON、正则、日期/时间处理。任何非平凡逻辑都需要外部 C 库绑定。
  - **生态断层**：Lua 的类型系统、工具链和测试框架与主技术栈（Rust/Go/Python）完全脱节，丧失了强类型保护、Linter 静态检查、成熟的单元测试能力和 IDE 智能感知。
  - **单线程阻塞风险**：Redis 核心是单线程的。如果 Lua 脚本包含复杂逻辑或较长循环，整个 Redis 服务器会阻塞所有其他客户端的请求，直到脚本执行完毕。这种工程妥协的代价是显著的生产力损失。

**SurrealDB 的对比**：SurrealDB 采取了相反的方法——不是嵌入一个糟糕的通用脚本语言（Lua），而是设计了一种**专用查询语言（SurQL）**，在单一连贯语法中处理数据建模、查询、权限和业务逻辑。在 AI 时代，DSL 的学习成本可以忽略不计——AI 助手能瞬间写出完美的 SurQL。设计良好的 DSL 消除了"嵌入式语言税"——在应用代码和数据库脚本之间无需上下文切换。

这是使 SurrealDB 优于 Redis 的同一第一性原理思维：**不要拼接一个糟糕的嵌入式语言；从头设计一个正确的接口。**

### 模式

Redis 的每个"优势"在审视下都溶解了：

| 用例 | Redis "优势" | 为什么是假的 |
|------|-------------|-------------|
| 缓存 | 快速内存 | 网络开销主导；本地 KV 更快 |
| 分布式锁 | 简单 API | Redis 是单点故障；自己构建很简单 |
| 排行榜 | 有序集合 | 全内存浪费；数据库分层缓存便宜 1000 倍 |
| 地理空间 | GEO 命令 | 数据太大无法放内存；PostGIS 更优 |
| HLL | 内置 | 很少使用；作为库更好 |
| 消息传递 | Pub/sub、streams | 无持久化、无消费者组；NATS/Kafka 是正确工具 |
| Lua 脚本 | 原子多步操作 | Lua 对开发者不友好（从 1 开始、无标准库、阻塞单线程）；使用真正的数据库事务 |

Redis 的"数据结构丰富性"不是优势——它是一个**设计异味**。它表明 Redis 试图成为所有东西（缓存、锁服务、消息队列、数据库），但没有一样做得好。现代架构应该为**专用任务使用专用工具**：缓存用进程内内存或本地 KV（Redb/RocksDB），分布式协调用基于 Raft 的存储（etcd、Consul），复杂数据结构用 PostgreSQL/SurrealDB，消息传递用 NATS/Kafka，地理空间用 PostGIS。

## 7. 计算下沉 vs 数据搬运：Redis 的根本性架构缺陷

### "补丁式"架构

当与其他遗留工具结合时，**Redis + MySQL** 创建了一个脆弱的"分布式单体"：
1. **应用查询 Redis**（网络跳数）检查会话/缓存。
2. **应用查询 MySQL**（网络跳数）获取数据。
3. **应用更新 Redis**（网络跳数）失效缓存。

**结果**：一个简单的用户请求触发 **2-3 次网络跳数**。每次跳数都是潜在的故障点和延迟乘数。

详见 [`mysql-critique.md`](./mysql-critique.md) 了解 MySQL 的 SQL 反模式和分片幻觉的详细批判。

### 黄金铁律：计算靠近数据

在计算机科学中，有一个万古不变的黄金铁律：**永远要把计算移动到数据所在的地方**，而不是把海量数据通过网线搬运到计算所在的地方。Redis 的设计完全违反了这个原则。

### 应用端 Join 的灾难

Redis 只支持 Key-Value 和基础集合（Set/List/Hash）。它**无法处理数据之间的关系（Relations）**。

如果你的业务对象稍微复杂一点（例如：获取一个用户、他的订单、订单里的商品、以及商品的当前库存），Redis 会迫使你在代码里进行 **N+1 次网络查询**：

1. 你得先查出用户的 Key，拿到一堆订单 ID。
2. 在代码里写个循环，去 Redis 里挨个或者批量查出这堆订单。
3. 拿到订单后，再在代码里解析出商品 ID，再去发网络请求查商品和库存。

即使你用了 Pipeline 把请求打包，你的应用服务器依然在做一件低效的工作——在内存中反复反序列化各种格式的响应，然后用 CPU 拼装成业务对象。**你在应用端实现了一个缺乏优化器的"伪查询引擎"。**

而在 SQL 的世界里，一个优雅的声明式 JOIN 或者 `JSON_AGG`，数据库引擎直接在**紧贴裸金属的共享内存**里用极速完成了关系的编排，通过网线只吐给你一条清爽的最终 JSON 结果。

### 数据库即应用服务器：计算下沉的降维打击

**PostgreSQL 内嵌高级语言**：Postgres 早就不是一个死板的存储盘，它是一个**全能的计算平台**。通过允许在数据库内部嵌入非受限的 Python 运行环境（`plpython3u`）、V8 引擎（`plv8`）或者原生的 PL/pgSQL，开发者可以直接在数据库内部执行极其复杂的业务逻辑、数据清洗、正则匹配甚至 AI 推理。**数据和代码在同一个内存空间里打交道，网络延迟和序列化开销直接降到了绝对的 0。**

**SurrealDB 的革命性查询（SurQL）**：SurrealDB 原生融合了图数据库（Graph）、文档数据库（Document）和关系数据库（Relational）。它的高效查询语言支持直接通过 `->like->product` 这样的图语义，在单次声明式查询中直接横穿极其深层的嵌套关系。**你根本不需要分多次查询，也更不需要什么外部缓存。** → 详见 [SurrealDB: 统一数据层的架构哲学](./unified-data-layer.md)

### 2010 年代 NoSQL 狂热的集体退潮

2010 年代的"NoSQL/多语种持久化（Polyglot Persistence）"热潮已经退去，技术社区正在重新审视当时的架构选择。

当年的愿景——"一个项目里使用五个数据库：缓存用 Redis、图用 Neo4j、文档用 Mongo、搜索用 Elastic、关系用 MySQL"——在实践中演变为：
- 分布式事务频发故障
- 数据同步延迟难以消除
- 运维复杂度指数增长
- 基础设施成本持续攀升

通过坚定地拥抱 PostgreSQL 强悍的 JSONB 聚合能力/内嵌计算生态，或者选择 SurrealDB 这种原生多模态引擎，你在架构上实现了最高级别的**"奥卡姆剃刀"式的极简**：

- 没有网络多轮延迟
- 没有序列化内耗
- 没有 Lua 地雷
- 没有多库同步

**单次请求，单次返回，直达本质。**

## 8. 替代方案

Redis 唯一剩下的合理性是**惯性**——来自 PHP 时代的历史约束。

1. **用于缓存/状态**：**Fjall**（嵌入式 LSM-Tree KV，Rust 原生）— 完整架构见 [Fjall + Openraft ADR](fjall-openraft-adr.md)。
2. **用于分布式状态**：**Fjall + Openraft**（嵌入式 Raft 共识）— 本批判的建设性对应物。在 L0（进程内 KV）和 L3（强一致性分布式协调）两个层面替代 Redis。
