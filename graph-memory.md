# 图谱化记忆

*记忆架构的计算时机光谱，以及属性图在 Agent 记忆系统中的定位。*

### 尾提示词

详见 [尾提示词](tail-prompt.md)。核心：注入 prompt 末尾的临时指令，利用 KV cache 旁路分支，turn 结束后丢弃。压缩场景中触发工具调用后旧历史直接丢弃。

## 计算时机光谱

记忆方案的本质区别不在数据结构，在**计算发生在哪个阶段**。

```
写时重 ─────────────────────────────────── 读时重

LLM Wiki        KG (属性图)      KV 模式        向量模式
记忆宫殿        SurrealDB         SurrealDB       SurrealDB + RAG
                (graph records)   (KV records)    (embedding index)

写: 整理/嵌入/拓扑推断      写: 简单切分/存储
读: 直接载入/遍历           读: 匹配/排序/向量搜索
```

光谱上的位置不代表不同的存储产品——SurrealDB 的多模型架构覆盖全部区间。位置取决于**使用方式**：graph record 走写时重，KV record 走读时重，embedding index 在中间。

### 写时计算（Write-Time Heavy）

写入阶段投入大量算力整理、嵌入、建立索引。读取时直接载入，零匹配开销。

**代表**：LLM Wiki、记忆宫殿、预计算知识图谱。

**优势**：读取确定性高，不会漏。读一次就是完整答案，不需要搜索排序。

**代价**：写入成本高。每次更新需要重新整理拓扑关系、重新嵌入。增量更新困难——加一条新知识可能影响全局索引。

**适用**：知识变化频率低、读取频率高的场景。文档型知识库、项目架构决策记录。

### 读时计算（Read-Time Heavy）

写入阶段只做简单存储（切分、向量化）。读取时通过匹配算法动态检索。

**代表**：向量检索（RAG）、语义搜索。

**优势**：写入成本极低。新知识即写即用，不需要重建索引。

**代价**：读取是概率性的。向量匹配可能漏掉语义相关但词汇不匹配的知识。检索质量依赖 embedding 模型和 query 质量。

**适用**：知识变化频率高、读取需要模糊匹配的场景。对话历史、代码片段检索。

### 中间区域：属性图（Property Graph）

写入时建立实体关系和属性，但不做深度拓扑推断。读取时支持精确遍历和路径查询，但不需要向量匹配的计算开销。

**代表**：SurrealDB（graph records）、属性图数据库、带权重的三元组存储。

**优势**：写入成本适中（建立关系即可），读取确定性高（图遍历是确定性的）。支持多跳推理（A → B → C 的传递关系），这是向量检索做不到的。

**代价**：实体对齐和去重需要额外逻辑。跨文档的同名异义问题需要消歧。

**适用**：知识有明确结构关系、需要推理链的场景。架构决策链、因果关系追踪、技术选型排除链。

## 光谱的本质：索引时机

三种方案的区别可以压缩为一个变量：**索引什么时候建**。

| 方案 | 索引时机 | 索引内容 | 读取方式 |
|:--|:--|:--|:--|
| Wiki / 记忆宫殿 | 写入时 | 全文 + 结构 + 拓扑 | 直接载入 |
| 属性图 | 写入时 | 实体 + 关系 + 属性 | 图遍历 |
| 向量检索 | 读取时（隐式） | embedding 向量 | ANN 搜索 |
| KV Cache | 写入时 | 键值对 | 精确查找 |

"写时重"意味着写入时就把索引建好，读取时直接查索引。"读时重"意味着写入时只存原始数据，读取时动态建索引（向量化 + 近似搜索）。

**索引时机决定了更新成本和查询质量的权衡**：

- 写时索引：更新成本高（改一条要重建索引），查询质量高（索引是精确的）
- 读时索引：更新成本低（写入即完成），查询质量有上限（embedding 模型的天花板）

## 属性图在 Agent 记忆中的定位

Agent 记忆需要同时满足两个矛盾的需求：

1. **精确性**：架构决策链不能模糊匹配。"我们拒绝了微服务拆分"必须被精确检索，不能因为语义相似返回"我们推荐了微服务"。
2. **时效性**：知识在变化。新的决策会覆盖旧的，权重需要动态调整。

向量检索满足时效性但不满足精确性。Wiki 满足精确性但不满足时效性（更新成本高）。属性图在中间——写入时建立精确关系，读取时通过权重动态排序。

### 文档 vs 原子三元组

文档化记忆有三个结构性缺陷，原子三元组逐一解决：

**语义漂移**：文档被反复修改后，主题偏移。原本指向它的链接/引用还在，但目标已经不是原来那个东西了。一篇文档写的时候是"A 方案的评估"，改了几次变成"B 方案的实施记录"，其他文档里"A 方案的评估"的引用永远指着错误的目标。原子三元组不存在这个问题——边是不可变的，新知识 = 新边，旧边的语义位置永远不会变。

**粒度困境**：粗粒度（整篇文档）信息密度低，80% 内容和当前查询无关；细粒度（段落/句子）退化成 RAG 模式，但没有 embedding 索引，搜索成本比向量检索高。原子三元组卡在最优粒度：一条三元组 = 一个事实，零冗余，零噪声。查询是图遍历（确定性），不需要向量搜索（概率性）。

**关系隐式性**：文档中的关系靠读者推理建立——"A 导致了 B"可能分布在两个段落里。原子化抽取可能丢失隐含关系，但可以通过两种方式缓解：抽取时识别隐含因果链；查询时支持多跳遍历（A → B → C 的传递推理）。

**更新复杂度**：文档更新是原地修改，必须先读完整文档才能找到修改位置。文档随更新增长，每次读取量递增，n 次更新的总读取量是 O(n²)。原子三元组是只追加，新知识直接写入不需要读旧知识，n 次更新总写入量是 O(n)。这不只是性能问题，是正确性问题——文档更新时读到的内容可能已经是过时的（并发修改、缓存不一致），原子追加没有竞态条件。

| 维度 | 文档 | 原子三元组 |
|:--|:--|:--|
| 语义稳定性 | 漂移（修改改变语义位置） | 不变（新知识 = 新边） |
| 信息密度 | 低（大量噪声） | 高（1 fact = 1 triple） |
| 查询方式 | 关键词/向量（概率性） | 图遍历（确定性） |
| 关系显式性 | 隐式（需推理） | 显式（直接存储） |
| 引用准确性 | 随修改退化 | 永久准确 |
| 更新复杂度 | O(n²)（原地修改需读全文） | O(n)（只追加不读旧） |

### 原子三元组的检索困境

原子三元组解决了存储问题，但引入了新问题：**检索成本**。

粒度太细的后果不只是"碎片化"——检索时要遍历大量分散边才能拼出一个完整判断，成本比向量检索还高。一条架构决策可能由 10-20 条边组成，检索时要把这些边全部拉出来拼凑，读取成本随决策复杂度线性增长。

**根本矛盾**：**存储要原子化（精确、无冗余），检索要上下文化（连贯、有语境）。** 同一个粒度无法同时满足两个需求。

### 双层模型

解法不是选"原子三元组"或"摘要文档"，是两层都做——存储层原子化，检索层聚合成簇。

```
写入层（原子）: 一条边 = 一个事实，只追加
↓
组织层（聚簇）: 相关边按实体/事件/主题分组
↓
检索层（视图）: 以簇为单位返回，不返回散装边
```

类比数据库：原始行是原子的，物化视图是聚合的。查询走视图，写入走行。

### 两阶段提取管线

直接从口语/多人对话抽取三元组，信号会被噪声淹没。采用两阶段管线：先规范化，再分类提取。

#### 为什么直接抽取会乱

| 问题 | 例子 | 直接抽取的结果 |
|:---|:---|:---|
| 口语填充词 | "嗯那个…我觉得吧这事儿挺简单" | 三元组里混入 hedging，主语模糊 |
| 指代消解 | "他说那个不行" | "他"是谁？"那个"是什么？无法成三元组 |
| 多人归属 | A说X，B反驳Y，C总结Z | 丢失"谁说的"，事实和观点混在一起 |
| 重复/碎片 | 同一观点说三遍，每次措辞不同 | 三条冗余三元组，去重困难 |
| 隐含逻辑 | "上次那个方案不是没用吗" | 反问句→否定判断，直接抽取丢失语义 |

#### 阶段1：规范化（不是总结）

关键：**规范化是保义的（lossless），总结是损义的（lossy）**。

规范化做：
- ✅ 指代消解（他→张三）
- ✅ 去重合并（三遍同一观点→一条）
- ✅ 归属标注（A说的 vs B说的 vs 共识）
- ✅ 口语→书面（"这事儿挺简单"→"该任务复杂度低"）
- ✅ 分类标注：事实 / 规律 / 逻辑 / 偏好

规范化**不做**：
- ❌ 总结（"他详细说了三点理由…"→不能丢掉三点只保留一句）
- ❌ 改写语义（不能把"可能"变成"确定"）
- ❌ 过度过滤（看似无意义的口语可能携带语气/态度信号）

#### 阶段2：分类提取

不同类别用不同提取模板：

| 类别 | 三元组模式 | 例子 |
|:---|:---|:---|
| **事实** | `(entity, relation, entity)` | `(Postgres, supports, pgvector)` |
| **规律** | `(condition, implies, result)` | `(高并发+写密集, implies, 用PG不用SQLite)` |
| **逻辑** | `(premise, therefore, conclusion)` | `(cognee不支持SurrealDB, therefore, 统一栈选Postgres)` |
| **偏好** | `(user, prefers, thing)` | `(user, prefers, 保守操作)` |

偏好的权重处理特殊：`tool_invoke_count` 不适用（偏好不是工具调用），但 `read_count` 会很高（每轮注入）。偏好的 `time_decay` 应该更慢——偏好是稳定的。

#### 什么时候跳过阶段1

不是所有输入都需要规范化。输入已经是结构化的（代码、配置、正式文档、wiki），直接进阶段2：

```
输入分类器（轻量，可以是规则不是LLM）
  ├── 结构化输入 → 跳过阶段1 → 直接提取
  └── 非结构化输入 → 阶段1规范化 → 阶段2提取
```

#### 成本

两阶段 = 2次 LLM 调用。但：
- 阶段1可以用小模型（规范化是语法层面的，不需要强推理）
- 阶段2必须用强模型（提取三元组需要语义理解）
- 如果输入短（<500字），可以合并成一次调用，prompt 里分两段输出

#### 与 Cognee 的对照

Cognee 确实有类似的 framing 步骤——它在提取图之前先做一层 data → structured representation 的转换。这个两阶段管线的思路和现有实践一致。详细的Cognee 技术分析和与本文档的逐项对照见 [Agent 记忆选型](agent-memory.md)。

### 三种聚簇策略

#### 1. 实体锚定（Entity-anchored）

所有提及实体 X 的边，天然挂在 X 节点上。检索时找实体 → 返回它的邻域子图 → 子图内是连贯的上下文。

```
实体节点 "NixOS 模块归属"
├── 边: 按功能划分, 不按实现语言
├── 边: lsp-bridge 放 system/units/
├── 边: pop-launcher 合并到 cosmic.nix
```

适用场景：查询"X 相关的所有事实"。类比图邻域查询。

#### 2. 时间线聚合（Temporal-clustering）

同一事件/决策的所有边，按时间窗口分组。解决"某个决策当时考虑了什么"这类查询。

```
事件节点 "2026-07 NixOS Overlay 架构重构"
├── 边: 之前用散落目录 → 维护成本高
├── 边: 引入 developMode 门控
├── 边: vivaldi.nix 作为标准模板
└── 边: 写入 ADR-011 记录决策
```

检索时按事件 ID 拉出完整因果链，而不是从二十个散装边里拼凑。

#### 3. 因果链聚合（Causal-chain clustering）

按"为什么"的推理链分组。不是按实体（谁）、不是按时间（何时），而是按因果逻辑（为什么）。

```
判断节点 "SurrealDB 架构选型"
├── 边: SurrealDS = 分布式事务引擎
├── 边: RocksDB = 单节点存储
├── 边: 查询层与存储层分离 → 对上层透明
├── 边: Object storage = Scale 付费特性
└── 边: BSL 许可证 → 社区版可用但无 S3
```

这条链的每条边单独检索出来都意义有限，但作为因果链放在一起，就是一个完整的技术判断。

### 策略选择

| 策略 | 检索模式 | 类比 |
|:--|:--|:--|
| 实体锚定 | "X 相关的所有事实" | 图邻域查询 |
| 时间线 | "那个事件前后的决策" | 时序索引 |
| 因果链 | "这个判断的完整推理" | 物化视图 |

实际操作中，一条边可能同时属于多个簇（同一事实既挂在实体上，又在因果链里）。这不是冗余——是同一个原子在不同视图中的投影。

**核心洞察**：存储原子化保证不冗余，检索聚簇保证有上下文。聚簇是视图不是副本，所以不违反原子化原则。

### 权重系统

每条边携带系统级元数据：

```
业务属性（relation_props）:  since:2011, scope:global, reason:xxx
系统属性（weight_meta）:     read_count, write_count, last_read, last_write
```

两层不混合。业务属性描述"这条知识说了什么"，系统属性描述"这条知识有多重要"。

#### 读写比矩阵

read_count 和 write_count 构成二维坐标，定义四类知识：

```
                    写多（流行）
                       │
         噪声区        │       核心区
      read少 write多    │    read多 write多
      热门但没人查      │    反复确认反复使用
      → 考虑降权        │    → 最高权重
                       │
   ────────────────────┼────────────────────
                       │
         冷门区        │       深水区
      read少 write少    │    read多 write少
      低价值            │    有人查但门槛高
      → 降级为冷数据    │    → 高权重，稀缺性加成
                       │
                    写少（深度）
```

| 象限 | read | write | 含义 | 处理 |
|:--|:--|:--|:--|:--|
| **核心区** | 多 | 多 | 反复使用、反复确认的知识 | 最高权重，优先返回 |
| **深水区** | 多 | 少 | 有人查但写入门槛高（专家知识、深水陷阱） | 高权重 + 稀缺性加成，不惩罚低写入 |
| **噪声区** | 少 | 多 | 写入频繁但无人查询（过时决策、无效记录） | 降权，标记为噪声候选 |
| **冷门区** | 少 | 少 | 低价值知识 | 降级为冷数据，脱落到 S3 |

**关键洞察**：写入少不代表不重要。深水区的知识（如特定技术栈的深水陷阱）写入门槛高——只有踩过坑的人才会写，但每次被查询都是高价值命中。传统权重公式用 write_count 惩罚这类知识，读写比矩阵纠正了这个偏差。

**读写独立性的保证**：原子化三元组天然解耦读写。文档模式下读和改是同一条操作链（read++ write++ 绑定），导致读写正相关，2D 矩阵退化为 1D。原子图模式下读 = 遍历查询现有边（read_count++），写 = 新增节点/边（write_count++）。两件事操作不同的物理对象，existing 边不受新操作影响。这是原子化带来的架构优势——四象限分类在物理层就是稳定的。

#### 权重计算

```
weight = read_score + write_score + rarity_bonus - time_decay

read_score  = log(1 + read_count)        # 对数衰减，防止头部垄断
write_score = log(1 + write_count) × 0.5  # 写入权重低于读取
rarity_bonus = 1.0 if read_count > threshold AND write_count < threshold else 0  # 深水区加成
time_decay  = days_since_last_access × 0.01  # 缓慢衰减
```

高权重的边在查询时优先返回。低权重的边不删除，但降级为冷数据。

### 写入-读取循环

```
写入: 原始输入 → [阶段1: 规范化+分类] → [阶段2: 分类提取三元组] → Fjall（工作记忆）→ 权重积累
                                                                    ↓ (weight > threshold)
读取: LanceDB（长期记忆）→ 向量 + 图混合检索 → 按权重排序返回
                    ↓ (weight < threshold)
冷存: S3 → 需要时 RAG 检索
```

和双轨记忆架构的生命周期完全对齐。新增的环节是：**权重决定了三元组在双轨中的位置**，而不是简单的"新=热，旧=冷"。

#### 并行提取机制（⚠️ 过时，被Prefix Checkpoint 方案替代）

> 此方案要求对 Agent 进行改造（在 system prompt 中指示 LLM 同时输出回复和抽取三元组的函数调用），存在三个问题：(1) 需要改造 Agent 框架；(2) 函数调用可能干扰 LLM 对用户回复的注意力；(3) 没有考虑 KV cache——每轮都引入新的函数调用 token，cache 命中率低。

记忆更新通过 LLM 的并行函数调用在后台发生。模型在一次响应中输出一组工具调用，Hermes 解析后异步执行。用户看到流畅的文本回复，系统在后台完成多维状态更新——不需要一个庞大的函数，模型动态编排细粒度工具。

#### 纯追加指令（⚠️ 过时，被Prefix Checkpoint 替代）

阈值到达后，在 prompt 末尾追加压缩指令，LLM 直接输出 checkpoint + memories（不通过 tool call）。

```
Turn 1..N: 正常对话，[checkpoint] + 增量逐条追加到 prompt，KV cache 持续命中
              ↓ 达到阈值（100 条）
Turn N+1:  prompt 末尾追加 "分析以上对话，输出 checkpoint 和 memories"
           → LLM 直接输出结果，指令本身不写入 session
Turn N+2:  新 checkpoint + 新增量，重新开始
```

优势：LLM 注意力完全集中在压缩任务上（不需要同时回答用户），提取质量可能更高。
问题：(1) 需要改 Agent loop（识别特殊输出并处理）；(2) 压缩和回答分开，两次 LLM 调用；(3) 压缩时机不在记忆系统控制下。

#### 原位检查点压缩（Prefix Checkpoint，当前方案）

替代上述两种方案。核心思路：压缩不是单独的 LLM 调用，而是融入 Agent 的正常对话 turn——下一个用户提问时，注入尾提示词（不是 system prompt），Agent 一次 turn 同时完成提取和回答。

**四步闭环**：
1. **拦截与触发**：记忆系统监控 token 长度，达到阈值时修改用户消息注入压缩指令，前文 token 绝对不变，100% 命中 KV Cache
2. **原位函数调用**：主模型在完整上下文基础上，按顺序执行 generate_summary() + create_checkpoint()
3. **正常回复用户**：完成元任务后，继续基于压缩后上下文回答用户原始问题
4. **历史净化**：turn 结束后还原用户消息为原始内容，保证历史可重放、无伪指令

```
Turn 1..N: 正常对话，[checkpoint] + 增量逐条追加到 prompt，KV cache 持续命中
              ↓ 达到阈值（100 条）
Turn N+1:  用户提问 X
           prompt 末尾追加: "先回答用户的问题，然后分析以上对话，
                           调用 memory_store 提取记忆，标记 checkpoint"
           → Agent 一次 turn: 先回答 Y，再 tool_call(memory_store) + tool_call(checkpoint)
Turn N+2:  历史变为 [ckpt_1] + [X] + [Y]，重新开始
```

**未压缩内容在 session 中**：阈值内的消息虽然还没被压缩为 checkpoint 或提取为长期记忆，但原始内容完整保存在 session_messages 表中，Agent 通过增量消息直接看到。不是"丢失"或"滞后"，只是还没被压缩——原始数据始终可用。

**历史纯净性**：尾提示词是临时的——只存在于当次 turn 的 prompt 中，从不写入 session。session 中存储的是 `[checkpoint] + [用户原始消息] + [Agent 回答]`，而不是 `[checkpoint] + [尾提示词 + 用户原始消息] + [Agent 回答]`。turn 结束后，如果 prompt 中修改了用户消息（追加指令），需要还原为原始内容，保证历史可重放、无伪指令。

**优势**：
- 不改 Agent loop——压缩通过正常 tool call 完成，记忆系统完全控制
- 不单独调 LLM——复用 Agent 的正常 turn，answer + compress 共享 KV cache
- 用户体验无感——正常回答用户，同时后台完成压缩和记忆提取
- cache 天然命中——历史部分全部缓存，只有尾提示词 + 用户问题是新 token

**三种方案对比**：

| 维度 | 并行提取（旧） | 纯追加指令（旧） | Prefix Checkpoint（新） |
|:--|:--|:--|:--|
| Agent 改造 | 需要 | 需要 | 不需要 |
| LLM 调用 | 每轮 | 压缩单独一次 | 复用正常 turn |
| 注意力 | 分散 | 集中在压缩 | 分散（但 tool call 机制保障） |
| 控制权 | Agent | 不确定 | 记忆系统 |

### 基础策略：Checkpoint 压缩

Prefix Checkpoint 方案下，压缩融入 Agent 的正常 turn——阈值到达时，注入尾提示词，Agent 一次 turn 完成：

```
100 条消息 → 尾提示词 → Agent 一次 turn:
  1. 回答用户（LLM 直接输出）
  2. memory_store(...)  ← LLM 主动判断，提取长期记忆（flat KV 或 triples）
  3. memory_checkpoint(...) ← Agent 框架执行，纯 DB 操作
```

#### 一次调用双层输出

核心思想不变：一次 LLM 调用同时产出两层输出。只是实现方式从"直接输出 JSON"变为"通过 tool call"：

| 维度 | 旧方案（CheckpointCompressor） | 当前方案（Prefix Checkpoint） |
|:--|:--|:--|
| 触发方式 | 独立 LLM 调用 | prompt 尾提示词 → Agent 正常 turn |
| 输出方式 | LLM 直接输出 JSON（checkpoint + memories） | LLM 回答用户 + tool call（memory_store + memory_checkpoint） |
| 缓存 | 不共享（新 API 调用） | 共享 KV cache（同一 turn） |
| 长期记忆 | LLM 在压缩调用中提取 | LLM 在 tool call 中提取 |

两种方案都是一次调用产出两层，但当前方案复用 Agent 的正常 turn，cache 命中率从 0% 提升到 ~99%。

#### 提取格式演进

| 阶段 | 提取格式 | 输出 |
|:--|:--|:--|
| 当前（Phase 2.5） | flat KV | `{content, category, importance, tags}` |
| 未来（图谱化） | graph triples | `{subject, predicate, object, category}` |

提取机制不变（LLM 通过 `memory_store` tool call），只是输出格式从 flat 变为 structured。

**可视化管线**：bracket 格式（`head[TYPE]RELATION[props]tail[TYPE]`）可确定性转换为 Mermaid 和 Cytoscape.js，不需要 LLM 参与。Mermaid 用 dagre 层次化布局（适合检查三元组提取是否正确），Cytoscape.js 用力导向布局（适合检查聚簇效果）。两者同一输入、两个输出。

#### 缓存利用

Prefix Checkpoint 的缓存优势：checkpoint 不可变，可永久缓存。增量部分每次变化，但历史部分（checkpoint）走 KV cache。

当前方案不需要独立的压缩 prompt —— 压缩通过 Agent 正常 turn 完成，system prompt + few-shot 由 Agent 框架管理，记忆系统不直接控制 prompt 缓存。

#### 提取 Prompt

提取 prompt 定义三元组输出格式：

```
你是一个对话压缩器。分析以下对话历史，输出两部分：

1. checkpoint：用 2-3 句话概括这段对话的核心内容、关键决策和当前状态
2. triples：提取值得长期记住的信息，格式：
   {"subject": "...", "predicate": "...", "object": "...", "category": "fact|rule|logic|preference"}
```

**依赖关系**：triples 依赖 checkpoint——LLM 先理解对话产出摘要，再基于摘要提取三元组。早期方案中两者是独立输出（无依赖），现在合并为一次调用后自然形成依赖：checkpoint 是对话的全局理解，triples 是从中提取的结构化知识。

这是 `memory_store` tool 内部的 prompt 设计，不影响 Prefix Checkpoint 的整体流程。

### 编码场景：设计文档作为代码库 Checkpoint

M 问"有没有更好的方法让 AI 更了解我们的代码，不用每次都搜索"，O 回答"每次改完让他总结/更新设计文档，因为前面的有缓存，边际成本低，下次用的时候不用大量读取了，改的时候也很精确"。

这不是一个 tips——是计算时机光谱在编码场景的具体实例。

**没有设计文档时（读时重）**：每次会话 AI 重新读代码 → 理解 → 执行 → 遗忘 → 下次重新读。O(n) 的代码扫描，理解质量取决于代码可读性和 context window。

**有设计文档时（写时重）**：首次写设计文档（理解+结构化），后续读缓存的设计文档 → 精确执行。首次成本高，后续成本低。

**设计文档 = 代码库的 Checkpoint**：
- **不可变性**：两次修改之间不变 → 可缓存（和 checkpoint 摘要的物理属性一致）
- **捕获决策**：不只是代码结构，还有"为什么这样设计" → 精确指导未来修改
- **增量更新**：改完代码后更新文档的受影响部分 → 维护成本 O(1)

**边际成本递减**：prompt caching 机制下，设计文档一旦进入缓存，后续调用中这部分 token 的计算成本趋近于零。"前面的有缓存，边际成本低"——只有新增的代码变更需要增量处理。

**精确性收益**："改的时候也很精确"——有设计文档时，AI 知道该改哪里、为什么改、改了会影响什么。没有设计文档时，AI 需要从代码反推架构，容易遗漏依赖或误解设计意图。

**与 checkpoint 压缩的同构**：

| 维度 | 对话 checkpoint | 设计文档 |
|:--|:--|:--|
| 写入时机 | 每 N 条消息压缩 | 每次代码修改后更新 |
| 内容 | 对话核心内容+决策 | 代码架构+设计决策 |
| 缓存属性 | 不可变，可永久缓存 | 两次修改间不变，可缓存 |
| 读取方式 | 注入 Agent prompt | 注入 Agent prompt |
| 边际成本 | 趋近于零（缓存命中） | 趋近于零（缓存命中） |

两者是同一个模式的不同实例：**写时投入整理成本，读时享受零匹配开销**。

编码场景中的现有工具分析（CodeGraph 等）见 [Agent 记忆选型](agent-memory.md)。

## 图谱化 Skill

### 问题：Skill 边界的人工困境

Hermes 自主进化写 Skill 时，容易大量重复——Skill 的范围容易重叠，大范围扫描重新划分边界消耗太大也太慢（LLM 扫描所有 Skill 描述的 token 成本不可接受）。

当前 Hermes 的 Skill 管理是**人工划定边界**：人工判断边界 → 写 SKILL.md → 静态加载（每轮注入全量 Skill 简介）。问题是：

- **边界重叠**：LLM 写新 Skill 时不完全知道已有 Skill 的边界，导致重复
- **扫描成本高**：重新划分边界需要 LLM 读所有 Skill 描述，token 成本 O(N) × 频率
- **token 膨胀**：每轮注入全量 Skill 简介导致 context 膨胀

### 工具指数（Tool Index）

核心思路：给每条边加一个**工具指数**——Agent 调用工具时加载的边，`tool_invoke_count++`。

这本质上是 graph-memory 已有的 `read_count` 的**子类**：不是所有读取都算，只有**工具调用触发的读取**才计数。工具调用说明这条知识有**实际执行价值**，不只是"看了一眼"。

```
系统属性（weight_meta）: read_count, write_count, tool_invoke_count, last_read, last_write
```

`tool_invoke_count` 独立于 `read_count`，权重公式给 `tool_invoke_count` 更高系数：

```
weight = read_score + write_score + tool_score + rarity_bonus - time_decay

tool_score  = log(1 + tool_invoke_count) × 1.5  # 工具调用权重最高
read_score  = log(1 + read_count)               # 被动读取
write_score = log(1 + write_count) × 0.5        # 写入权重低于读取
```

### 涌现式 Skill 边界

工具指数高的边自然聚集成一个"频繁使用的操作链"——这就是 Skill 的**隐式定义**。Skill 不是人工划定的边界，是使用数据**涌现出来**的子图。

对比当前 Hermes 的 Skill 管理：

| 维度 | 人工边界（当前） | 涌现式边界（新方案） |
|:---|:---|:---|
| 边界划定 | 人工判断 → SKILL.md | 使用数据自动聚类 → 子图 |
| 加载方式 | 全量注入每轮 | 向量搜索 → 高权重浮出 |
| token 成本 | O(N) × 每轮 | O(K) × 每轮（Top-K） |
| 更新 | 人工重写 | 使用数据自动更新权重 |

**潜在问题**：

1. **冷启动**：新 Skill 还没有使用数据，工具指数为 0，不会被检索到。需要 Bootstrap 机制——新写入的边给一个初始权重（类似 PageRank 的 dangling link 处理）。
2. **Skill 融合**：两个高频子图可能有重叠节点——是合并还是保持分离？需要**社区检测**算法（如 Louvain）来划分自然边界，而不是单靠阈值。
3. **多归属**：同一个工具可能被多个不同 Skill 调用——边的 `tool_invoke_count` 高不代表它只属于一个 Skill。一条边可以属于多个子图（同一个原子在不同视图中的投影，和 graph-memory 的聚簇设计一致）。

### 渐进式加载

不用放所有 Skill 的简介，而是直接搜索（向量/文本），结果中工具指数高的自动作为 Skill 部分。

这解决了 Hermes 当前的问题：每轮注入全量 memory + 所有 Skill 简介 → token 膨胀。改为向量搜索 → 按权重排序 → 只注入 Top-K 高权重边。

前提：Skill 的描述本身需要被原子化入库。当前 SKILL.md 是 markdown 文档，需要拆成原子三元组入库后才能参与向量搜索。

### 脚本管理：两种方案

#### 方案 A：脚本作为节点，SKILL 描述指向它

每个脚本作为一个节点（包含调用方法），SKILL 描述中的一部分指向它。这是**引用模式**——Skill 是元数据（描述+指针），脚本是实体（代码+调用方法）。

```
[Skill 节点] --references--> [Script 节点]
  ├── 边: 描述（自然语言）
  ├── 边: 触发条件
  └── 边: 指向脚本节点（lang, entry_point, params schema）
```

和 aura 的 `set()` + `interface_schema()` 设计兼容——脚本存 Fjall `actor_defs` 分区，Skill 描述存图节点，指向脚本节点的 `actor_type_name`。

#### 方案 B：脚本拆成原子化 Actor，AI 串联

把脚本本身再拆分成原子化的 Actor，AI 把它们串联起来，适配 Aura 架构。每次执行生成一个场域。Aura 的设计可能需要重新审视——不是监听事件了，而是按引用调用。

> ⚠️ **待定**：此方案涉及 Aura 架构的调整，需要继续讨论。核心问题是：Aura 的 `emit`/`on` 是事件驱动（pub-sub），"按引用调用"是命令式（RPC），两者如何共存。

## 与 LLM Wiki 的关系

LLM Wiki 是写时计算的极端——人工整理，结构固定，读取确定。它是属性图的"人工预热版"。

当 Wiki 的内容被抽取为原子三元组并入库后，它从写时计算迁移到了中间区域：

- 写入时：Wiki 原文 → 三元组抽取 → 入库（一次性成本）
- 读取时：图遍历 + 权重排序（持续收益）

**迁移的收益**：Wiki 的知识获得了动态权重，能根据使用频率自动调整重要性。不再需要人工判断"哪条知识更重要"——使用数据会告诉你是哪条。

**迁移的成本**：三元组抽取的精度损失。散文中的隐含关系可能在抽取时丢失。这是从写时计算迁移到中间区域的固有代价。
