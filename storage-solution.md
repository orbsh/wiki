# 存储方案讨论总结

## 一、文件系统选择

| 场景 | 推荐 | 原因 |
| :--- | :--- | :--- |
| **K8s 服务器（系统盘）** | XFS | 大磁盘不需要 fsck、并发 I/O 优秀、容器生态默认 |
| **K8s 服务器（数据盘）** | XFS + `autoFormat` | 新盘自动格式化、已有盘不受影响 |
| **桌面工作站** | ext4 或 xfs | zram 已提供内存缓冲，文件系统选稳定即可 |

### XFS vs ext4 vs btrfs

*   **XFS**：服务器/k8s 首选。AWS EKS、RHEL、Fedora CoreOS 默认。无 fsck 等待时间，大文件和大磁盘性能优秀。缺点是不能缩容（服务器场景不需要）。
*   **ext4**：发行版保守默认。兼容性最好、支持缩容、什么场景都能用但不突出。适合不知道选什么的时候。
*   **btrfs**：桌面端快照曾是其杀手特性，但在服务器场景问题明显：
    *   COW 写放大，数据库/etcd 性能差。
    *   碎片化严重，需定期 `btrfs balance`。
    *   etcd 官方不推荐。
    *   日常维护负担大。

## 二、为什么 btrfs 在当前体系中变得尴尬

### 桌面端（NixOS）
*   NixOS 本身提供完整的系统回滚（`nixos-rebuild --rollback`），按代管理配置。
*   Home Manager 声明式管理 dotfiles。
*   文件系统快照的“保险”功能已经被 NixOS 的代际管理完全替代。
*   剩下只有 COW 的性能开销和维护成本。

### 存储端
*   过去用 btrfs/RAID/ZFS 是为了“自己扛可靠性”——冗余、快照、校验、自愈。
*   现在对象存储（S3）解决了分布式、冗余、按需扩展的问题。
*   本地磁盘阵列：维护成本高、换盘重建慢、扩容不灵活。

## 三、当前存储架构理念

```text
┌──────────────────────────────────────────────────┐
│ 应用层                                            │
│ K8s Pods / 容器 / 服务                            │
├───────────────────────────────────┬──────────────┤
│ 本地磁盘 (XFS)                    │ 对象存储 (S3)│
│ ┌─────────────────┐               │ ┌──────────┐ │
│ │ 系统盘 vda      │               │ │ 备份/归档│ │
│ │ bios_boot 1M    │               │ │ 静态资源 │ │
│ │ root 100% XFS   │               │ │ 跨节点数据│ │
│ └─────────────────┘               │ └──────────┘ │
│ ┌─────────────────┐               │              │
│ │ 数据盘 vdb      │               │ 需要可靠性/  │
│ │ /home/u/data XFS│               │ 共享时用 S3  │
│ └─────────────────┘               │ 不自己扛     │
└───────────────────────────────────┴──────────────┘
```

### 原则
1.  **本地磁盘 = 纯粹的块设备**：跑容器、存临时数据，坏了就换。
2.  **需要可靠性的数据 → S3**：冗余和扩展交给云厂商。
3.  **文件系统选“傻快”的**：不需要高级特性（快照/COW/校验），XFS 最合适。
4.  **分区越少越好**：BIOS boot 1M + 根分区 100% 就够了。
