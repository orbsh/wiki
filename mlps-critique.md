# 等保标准批判——与现代安全实践的差异

**审计对象**: jihua1 (39.105.33.191) / jihua2 (39.107.236.112)
**评估日期**: 2026-07-10
**参考标准**: NIST SP 800-63B (2017)、CIS Ubuntu 22.04 Benchmark、云原生安全最佳实践

---

## 1. 检查命令过时

等保检查清单使用 `cat /etc/redhat-release` 检查系统版本。这是 CentOS/RHEL 时代的产物，对 Ubuntu 服务器不适用。

现代 Linux 标准是 `cat /etc/os-release`（LSB 标准），所有主流发行版通用。等保标准制定时市场以 CentOS 为主，未跟上发行版生态变化。

---

## 2. 密码轮换策略与国际标准冲突

等保要求密码 90 天轮换。**NIST SP 800-63B (2017) 已明确建议取消强制密码轮换**：

> Verifiers SHOULD NOT require memorized secrets to be changed arbitrarily (e.g., periodically).

强制密码轮换的实际效果：
- 频繁换密码 → 用户选弱密码（"Password123" → "Password124"）
- 写在便签上 → 物理泄露风险
- 用模式记忆 → 可预测

现代做法：密码设长（≥12位）+ 不轮换 + 异常检测（异地登录告警）。

---

## 3. 访问控制检查项不适用于云环境

等保要求检查 `/etc/hosts.allow` 和 `/etc/hosts.deny`（TCP Wrappers）。

云环境实际情况：访问控制在云服务端（安全组/ACL）控制，不在本机。本机配置 TCP Wrappers 是冗余的——流量在到达服务器之前已被云安全组过滤。TCP Wrappers 是 1990 年代的产物，现代 Linux 发行版已不推荐使用。

---

## 4. 防火墙检查项不适用于云环境

等保要求"配置防火墙"，但没有区分云环境和传统环境。

### 4.1 本机防火墙无法应对 DDoS

DDoS 不是"绕过防火墙"，是用海量流量淹没目标。即使防火墙规则正确配置（DROP 所有非法流量），每个包仍要经过内核网络栈处理：

```
攻击流量 → 网卡 → 内核协议栈 → iptables 规则匹配 → DROP
                              ↑
                         CPU 在这里消耗
```

单核 CPU 处理 SYN 包的能力约 50-100 万pps。一个中等规模 DDoS（10Gbps）就能把 CPU 打满——即使所有包都被 DROP，CPU 也在忙着"拒绝"，无法处理合法流量。

本机防火墙的物理极限：
- 单 IP 暴力破解 → ✅ 可以（fail2ban 封 IP）
- 小规模扫描 → ✅ 可以
- 中等 DDoS（1-10Gbps）→ ❌ CPU 被包处理耗尽
- 大规模 DDoS（10Gbps+）→ ❌ 网卡先被打满，iptables 根本没机会执行

瓶颈不在防火墙规则，在物理带宽和 CPU。

### 4.2 云防火墙在虚拟网络层工作

云安全组工作在 VPC 虚拟网络层，在物理网卡之前：

```
互联网 → 云骨干网 → 安全组过滤 → 虚拟网卡 → 物理网卡 → 内核 → iptables
                          ↑
                     这里丢弃非法流量
                     不消耗服务器 CPU
```

云防火墙的优势：
1. **带宽吸收**：云骨干网带宽远超单机（Tbps 级别），DDoS 流量在骨干网被清洗
2. **零 CPU 开销**：安全组在 hypervisor 层执行，不消耗客户虚拟机的 CPU
3. **分布式清洗**：大型云厂商有全球 CDN + DDoS 清洗中心，攻击流量在边缘被丢弃
4. **弹性扩展**：云防火墙自动扩容，不会因为流量增长而崩溃

### 4.3 有云防火墙后本机防火墙是累赘

| | 有云防火墙 | 无云防火墙 |
|:--|:--|:--|
| 流量到达本机时 | 已被安全组过滤，基本是合法流量 | 包含大量攻击流量 |
| iptables 的工作 | 几乎无事可做 | 疯狂 DROP，CPU 被消耗 |
| 性能影响 | 零 | 严重（CPU 100%，服务不可用） |
| 是否需要 | ❌ 冗余 | ⚠️ 必须有，但扛不住 |

本机防火墙在云环境中的问题：
1. K8s kube-proxy 依赖 iptables 做服务路由，自定义规则可能干扰 K8s 网络
2. 增加运维复杂度（规则管理、调试、排错）
3. 对安全无实际贡献（云安全组已覆盖）
4. 在 DDoS 场景下无法提供保护（瓶颈在带宽/CPU，不在规则）

反过来看，云安全组默认全关，要开 10250（kubelet API）这种 K8s 内部端口还得有专业知识。知道这是什么端口还去开，那是配错了——没有合理理由暴露 kubelet API 到公网。这恰恰说明云安全组才是真正的安全边界：本机防火墙只是第二道冗余，而安全组是唯一需要认真配置的地方。

## 5. 内核更新的风险收益不对等

等保标准隐含要求系统保持最新补丁。但内核更新的实际收益有限：

1. **云防火墙是第一道防线**：攻击路径在云边界被阻断，无法触达就无法利用内核漏洞
2. **SSH 在 VPN 之后**：SSH 不直接暴露给公网，攻击面极小
3. **容器隔离**：业务运行在容器中，即使代码有漏洞，爆炸范围被容器边界限制
4. **更新有业务中断风险**：K8s 节点升级内核需要 drain + reboot，可能导致 Pod 迁移和短暂服务中断

当前两台服务器内核版本为 5.15.0-56（2022-11），已近 4 年未更新。在云防火墙 + VPN + 容器隔离的多层防御下，内核更新的边际安全收益低于业务中断风险。

---

## 6. 审计要求与容器化环境不匹配

等保要求运行 auditd 审计服务。但 K8s 环境中：
- K8s apiserver 有自己的 audit log（记录所有 API 调用）
- 容器内行为由容器运行时（CRI-O）记录
- auditd 记录的是宿主机系统调用，对容器化业务价值有限

等保标准制定时以传统物理机/虚拟机为假设，未考虑容器化部署模式。

---

## 总结

等保标准的核心问题：**以传统物理机/虚拟机环境为假设，未适配云原生架构**。

| 等保要求 | 传统环境 | 云环境 |
|:--|:--|:--|
| redhat-release 命令 | ✅ 适用 | ❌ 应用 os-release |
| 密码 90 天轮换 | ⚠️ 有争议 | ❌ NIST 不建议 |
| hosts.allow/deny | ✅ 适用 | ❌ 云安全组替代 |
| 本地防火墙 | ✅ 必须 | ❌ 云安全组替代 |
| auditd 审计 | ✅ 必须 | ⚠️ K8s audit log 覆盖 |
| 内核最新补丁 | ✅ 必须 | ⚠️ 多层防御下收益有限 |
