# MySQL：关系型遗留物与 SQL 反模式

**结论**：新架构中拒绝作为主数据存储。
**状态**：初期是设计者在数据库领域的业余性，后期被 Oracle 收购后变为商业策略——功能缺失不是做不了，是为了卖云服务。

## 执行摘要

MySQL 的流行掩盖了两个层面的问题。**初期（2009 前）**：设计者在数据库理论上的业余——GROUP BY 逻辑错误、术语偏离 SQL 标准、隔离级别欺骗，是设计者对集合论、事务模型和存储引擎架构缺乏基本理解的体现。**后期（2010 Oracle 收购后）**：功能缺失从能力问题变为商业策略——JSON 拖到 8.0，向量搜索至今没有。不是做不了，是 MySQL 功能越强，用户越不会迁移到 Oracle Cloud 或购买 Oracle 商业许可。

SQLite 用 10 万行代码实现了窗口函数、CTE、FTS5、JSON1、R-Tree。MySQL 8.0 加入同样的特性花了数百万行代码，且仍未完全达到同等水平。这不是工程量的差距，是设计能力的差距。

*与 Go 的类比：Rob Pike 的声望掩盖不了 Go 在类型系统、泛型、错误处理上的业余。MySQL 同理——流行度不等于技术深度。*

## 1. 类型系统缺陷

### 用字符串存储时间是极其业余的

MySQL 常见的反模式：用 `VARCHAR` 或 `TEXT` 存储 ISO 8601 时间戳，理由是"兼容性好"或"排序没问题"。后果：
- 无法直接做时间运算（加减月份、差值计算需要先解析）
- 索引效率低于原生类型（字符串比整数/时间类型占用更多存储）
- 每次使用都要解析成时间对象，增加运行时开销
- 语义不清晰，违反最小惊讶原则

**变种：Unix Epoch**。用 `BIGINT` 存储 Unix 时间戳稍好——存储和排序开销正常了，但可读性差、时区处理需要应用层介入、`WHERE created_at > NOW() - INTERVAL 7 DAY` 这样的查询变成手动计算。仍然是**应用层补偿数据库类型系统缺陷**的思路。

这不是"可以接受的变通"，这是**设计缺陷**。任何数据库（MySQL、PostgreSQL、SurrealDB）都应使用原生时间类型。

### ENUM 的字符串陷阱

MySQL 的 `ENUM` 存储为整数，但比较时隐式转换为字符串。这导致：
- `WHERE status = 'active'` 走索引，`WHERE status = 1` 不走
- 迁移时修改 ENUM 值需要 `ALTER TABLE`（锁表）
- 应用层类型与数据库层类型不一致

## 2. GROUP BY 逻辑错误

`GROUP BY` 操作将行集合转换为分组。输出必须与**分组**结构兼容：

1.  **分组键**：包含在 `GROUP BY` 中。组内所有行的值相同。可安全选择。
2.  **聚合**：传递给 `SUM()`、`MAX()` 等。组被缩减为单个标量。可安全选择。

**MySQL 反模式**：允许 `SELECT` 中的列**既不**分组**也不**聚合。

*   **逻辑错误**：你要求从*不同值的集合*中取*一个*值。哪一个？
*   **问题**：引擎任意选择一个值（通常是它遇到的第一行）并将其呈现为整个组的答案。
*   **本质**：这是语义类型错误。它试图将 `Set` 转换为 `Scalar` 而没有归约规则。它暗示单一真相，而实际上有多个。

这不是对"MySQL 不是合格的数据库引擎"的批评。这是对设计者对逻辑的基本理解的批评。如果你无法区分值的集合和单个值，你就缺少基本的集合论推理。这个"特性"多年来被庆祝为"便利"，这表明"行业标准"的门槛有多低。

虽然 MySQL 5.7+ 添加了 `ONLY_FULL_GROUP_BY` 来修复这个问题，但损害已经造成："宽松 SQL"的文化持续存在，引擎数十年来一直启用这个反模式。

## 3. 查询优化器的愚蠢

现代引擎（ClickHouse、SurrealDB）使用基于成本的优化（CBO）和动态规划来找到最佳执行计划。MySQL 的优化器以脆弱著称：

### 连接顺序无能
*   **问题**：优化器经常无法高效重排 `JOIN`，特别是多表时。它可能选择大表作为驱动表，强制对小表进行嵌套循环扫描。
*   **Hack**：开发者必须使用 `STRAIGHT_JOIN` 手动干预强制顺序。如果数据库引擎无法找出最佳顺序，这是优化器的根本性失败。

### 索引僵化（左前缀规则）
*   **问题**：复合索引如 `INDEX(a, b, c)` 严格受左前缀规则约束。过滤 `WHERE b = 1 AND c = 2`（跳过 `a`）的查询无法使用索引，回退到全表扫描（忽略较新的、有限的 Index Skip Scan 特性）。
*   **批评**：现代自适应引擎通过位图索引或自适应扫描处理乱序列过滤。MySQL 的僵化 B-Tree 实现迫使模式设计者提前数年猜测未来的查询模式。

### "Count"扫描
*   **问题**：InnoDB 表上的 `SELECT COUNT(*)` 通常需要全索引扫描，因为 InnoDB 不维护缓存的行数（由于 MVCC 可见性规则）。
*   **批评**：虽然对严格一致性在理论上合理，但在实践中意味着对大表的简单元数据操作极其缓慢。依赖全扫描进行基本元数据查询在现代高并发环境中是不可接受的。

## 4. 术语 vs. SQL 标准

MySQL 的命名约定经常偏离 ANSI-SQL 标准和数据库理论。这不仅仅是语义——它反映了学术严谨性与早期工程实用主义之间的差距。

### 4.1 WAL vs. Redo Log

| 术语 | 范围 | 描述 |
|------|------|------|
| **WAL**（Write-Ahead Logging，预写日志）| 理论/通用 | 原则：日志必须在数据页*之前*持久化到磁盘。由 Jim Gray 在《Transaction Processing: Concepts and Techniques》中提出。PostgreSQL、Oracle、SQL Server 等使用。|
| **Redo Log**（重做日志）| MySQL/InnoDB 实现 | InnoDB 的具体 WAL 机制。命名为"Redo"以与 **Undo Log** 配对——在崩溃恢复期间，InnoDB 从重做日志*重做*已提交的更改，然后通过 Undo Log *撤销*未提交的事务。|

**为何不同？**
- InnoDB 将 WAL 实现为具有特定恢复角色（"重做"）的循环缓冲区。名称反映*动作*而非*原则*。
- MySQL Server 层还有 **Binlog**（逻辑的，用于复制），它也是 WAL 的一种形式。将两者都称为"WAL"会产生歧义，因此 InnoDB 的物理日志获得特定名称"Redo Log"。

**底线**：Redo Log = InnoDB 的 WAL 实现。原则是通用的；名称是引擎特定的。

### 4.2 Schema vs. Database

在 ANSI-SQL 和数据库理论中，层次结构是严格的：

```
Catalog（集群/实例）
  └── Schema（逻辑命名空间/包）
        └── Table, View, Procedure...
```

- **Catalog**：顶层容器（通常是整个实例）。
- **Schema**：Catalog 内的逻辑命名空间，用于对象隔离和访问控制。跨 Schema 查询是透明的（例如 PostgreSQL：`SELECT * FROM s1.t1 JOIN s2.t2`）。
- **Database**：理论上指整个物理数据集合（即实例本身）。

**MySQL 的偏离**：`Instance → Database → Table`。无 Schema 层。

**为何？**
- 早期 MySQL（MyISAM 时代）将每个"Database"映射到 `/var/lib/mysql/db_name/` 下的文件系统目录。选择术语"Database"而非"Schema"是为了简单。
- 当 MySQL 5.0 试图与 SQL 标准对齐时，`DATABASE` 已经太根深蒂固无法更改。妥协：`CREATE SCHEMA` 是 `CREATE DATABASE` 的**同义词**——它们产生完全相同的结果。
- 与真正的 Schema（仅逻辑）不同，MySQL Database 绑定到物理存储、文件权限和字符集。跨数据库查询有效但有陷阱（`lower_case_table_names`、权限边界）。

**底线**：MySQL 的"Database"= 标准的"Schema"。MySQL 用户所说的"建库"理论上是"建模式"。

### 4.3 其他术语差距

| 概念 | 标准术语 | MySQL 术语 | 备注 |
|------|---------|-----------|------|
| 隔离：可重复读 | SQL 标准 RR 允许幻读 | InnoDB RR 通过 Next-Key Locks **阻止**幻读 | MySQL 的 RR 严格强于 SQL 标准定义 |
| 聚簇索引 | 不在关系理论中（表是无序集合）| InnoDB：数据与 PK 索引一起存储（"索引组织表"）| InnoDB B+Tree 设计的工程产物 |
| 回滚段 | Rollback Segment / Undo log | InnoDB："Undo Log" | Undo Log 本身作为数据管理，甚至有自己的 Redo Log（递归）|

这些差距存在是因为 MySQL 为早期开发者人体工程学优化而非理论正确性。当迁移到遵循标准 Schema 语义的分布式数据库（TiDB、CockroachDB）时，心智模型必须重新对齐。

## 5. 分片幻觉与聚簇索引执念

### 5.1 聚簇索引执念（"唯索引论"）

InnoDB 是**聚簇索引**（索引组织表）——数据行物理存储在主键 B+Tree 叶页中。这迫使 MySQL 开发者陷入主键选择的执念（经典的自增 vs. UUID 辩论），因为 PK *就是*物理存储顺序。

**现实（Oracle 和 PostgreSQL）**：两者默认使用**堆表**——数据存储和索引解耦。主键只是普通的唯一索引；它对物理行放置没有影响。这意味着：
- 无 PK 执念——任何列都可以作为 PK，对写入模式没有性能后果。
- 多个索引可以同样高效地作为访问路径，没有一个是"唯一"的物理顺序。

### 5.2 分片迷信（"分库分表迷信"）

InnoDB 的聚簇索引在历史版本中导致单表性能在约 1000 万行后下降，MySQL 生态因此发展出对应用层分片（分库分表）的执念——这是一种深度侵入性的架构手术。但 InnoDB 的"性能优势"在现代硬件和替代方案面前已不存在——PostgreSQL 的堆表 + 并行查询在相同硬件上表现更优。

**现实**：Oracle 和 PostgreSQL 通过以下方式处理单表数亿甚至数十亿行：
- **原生声明式分区**——`PARTITION BY RANGE/LIST/HASH`，对查询透明。
- **并行查询引擎**——PostgreSQL 中的 `max_parallel_workers_per_gather` 让单个查询跨分区跨越多个 CPU 核心。

许多在 MySQL 中需要完整分片手术的场景可以通过简单启用并行性和设置水平分区在现代 PostgreSQL 中解决——零应用代码更改。

### 5.3 应用层分片成本

MySQL 被设计为单节点数据库。每次水平扩展的尝试都是变通方案：
```
Request → Middleware → user_id % 1024 → DB-37/Table-12 → Query
```
*   **跨分片 JOIN**：不可能或需要昂贵的应用端组装。
*   **迁移**：手动数据手术。拆分分片需要停止写入、移动数据和更新路由规则。分布式原生系统通过后台重平衡做到这一点。
*   **一致性**：跨分片事务需要 TCC/Saga 模式，比真正的 ACID 更弱。

### 5.4 原生分区 vs. 应用分片

| 维度 | MySQL 分片 | 原生分区（PostgreSQL、Delta Lake）|
|------|-----------|-------------------------------|
| **路由** | 应用/中间件计算哈希 | 引擎通过分区键决定 |
| **跨分区查询** | 手动组装，N 次查询 | 单个 SQL，引擎处理 |
| **重平衡** | 手动，停机 | 自动，后台 |
| **用户体验** | "找到正确的数据库，然后查询" | 写标准 SQL，引擎做剩下的 |

## 6. 存储引擎混淆（"存储引擎的概念混淆"）

**MySQL 的方法**：插件式存储引擎（MyISAM、InnoDB、Memory 等）。这迫使 MySQL 开发者不断辩论"这个表应该用 InnoDB 还是 MyISAM？"——这个决定影响事务支持、锁粒度、崩溃恢复和外键行为的*表级*。

**与同类产品的对比**：PostgreSQL、Oracle、SQL Server 都使用**单一集成存储引擎**，覆盖所有场景（OLTP、JSONB、全文检索、GIS、时序）。MySQL 需要多个引擎才能勉强覆盖这些场景——InnoDB 处理事务，MyISAM 处理只读，Memory 处理临时表。这不是"灵活的取舍"，而是**其他数据库不需要承担的复杂性**。用户被迫在表级做引擎选择，而这个选择在其他数据库中根本不存在。

## 7. 隔离级别欺骗（"隔离级别的瞒天过海"）

### 7.1 RR 作为复制缺陷的掩护

MySQL 的默认隔离级别是**可重复读（Repeatable Read）**——看起来比 PostgreSQL 和 Oracle 默认的**读已提交（Read Committed）**更严格。但这不是为了正确性的设计选择；它是**工程缺陷的掩护**。

早期 MySQL 复制是**基于语句的 Binlog**（SQL 文本复制）。在读已提交下，相同的语句可能在主库和从库产生不同的结果（因为不同的快照看到不同的行）。通过提升到 RR，InnoDB 的 MVCC 快照在事务内变得稳定，掩盖了基于语句复制引起的不一致。

真正的修复后来随着**基于行的复制**到来——但到那时，RR 已经是默认值，生态已经围绕它构建。

### 7.2 可序列化：伪造的序列化

MySQL 的 `SERIALIZABLE` 隔离级别通过**暴力锁定每次读取**（将所有 `SELECT` 转换为 `SELECT ... LOCK IN SHARE MODE`）实现"可序列化"。这在并发下是性能灾难。

现代引擎（PostgreSQL 使用 **SSI——可序列化快照隔离**）通过**并发图中的冲突检测**实现真正的可序列化——无需额外锁。MySQL 完全没有这个。

### 7.3 可重复读："鬼压床"幻读 Bug

MySQL 的 RR 声称防止幻读，但它通过 **MVCC 快照幻觉**实现——不是真正的锁定。快照对*快照读*（普通 `SELECT`）是一致的，但一旦你混入*当前读*（`UPDATE`、`DELETE`、`SELECT ... FOR UPDATE`），幻觉就破碎了：

- **快照读**看到旧的 MVCC 快照（无幻读）。
- **当前读**看到最新提交的数据（幻读出现）。

这创造了 MySQL RR 下著名的**幻读一致性问题**——事务相信它受到幻读保护，但"幻像"行上的并发 `UPDATE` 打破了幻觉。PostgreSQL 的 RR（实际上是 SI——快照隔离）诚实地说明它保证什么；MySQL 的 RR 假装比实际更强。

### 7.4 Hermitage 测试：三记重拳

**Hermitage** 是数据库事务隔离正确性的并发异常形式分类法。结合 Martin Kleppmann 的 **Hermitage 事务测试项目**和 Jepsen 的 **Elle** 黑盒事务测试器，它提供了每个数据库一致性保证的严格、可重现的证据。

当 MySQL 的默认 RR 受到这个框架的审查时，它在三个方面失败：

#### (a) G2-item / G-single：写偏斜失控

在标准 RR 下，数据库必须防止由并发交错引起的**丢失更新**和**读偏斜**。Hermitage 测试暴露了 MySQL 的致命失败：

- **场景（写偏斜）**：两个并发事务各自读取显示"一个值班医生"的快照。两人都决定休假（规则：至少留一个）。两人都看到快照说安全。两人都提交。结果：零个值班医生。
- **MySQL 的行为**：RR 拒绝在普通 `SELECT` 上获取读锁，所以两个事务读取相同的快照并修改变量而没有冲突检测。两人都成功提交——损坏的数据落盘。
- **PostgreSQL 的行为**：PG 的 RR 实现**可序列化快照隔离（SSI）**——它在并发图中检测序列化冲突并以序列化失败中止一个事务。

这不是理论边缘情况。它是任何严肃的 RR 实现必须处理的规范 G2-item 异常。

#### (b) MAV 违规：时光倒流（"时光倒流"）

**MAV（单调原子视图）** 是基本的因果一致性不变量：在事务内，如果你的第一个查询观察到另一个已提交事务的效果，后续查询必须继续看到这些效果（或更新的）——数据绝不能"倒退"。

MySQL 的 MVCC 实现由于 **Read View** 创建和刷新的宽松时序而违反这个：

1. 事务 A 的第一个查询看到事务 B 刚提交的数据（最新快照）。
2. 事务 A 的第二个查询，在某些高并发交错模式下，读取 B 提交*之前*的*陈旧*快照。

这种**"先看到未来，然后回到过去"**在分布式系统文献中被分类为严重的**一致性违规**。这意味着 MySQL 的 RR 甚至不能保证单个事务的现实视图是单调递进的。

#### (c) 丢失更新：快照读陷阱

"八仙图"定义了八种高级并发冲突模型。在**丢失更新**测试中，MySQL 表现出几乎欺骗性的行为：

- 两个事务并发对同一行执行 `salary = salary + 1000`。
- 在严格的 RR 数据库中，第二个事务的读取应该阻塞直到第一个提交，确保正确的累加。
- MySQL 的快照读实现允许两个事务读取相同的陈旧值（例如 5000），独立计算（5000 + 1000 = 6000），并写回。一个事务的更新被静默覆盖和丢失。

修复需要显式的 `SELECT ... FOR UPDATE`——但这是*用户端变通方案*，用于隔离级别*声称*提供的一致性保证。

### 7.5 根因：InnoDB–SQL 层分裂（"美丽的脱节"）

为何 MySQL 的一致性比其文档暗示的差这么多？

根因是 InnoDB 存储引擎层和 MySQL 的 SQL 解析/MVCC 协调层之间的**根本性架构脱节**：

1. **营销 vs. 现实**：MySQL 基于"理想单线程业务流程"宣传其隔离级别——这是生产中从不存在的最佳情况。
2. **黑盒测试暴露**：Elle 和 Jepsen 等工具注入微秒级并发交错，专门设计用于探测事务实现中的因果链间隙。MySQL 的"无锁快照读"——为速度优化——在并发访问下创造巨大漏洞。
3. **分裂**：InnoDB 的 MVCC 快照机制独立于 SQL 层的事务协调运行。SQL 层*认为*它提供 RR 保证，但 InnoDB 的快照语义弱于 RR 要求的。结果：一个*声称*可重复读但实际交付介于读已提交和不可预测行为之间的数据库，取决于并发时序。

**实际含义**：如果你的业务有硬性一致性要求（金融对账、分布式库存、并发余额更新），你**不能信任 MySQL 的名义隔离级别**。你必须：
- 添加普遍的悲观锁（对每个为写入提供数据的读取使用 `SELECT ... FOR UPDATE`）——实质上在应用代码中降级为手动序列化。
- 迁移到 PostgreSQL（基于 SSI 的 RR）、TiDB（真正的分布式事务）或 CockroachDB（默认可序列化）。

Hermitage 测试是棺材上的最后一颗钉子：它们从第一性原理证明 MySQL 的"RR = SI + 幻读幻觉"不仅比宣传的弱——它主动违反甚至基本 SI 实现也保证的因果一致性不变量。

## 8. MySQL 8.0 改进：追赶还是领先？

MySQL 8.0 引入了显著的特性改进：窗口函数、公共表表达式（CTE）、JSON 增强、多值索引、不可见索引、降序索引。这些是真实的工程进步。

但综合来看，**MySQL 8.0 只是达到了其他数据库的基准水平，甚至可能未完全达到**。

**SQLite 的对照**：SQLite 用约 10 万行代码实现了窗口函数、CTE、FTS5 全文检索、JSON1、R-Tree 空间索引。MySQL 8.0 加入同样的特性花了数百万行代码，且在 JSON 支持（TEXT 别名 vs 二进制 JSONB）和向量搜索上仍落后。这不是工程量的差距，是设计能力的差距——10 万行代码的项目在特性上碾压数百万行代码的项目，说明后者的设计效率极低。

**Oracle 收购后的商业策略**：2010 年 Oracle 收购 Sun 后，MySQL 的功能演进明显放缓。JSON 拖到 8.0（2018），向量搜索至今没有官方支持。这不是技术能力问题——Oracle 有 PostgreSQL 级别的工程能力。这是商业策略：MySQL 功能越强，用户越不会迁移到 Oracle Cloud 或购买 Oracle 商业许可。MySQL 的功能缺失是**刻意的产品定位**，不是技术债。

| 特性 | MySQL 8.0 | PostgreSQL 16 | SQLite 3.45 | DuckDB |
|------|-----------|---------------|-------------|--------|
| 窗口函数 | ✅ | ✅ | ✅ | ✅ |
| CTE | ✅ | ✅ | ✅ | ✅ |
| JSON 支持 | ⚠️ TEXT 别名 | ✅ JSONB（二进制） | ✅ JSON1 | ✅ JSON |
| GIS | ✅ 基础 | ✅ PostGIS（完整） | ❌ | ❌ |
| 全文检索 | ✅ | ✅ tsvector | ✅ FTS5 | ✅ FTS |
| 向量搜索 | ❌ | ✅ pgvector | ✅ vec0 | ✅ 天然支持 |
| 物化视图 | ❌ | ✅ | ❌ | ✅ |
| 时序数据 | ❌ | ✅ TimescaleDB | ❌ | ✅ 列式天然适合 |
| 复制 | ✅ 主从/GTID | ✅ 逻辑/物理 | ❌ | ❌ |
| 并发写入 | ✅ 多用户 | ✅ 多用户 | ⚠️ 单写者 | ⚠️ 单写者 |
| 部署复杂度 | 中等（服务端）| 中等（服务端）| 极低（嵌入式）| 极低（嵌入式）|

**关键发现**：
- MySQL 8.0 的特性改进是对 PostgreSQL/SQLite 已有功能的追赶，不是创新
- **JSON 支持的本质差距**：MySQL 的 JSON 类型内部存储为 TEXT——插入时验证格式，查询时提供便捷函数（`JSON_EXTRACT`、`->>`），但没有二进制分解格式。索引只能通过多值索引针对预定义路径。PostgreSQL 的 JSONB 使用二进制分解存储，支持 GIN 索引（自动索引所有键值）、包含查询（`@>`）、键存在性检查（`?`）。这不是"差距多大"的问题——MySQL JSON 是 TEXT 换了个名字，PostgreSQL JSONB 是真正的 JSON 数据类型。MySQL 在 JSON 上的做法是**换个名字凑热度**，与向量搜索的处理方式如出一辙。
- MySQL 对 SQLite 在特性上没有显著优势——SQLite 3.25+ 同样有窗口函数、CTE、FTS5
- MySQL 唯一的差异化优势是**成熟的主从复制生态**——但这不是功能优势，是运维便利

## 结论

MySQL 是两个时代问题的叠加——初期是设计者在数据库领域业余性的产物，后期是 Oracle 商业策略的工具。工程妥协伪装成设计决策，功能缺失伪装成技术债。
1.  **类型系统**：用字符串存储时间是极其业余的——无法直接运算、索引效率低、语义不清晰。
2.  **SQL 语义**：破损的 `GROUP BY` 逻辑为了便利——集合论类型错误被庆祝为"人体工程学"。
3.  **优化器**：愚蠢的连接顺序和僵化的索引需要手动 hack（`STRAIGHT_JOIN`）。
4.  **术语**：WAL→Redo Log，Schema→Database——工程实用主义高于理论正确性。
5.  **扩展**：聚簇索引执念→分片迷信。PostgreSQL/Oracle 通过原生分区和并行查询处理数十亿行——零应用手术。
6.  **存储引擎**：独特的历史意外，迫使用户做出其他 RDBMS 不要求的选择。
7.  **隔离级别**：RR 作为复制缺陷掩护；Serializable 作为暴力锁定；幻读防止作为 MVCC 幻觉。
8.  **一致性（Hermitage 验证）**：写偏斜（G2-item）失控；MAV 单调视图违反（时光倒流读）；通过快照陷阱丢失更新。InnoDB–SQL 层分裂意味着 MySQL 在并发工作负载下交付的少于其名义隔离级别承诺的。

它仍然是遗留企业的"安全"选择，但在针对现代架构的第一性原理测试中失败。

**替代方案推荐**：

| 场景 | 首选 | 理由 |
|:--|:--|:--|
| **OLTP（事务密集）** | PostgreSQL | SSI 可序列化、JSONB、并行查询、原生分区 |
| **嵌入式 / 单机** | SQLite | 零部署、ACID、窗口函数、FTS5 |
| **分析型查询（OLAP）** | DuckDB | 列式存储、向量化执行、零部署 |
| **多模型 / 图+文档+关系** | SurrealDB | SurQL 统一查询、图遍历、权限内置 |
| **分布式事务** | CockroachDB / TiDB | 默认可序列化、水平扩展 |
| **遗留迁移** | PostgreSQL | 从 MySQL 迁移成本最低，生态兼容性最好 |

**新项目**：完全跳过 MySQL。PostgreSQL 在每个维度上都持平或超越——ACID、扩展性、JSONB、全文检索、地理信息、物化视图。MySQL 没有一个维度真正胜出。
