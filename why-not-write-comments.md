# 为什么不应该写注释

> 注释会说谎，因为保持同步的成本太高，而没有任何机制强制同步。消除同步的需求，比维护注释更有效。

---

## 一、注释会说谎

代码在变，注释不一定跟着变。这是注释最致命的缺陷——它不是被编译器验证的，所以它是**不可信的**。

```python
# 计算用户余额（不含冻结金额）
def get_balance(user_id):
    return db.query("SELECT balance FROM accounts WHERE id = %s", user_id)
```

注释说"不含冻结金额"，但代码查的就是 `balance` 字段——冻结金额在哪？这段注释是三年前写的，当时确实有冻结逻辑，后来重构时删了，没人删注释。

**注释说谎比没有注释更危险**：没有注释时，读者会去看代码；有注释时，读者会相信注释然后跳过代码。一个过期的注释就是一个精心伪装的 bug。

---

## 二、为什么同步如此困难

注释腐化的根本原因不是"程序员懒"，而是**同步有额外成本，而这个成本没有回报**。

```
代码改了 → 注释需要同步 → 开发者评估"这条注释还对吗" → 手动修改 → 没有机制验证改对了
```

每一步都有摩擦，最后一步没有兜底。结果是注释必然腐化。

**所有其他代码附属物都有兜底机制，唯独注释没有**：

| 内容 | 编译器检查 | CI 检查 | 开发者主动维护 |
|------|-----------|---------|--------------|
| 代码 | 是 | 是 | 是（编译不过就改） |
| 类型签名 | 是 | 是 | 是 |
| 测试 | 否 | 是 | 是（挂了就修） |
| 注释 | 否 | 否 | **否**（没人逼你改） |

---

## 三、消除同步需求

解决方案不是"更努力地维护注释"，而是**消除同步的需求**。消除方式有两条路径：

### 路径一：代码自己描述自己

类型系统、测试、命名、结构——这些工具**与代码天然同步**，因为它们本身就是代码的一部分。改了代码，这些东西不改就编译不过/测试挂掉。

### 路径二：注释不描述行为，而描述代码无法表达的东西

行为描述（what）由代码 + 类型 + 测试承担。注释只承担代码**在结构上无法表达**的信息——但这些信息的最佳载体也不是行内注释，而是 git commit message / ADR / wiki。

以下章节分别展开这两条路径。

---

## 四、代码自己描述自己

### 4.1 类型系统 > 注释

注释最常见的用途是"说明这个函数返回什么""这个参数是什么类型"。但类型系统做同样的事，且**被编译器强制执行，永远不会过期**。

| 层级 | 语言/工具 | 注释替代能力 | 说明 |
|------|----------|------------|------|
| 强 | **Rust** | 几乎完全替代 | 所有权、生命周期、代数数据类型让参数语义不可能用错 |
| 次强 | Swift / Kotlin / TypeScript | 大部分替代 | 空安全、密封类、泛型约束覆盖 80% 的注释需求 |
| 可用 | Python with type hints | 部分替代 | `mypy`/`pyright` 静态检查，运行时不强制但开发时有效 |
| 弱 | Go / PHP | 几乎无法替代 | Go 缺少代数类型，PHP 类型系统薄弱——**这两门语言的注释需求最高，恰恰也是注释文化最差的语言** |

**因果关系**：不是"因为写注释所以代码清晰"，而是"因为类型系统弱所以被迫写注释"。选对语言，注释需求自然消失。

### 4.2 测试 > 注释

类型系统约束**数据的形状**，测试约束**行为的契约**。两者思路一致：让机器自动执行验证，而非靠人肉阅读注释。

**TDD**（测试驱动开发）：和类型系统同一个哲学——把约束交给机器。测试用例就是"可执行的注释"——它不仅描述行为，还验证行为。

```python
# 注释：当用户余额不足时抛出异常（可能过期）
# 测试：永远不会过期
def test_raises_error_when_balance_insufficient():
    account = Account(balance=0)
    with pytest.raises(InsufficientBalanceError):
        account.transfer(to="bob", amount=100)
```

**BDD**（行为驱动开发）：用 `Given/When/Then` 自然语言描述，比 TDD 函数名更清晰。但代价是**更厚的中间层**——Cucumber/Gherkin 需要胶水代码将自然语言映射到测试逻辑。

| 维度 | TDD | BDD |
|------|-----|-----|
| 描述清晰度 | 中（函数名） | 高（自然语言场景） |
| 中间层厚度 | 薄（直接写测试） | 厚（Gherkin + 胶水代码） |
| 维护成本 | 低 | 中高（两层同步） |
| 适用场景 | 单元/集成测试 | 面向非技术 stakeholder 的验收测试 |

**务实选择**：绝大多数场景 TDD 足够。BDD 只在需要让产品经理/业务方直接读测试时才有价值。

### 4.3 命名与结构 > 解释性注释

```python
# 不好的写法（需要注释）
# 遍历用户列表，过滤出活跃用户，取前 10 个
result = []
for u in users:
    if u.active:
        result.append(u)
        if len(result) >= 10:
            break

# 好的写法（代码自解释）
active_users = [u for u in users if u.active][:10]
```

变量名 `active_users` 本身就是注释。列表推导式的结构本身就是逻辑。如果你需要写注释来解释一段代码在做什么，说明这段代码的命名或结构有问题。

**自解释代码的原则**：
- 函数名 = 动作 + 对象（`calculate_total_price`，不是 `calc`）
- 变量名 = 它是什么（`active_user_count`，不是 `count`）
- 布尔变量 = 问句（`is_valid`，不是 `flag`）
- 提取子函数 = 注释的替代品

### 4.4 程序员不是学徒

注释最常见的形态是**解释代码在干什么**：

```python
# 遍历用户列表
for user in users:
    # 检查用户是否活跃
    if user.is_active:
        # 添加到结果列表
        result.append(user)
```

这是在**侮辱代码和读者**。读者是程序员，不是学徒。他们能读懂 `for user in users` 在做什么。

**两个现代理由让"解释性注释"更加多余**：

1. **代码应该自解释**：好的命名 + 清晰的结构 = 不需要注释。如果写不出来，说明代码需要重构，不是需要注释。
2. **AI 辅助理解**：2026 年的开发者可以选中任意代码片段，让 AI 解释它在干什么。注释的"翻译"功能已经被 AI 取代。

### 4.5 不要写正则，用解析器

有人说"正则太难读了，注释是唯一出路"——但正确答案是**不要写正则**。

用解析器替代：
- **生成器**：`tree-sitter`、`ANTLR`、`lalrpop`
- **组合子**：`nom`（Rust）、`pest`（Rust）、`combine`（Rust）、`parsimonious`（Python）

解析器产出的是有类型、可调试、可测试的结构化数据；正则产出的是脆弱的字符串匹配。如果一段正则复杂到需要注释，它已经复杂到该被解析器替代了。

## 五、注释只描述代码无法表达的东西

类型系统、测试、命名解决了"行为描述"的问题。但有些信息**在结构上无法被代码表达**——这些是注释的正当用途。

### 5.1 契约（前提条件 / 不变量）

```rust
/// 前提：caller 必须保证 `amount > 0` 且 `from != to`
/// 不变量：调用后 `self.balance` 的变化量 = -amount
fn transfer(&mut self, to: &Account, amount: u64) { ... }
```

类型系统能表达"这个参数是 u64"，但无法表达"这个参数必须大于 0"。契约注释补充的是**值域约束**。

### 5.2 设计意图

```rust
// 使用 B-tree 而非 HashMap：数据量 >100K 时 B-tree 的缓存局部性
// 显著优于 HashMap 的随机内存访问（实测快 3x）
```

不是解释代码在干什么（B-tree 的代码本身就是 B-tree），而是解释**为什么选这个数据结构**。

### 5.3 决策排除（Why Not）

```rust
// 选择 Raft 而非 Paxos：Raft 的 Leader 选举更简单，
// 且我们的场景不需要 Paxos 的灵活 Leader 切换。
// 为什么不直接用 etcd 的 Raft？因为我们只需要日志复制，
// 不需要 KV 存储层。
```

### 5.4 摩擦标记（HACK / FIXME）

```rust
// HACK: 等上游修复 #1234 后删除此 workaround
// FIXME: 这里应该用 atomic，但当前 crate 版本有 bug
// XXX: 临时方案，Q3 重构时替换
```

这类注释的价值在于**可追踪性**——配合 Issue tracker，形成闭环。

### 5.5 但这些信息的最佳载体也不是行内注释

| 信息类型 | 行内注释（次优） | 最佳载体 |
|---------|----------------|---------|
| 契约 | 函数头注释 | Rust doc comment / 类型约束（`where T: Positive`） |
| 设计意图 | 代码旁边 | ADR / wiki（有完整上下文） |
| 决策排除 | 代码旁边 | git commit message + ADR |
| 摩擦标记 | `// HACK` | Issue tracker（有追踪、有负责人、有过期机制） |

**原则**：行内注释是这些信息的**降级载体**。能放 commit message / ADR / Issue tracker 就不要放代码里——因为行内注释没有追踪机制，而这些外部工具有。

---

## 六、结论

```
注释会说谎 → 因为同步难 → 消除同步需求
                    │
        ┌───────────┴───────────┐
        ↓                       ↓
  代码自己描述自己        注释描述代码无法表达的
  · 类型系统              · 契约（前提/不变量）
  · 测试（TDD/BDD）       · 设计意图
  · 命名与结构            · 决策排除（Why Not）
  · 解析器替代正则        · 摩擦标记（HACK）
  · AI 辅助理解           ↓
                    最佳载体是 commit message / ADR / Issue
                    行内注释是降级载体
```

> **默认立场：不写注释，除非你能证明它不可替代。**
