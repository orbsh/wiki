# Dify：AI 时代的 Harbor 复制品

**结论**：拒绝。
**状态**：Harbor 的 AI 版本——同样的膨胀架构，更深层的范式矛盾。

## 执行摘要

Dify 是 AI 应用领域的"Harbor"：名气大、架构臃肿、部署困难、资源消耗惊人。它试图用低代码拖拽的方式让非技术人员构建 AI 应用，但这犯了两个根本性错误：一是低估了 AI 的本质（自主规划 vs 预设流程），二是高估了低代码的价值（AI 是低代码的天敌，不是盟友）。

## 与 Harbor 的同源性

| 维度 | Harbor | Dify |
|:---|:---|:---|
| **架构** | PostgreSQL + Redis + Nginx + 多容器 | PostgreSQL + Redis + Nginx + 多容器 |
| **体量** | 存储简单 blob 需要 10+ 组件 | 构建简单 chatbot 需要 10+ 组件 |
| **部署** | Helm chart 噩梦，升级需要数据库迁移 | Docker Compose 噩梦，升级需要数据库迁移 |
| **资源** | 空闲时占用数 GB 内存 | 空闲时占用数 GB 内存 |
| **名气** | CNCF 毕业光环 | GitHub star 光环 |
| **受众** | 新手觉得"功能完整" | 新手觉得"开箱即用" |

**本质**：Harbor 是"容器镜像管理"的过度工程，Dify 是"AI 应用构建"的过度工程。同一个问题的不同表现——用复杂性包装简单需求。

## 范式矛盾：低代码 vs AI 的天敌

### 核心悖论

Dify 的价值主张是"用拖拽构建 AI 应用"。但这个主张隐含了一个前提：**AI 应用的流程是可以提前设计好的**。

如果流程可以提前设计好——拖拽节点、连线、配置参数——那这个应用本质上是一个**确定性工作流**，AI 只是其中一个节点。这不是"AI 应用"，这是"带 AI 节点的自动化流水线"。

真正的 AI 应用应该是：**给定目标，AI 自主规划执行路径**。用户不需要知道流程长什么样，AI 自己决定调用哪些工具、按什么顺序、如何处理异常。

### 低代码是 AI 的天敌

| 维度 | 低代码思维 | AI 思维 |
|:---|:---|:---|
| **流程** | 提前设计，固定不变 | AI 动态规划，自适应 |
| **异常处理** | 每个分支都要人工连线 | AI 自主判断，自动恢复 |
| **扩展性** | 加新功能 = 加新节点 = 重新连线 | AI 自动发现新工具并使用 |
| **用户角色** | 需要"懂流程"的人拖拽 | 需要"懂目标"的人描述 |

低代码要求用户理解并设计流程——这恰恰是 AI 应该做的事情。让用户做 AI 的工作，然后给用户一个"AI 应用平台"，这是本末倒置。

**AI 是低代码的天敌**：如果 AI 能自主规划，低代码的"可视化编排"就毫无价值；如果 AI 不能自主规划，那它就不是 AI，只是一个 ML 算法。Dify 选择了后者——它本质上是一个**ML 算法编排工具**，而非 AI 应用平台。

## 体验问题

### 代码编辑：灾难

Dify 的代码编辑器是"低代码"哲学的必然产物——为了照顾"不会写代码的人"，它提供了一个简化版编辑器。结果是：

- **对不会写代码的人**：仍然不会写代码，拖拽节点并不能让他们理解 AI 的工作原理
- **对会写代码的人**：编辑器太简陋，效率远低于本地 IDE，还不如直接写 Python

这就是低代码的经典陷阱：**不懂编程的用不了，懂编程的嫌太低效**。

### 布线与布局：可视化陷阱

低代码平台的核心卖点是"可视化拖拽"。但 Dify 连自己最核心的卖点都未能实现：布线算法只做几何排列，不理解语义，结果是**可视化同样拙劣**：

- **自动排列后连接边交叉混乱** — 稍微复杂一点的流程（分支、循环、条件），自动排列后连线纵横交错，比手写代码难读 10 倍
- **手动调整成本极高** — 对齐节点、调整连线路径需要大量鼠标操作
- **自动排列覆盖手动调整** — 手动调好的布局，一次自动排列全部打乱
- **仅适用于线性流程** — 现实应用不可能是线性的，有分支、有循环、有条件判断的流程，可视化排列是灾难

**本质**：可视化本身是不是伪需求另说，但 Dify 的布线证明它连可视化都未能做好——连核心卖点都如此拙劣，遑论其他。

### 代码编辑：不能调整大小的牢笼

- **编辑框不能调整大小** — 一个很小的窗口里写代码，正常的一行得三个窗口并列才能放下
- **语法高亮简陋** — 不如 VS Code 的十分之一，其他编辑体验可想而知
- **工作流**：智能复制出来 → 改完 → 粘贴回去。这不是编辑代码，这是在和 UI 搏斗

### 版本控制：与 Git 不兼容

低代码的流程定义存储在数据库里（JSON/YAML），不是文件系统上的代码。结果：

- **无法用 Git 管理** — 流程变更没有 diff、没有 history、没有 branch
- **无法 code review** — 流程修改只能在 Web UI 上"看看"，不能像代码一样审查
- **无法回滚** — 没有版本历史，改错了只能手动改回来

**讽刺**：Dify 号称"AI 应用开发平台"，但它自己连最基本的工程实践（版本控制、code review、回滚）都做不到。

### 部署与维护

- **部署**：需要 PostgreSQL、Redis、Nginx、多个 Worker 容器，Docker Compose 配置几十行
- **升级**：每次版本升级都可能需要数据库迁移，且官方不保证向后兼容
- **备份**：多组件状态分散在不同地方（数据库、文件系统、Redis），备份恢复流程复杂
- **资源**：空闲时占用 2-4GB 内存，对于一个 chatbot 来说是荒谬的

**对比**：一个 Python 脚本 + SQLite + 本地 embedding 模型，50 行代码就能实现 Dify 的核心功能，资源占用 < 100MB。

### 降低门槛的幻觉

Dify 声称"让非技术人员也能构建 AI 应用"。但现实是：

- **非技术人员**：不懂 prompt engineering，不懂向量数据库，不懂 embedding 模型选择。拖拽节点并不能弥补这些知识缺口。
- **技术人员**：直接写代码更快、更灵活、更可控。Dify 的拖拽界面反而增加了认知负担——你需要先理解 Dify 的抽象，再把你的需求翻译成 Dify 的节点。

**真相**：Dify 没有降低门槛，它只是把门槛从"学 Python"换成了"学 Dify"。而 Python 是通用技能，Dify 是特定工具——学哪个更值得，不言自明。

## 热度消退的结构性原因

Dify 在 2024 年获得大量关注，但热度迅速消退。这不是因为 Dify 做得不好，而是它解决的那个问题（让非技术人员编排 LLM）在 Agent 模式下不再需要。

### 范式被 Agent 模型取代

Dify 的核心隐喻是 DAG——人预先定义每个分支，LLM 在节点里当执行器。LLM 的能力演进有明确的先后顺序：先获得工具调用能力（function calling），再获得自主规划能力。有工具但不会规划——知道能调什么，不知道什么时候调——Dify 让人替 LLM 做这个决策。一旦 LLM 获得自主规划能力，预定义 DAG 反而成了约束 AI 生成空间的负优化：人花 3 小时画流程图，本质上是在替 LLM 做「下一步去哪」的决策。

### 低壁垒 = 低差异化

Dify、Flowise、LangFlow、Coze 本质都是同一套东西：前端拖拽 → JSON DAG → 后端串 LLM API 调用。可视化编辑器是多余的中间人——你一旦理解了原理，它就从"降低门槛"变成"增加认知负担"。

### 天花板来得太快

简单场景（单轮 RAG、简单分类）能用。稍微复杂一点（多步推理、条件分支、状态管理）→ 拖拽编辑器崩溃，发现需要写代码。而一旦需要写代码，Dify 的可视化层就成了多余的中间人。

### 成本结构不支持

每次节点调用 = 一次独立 API 请求。三步工作流 = 三次 LLM 调用 = 三倍 token 成本 + 三倍延迟。Agent 单次推理中完成多步决策，共享上下文，成本和延迟都更优。

### 企业真实需求不在这里

企业要的不是"让业务人员画流程图"，而是：自定义逻辑嵌入现有系统、权限/审计/可观测性、与已有服务的深度集成。这些需求用 SDK + 代码写，比用可视化编辑器快一个数量级。

**类比**：Dify 类工具之于 AI Agent，类似 Scratch 之于编程。它降低了入门门槛，但一旦你理解了原理，它就变成了限制你的东西。详见 [可视化迷思](visual-myth.md)。

### 多 Skill 协同：编排问题

Dify 工作流的替代方案不是"更好的工作流"，而是**多 Skill 协同的解法分层**。

当前 AI 应用面临的核心问题是**多 Skill 协同**——一个业务流程需要多个 Skill 按顺序/依赖关系协作，中间涉及意图路由、数据传递、状态同步、错误回滚。四个层次各有缺陷：

| 解法 | 编排方式 | 优势 | 缺陷 |
|:---|:---|:---|:---|
| **Dify 工作流** | 人工预定义 DAG | 可视化、确定性强 | 刚性，无法处理未预见的分支 |
| **Skill（Agent 自主编排）** | LLM 决定调用哪个 Skill、什么顺序 | 灵活，自适应 | 编排过程不可审计，一个 Skill 数据不对会级联污染，错误不可见 |
| **AI 直接写代码** | 无编排，代码即逻辑 | 最高效，完全可控 | 需要 LLM 有足够能力生成正确代码 |

**三层问题**：

1. **意图路由**：用户请求进来，系统决定调用哪个 Skill。这是最表层的问题——需要路由时，Skill 是目前最高效的形式，工作流更差。

2. **数据流转**：上一个 Skill 的输出是下一个的输入。多 Skill 协同的核心痛点在这里——一个 Skill 返回格式不对、字段缺失、类型错误，下游 Skill 静默失败或产生错误数据，且**不可见**。这是 W 描述的"万一一个 Skill 没有限制好，会出现数据错误而且不可见"。

3. **可观测性**：编排过程是黑箱。出了错不知道哪一步出的问题、数据在哪一步被污染、错误是否已经级联扩散。这和可视化工具的「可审计性」缺陷同源——GUI 操作不可审计，Skill 编排决策同样不可审计。

**关键判断**：

1. **不需要编排时，AI 写代码降维打击工作流**——如果逻辑是确定性的（数据转换、格式处理），直接生成代码比画流程图快一个数量级。

2. **路由不可消除**——这是 AI 应用最显著的价值增量。CRUD 功能原本就有，AI 查待办比原来慢多了；AI 的真正价值在于**理解意图、选择路径、组合能力**。

3. **完整的 AI 应用架构需要解决编排可观测性**，而不是退回工作流的刚性确定性。Skill 灵活性 + 编排过程可审计，是当前架构需要找到的平衡点。

## 与 Harbor 的共同根源

Harbor 和 Dify 的膨胀不是偶然的，它们共享同一个反模式：

1. **"功能完整"幻觉**：把所有可能需要的功能都内置，导致架构膨胀。Harbor 内置了漏洞扫描、镜像复制、Web UI；Dify 内置了工作流编排、模型管理、知识库管理。
2. **"企业级"营销**：用"企业级"作为复杂性的借口。Harbor 说"企业需要 RBAC 和审计"；Dify 说"企业需要可视化编排和权限管理"。
3. **依赖地狱**：为了实现"功能完整"，引入大量外部依赖。Harbor 需要 PostgreSQL + Redis + Trivy；Dify 需要 PostgreSQL + Redis + Celery + Nginx。
4. **名气掩盖问题**：GitHub star / CNCF 毕业光环让新手忽略架构缺陷。Harbor 的 star 数让人忽略它的膨胀；Dify 的 star 数让人忽略它的范式矛盾。

## 范式层面的失败

Dify 的真正问题不是架构臃肿，而是**它代表了一种思维模式——用确定性框架约束概率性系统**。

AI 是概率系统，80% 置信度不是"不可靠"，是"这个系统的本质"。试图消除概率性 = 试图让鱼爬树。Dify 的工作流把 LLM 的概率性决策替换成人工预定义的确定性 DAG——这不是"增强 AI"，是"绕过 AI"。

### Harness vs Workflow：两种方向

| | Workflow（Dify 代表） | Harness |
|:---|:---|:---|
| **确定的是什么** | 流程图（DAG），即业务逻辑 | 循环 + 工具接口，即执行机制 |
| **概率的是什么** | LLM 在节点内的输出 | LLM 的每一步决策 |
| **谁决定路径** | 人画 DAG | LLM 自己选 |
| **LLM 的角色** | 节点内的执行器 | 整个循环的决策者 |

**Harness 给 AI 的四种能力**：工具（行动力）、循环（体力）、记忆（状态）、反馈（感知 + 纠错）。AI 的智能在这些基础设施上**放大**。Workflow 把这些全砍了，只剩下一个"AI 节点"执行预定义动作——AI 的智能在这些约束下**压缩**。

**工作流 = CRUD 的回归**。"能用，以前都是这样的"——但吃不到 AI 的红利。AI 的价值增量恰恰在不确定性中：理解模糊意图、处理非标数据、在多个可能性中选择。工作流把这些全砍了。

**企业业务"奇葩"到人都不能理解——恰恰说明确定性规则无法覆盖，必须用概率推理**。用工作流处理"奇葩业务"，和用 Excel 处理"奇葩业务"没有本质区别——都是人在替系统做决策。

> 详见 [Agent 原则：确定性 vs 非确定性工作边界](agent-principles.md)

## 结论

Dify 是 AI 应用领域的 Harbor：用复杂性包装简单需求，用名气掩盖架构缺陷，用"低代码"作为不思进取的借口。

**它犯的根本错误**：试图让 AI 适应人的流程（拖拽编排），而不是让人适应 AI 的能力（自主规划）。这不是技术路线选择，是对 AI 本质的误解。Workflow 约束的是业务逻辑（做什么），Harness 约束的是执行机制（怎么做）——前者压缩决策空间，后者释放决策空间。详见 [Agent 原则](agent-principles.md)。

**首选替代方案**：直接写代码。Python + SurrealDB + 本地 embedding 模型，50 行代码，100MB 资源，完全可控。
