# 原型/快速开发迷思

> "先用 Python 快速验证，验证完再用 Rust 重写"——这个叙事的前提是错的。

---

## 一、类型是契约，原型也不例外

常见做法：原型阶段用 Python/TypeScript，省略类型标注，"先跑起来再说"。理由是类型声明拖慢开发速度。

这个逻辑隐含一个假设：**类型是额外工作**。事实恰恰相反——类型是你本来就有的心智负担，写下来只是把它从脑子里转移到代码里。省略类型不是省掉了工作，是把工作藏起来了。

原型的"快"来自跳过功能范围、跳过测试覆盖、跳过部署流程——不来自跳过严谨。跳过严谨的原型产出物不可信：跑通了不代表设计是对的，因为没有类型约束的情况下，你无法区分"碰巧能跑"和"逻辑正确"。

Python/TypeScript 允许省略类型，但这不代表应该省略。**可选 ≠ 推荐**。Python 社区渐进式类型实践（mypy/pyright）的普及本身就在证明：省略类型是一个需要补课的错误，不是值得保留的优势。

→ 类型系统的必要性详见 [编程语言选型](language-selection.md) §二

## 二、AI 改变了类型声明的成本

传统叙事中，类型声明的"成本"是手写繁琐。这个成本在 AI 辅助下已经趋近于零。

AI 擅长生成类型声明——从数据示例直接推断，从需求描述直接生成，准确率极高。这不是"AI 能帮你写代码"的泛泛之谈，而是类型声明本身的结构特征（输入输出关系明确、模式固定）恰好匹配 AI 的生成能力。手写一个 struct 定义和让 AI 生成，时间差在秒级。

更有价值的是类型声明的连锁效应：**类型一旦存在，IDE 的自动补全能力跃升一个台阶**。IDE 理解了数据结构，就能精确推导方法调用、字段访问、链式操作的上下文。Rust 本身的复杂性（生命周期、trait bound）在 IDE 的语义理解能力面前被大幅抵消——你不需要记住所有细节，IDE 帮你导航。

这改变了一个关键变量：**Rust 的认知负荷不再等于"必须全部记在脑子里"**。类型系统 + AI + IDE 形成闭环，把"理解复杂性"变成了"按需查询"。

## 三、类型系统完备性：理论与现实

理论上，类型系统完备的语言都可以用于原型。但完备性是一个区间，不是一个二值：

| 语言 | 类型完备性 | 原型便利性 | 长期演进能力 |
|------|-----------|-----------|-------------|
| Go | 不完备（无泛型、无枚举） | 高 | 差——样板代码膨胀 |
| Zig | 不完备（有意为之的"简单"） | 高 | 差——表达力不足 |
| Python/TypeScript | 完备但可选 | 极高 | 差——类型可选导致事实忽略 |
| Swift | 完备 | 高 | 中——Apple 生态绑定 |
| Scala | 完备 | 高 | 中——JVM 依赖、工具链碎片化 |
| MoonBit | 完备 | 高 | 未知——生态未成熟 |
| **Rust** | **完备** | **中** | **强——性能、安全、生态俱全** |

Go/Zig 的不完备是**有意的反智**——用"简单"的名义回避类型系统应有的严谨，复杂度从编译期转移到运行期（→ 详见 [Zig 批判与反智主义](go-zig-anti-intellectualism.md)）。

Python/TypeScript 的完备但可选更隐蔽：类型存在，但不强制，实际项目中往往被忽略。等到规模增长需要类型时，补课成本远高于一开始就有。

Swift/Scala/MoonBit 在原型阶段确实比 Rust 更方便——语法更轻、编译更快、上手更低。但原型结束进入长期维护时，各自暴露出硬伤：

- **Swift**：Apple 生态绑定，跨平台能力弱，服务端生态不成熟
- **Scala**：SBT 工具链极慢，JVM 部署复杂，版本兼容性差
- **MoonBit**：WASM 环境问题（WASI 兼容性、GC 支持），特性堆叠稀释设计锐度，生态未建立

原型阶段的便利换不来长期收益——项目进入维护期后，**性能、严谨性、发布便捷性**是硬需求，这三项 Rust 全部满足，而上述语言各有缺失。

## 四、生命周期/所有权：所有语言的隐性成本

Rust 的借用检查器和生命周期标注常被视为"额外负担"。但数据流分析是**所有语言都需要做的事情**——区别只是显式还是隐式。

编写 Python 时同样需要考虑：
- dict 传入后是否被原地修改？
- 列表排序时是否产生了意外复制？
- 中间变量是否多余？
- 变量的作用域是否过长？
- 按变量的声明和使用对逻辑分组，使用挨着声明
本质上是数据流分析，是编写优雅简洁代码的基本原则，逻辑层面的低耦合高内聚

这些问题不会因为语言的动态特性而消失。区别仅在于：编译器检查变成了人脑检查——正确时无反馈，错误时无警告，直到运行时暴露。

Rust 的所有权系统将这些分析**显式化并交给编译器验证**。代价是编写时需要多一步思考，收益是思考完成后由编译器保证正确性。其他语言的代价是省略这一步思考，运行时再追溯，成本高出数个量级。

→ 详细论证见 [Zig 批判与反智主义](go-zig-anti-intellectualism.md) §二

## 五、长期视角：没有小项目

"这个项目很小，用 Rust 是杀鸡用牛刀"——这是一个因果倒置的判断。

项目"小"有两种情况：
1. **真的小**：生命周期短、功能单一、用完即扔。这种情况更适合插件架构，Rust 可以作为基础。
2. **现在小，将来不小**：功能验证通过后会持续迭代、团队会扩大、规模会增长

第二种情况才是大多数项目的真实走向。而这类项目"小"的原因，往往是因为开发者选择了一个小的原版（Python/JS）来保持小——一旦规模上来，用 Rust 重写，**原版保持小但废弃**。

重写的成本远高于一开始就用 Rust。不是因为 Rust 写起来慢，而是因为重写意味着：重新理解业务逻辑、重新处理边界情况、重新测试——而这些工作中，类型系统恰好是最能降低重写成本的工具。

## 六、结论

| 常见迷思 | 实际情况 |
|----------|---------|
| Rust 不适合原型 | 类型声明由 AI 生成，成本趋近于零；类型系统 + IDE 补偿复杂性 |
| 先用 Python 验证再重写 | 重写的隐性成本远超预期，且验证阶段跳过了类型约束 |
| 原型阶段不需要类型 | 类型是心智负担的显式化，省略只是藏起来，不是消除 |
| Go/Swift 更适合快速开发 | 编译便利但长期演进能力不足，性能和严谨性最终会成为瓶颈 |
| Rust 的生命周期太复杂 | 其他语言同样需要数据流分析，只是在脑子里做，错了没有提示 |

**Rust 用于原型的真正障碍是编译时间，不是类型系统。** 编译时间可以通过增量编译、`cargo check`（不链接的快速类型检查）、后台编译缓解。类型系统在 AI 辅助下的成本已经低到不构成决策因素。
