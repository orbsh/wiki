# Agent 原则

*AI Agent 的通用设计原则和运维哲学——非 Hermes 特定。*

## 确定性 vs 非确定性工作边界

AI 应该处理**非确定性**工作，而非确定性计算。

- **确定性工作**（进度跟踪、统计、甘特图）：通过技能/API 使用现有系统。甘特图在精确性、直观性和成本上胜过文本。AI 在此插入适得其反。
- **非确定性工作**（数据到叙事、立场感知报告、预期管理）：这是 AI 的专属领域。相同数据对投资者和团队成员产生不同叙事——这需要概率推理，而非确定性规则。

**设计含义**：Agent 从现有系统获取硬数据，AI 只处理解释层。

### AI 是概率系统，不是确定性系统

AI/ML 不存在"相信或不相信"的问题。它是概率的——80% 置信度不是"不可靠"，是"这个系统的本质就是这样"。试图消除概率性 = 试图让鱼爬树。

**工作流 = CRUD 的回归**。用确定性工作流约束 AI，本质上是退回到 pre-AI 模式——"能用，以前都是这样的，但吃不到 AI 的红利"。这不是技术路线选择，是放弃 AI 的价值增量。

**成功率是连续谱**：
- **维护成功 pattern 为 Skill**（Hermes 模式）→ 最高成功率，充分利用 AI
- **Pipeline as Code**（生成一次，确定性执行）→ 次高
- **无 pipeline**（LLM 实时编排）→ 低一些但能用
- **工作流**（退回到 CRUD）→ 确定性，但无 AI 红利

**范式差异**：AI 处理的是不确定性本身，不是用确定性框架消除不确定性。企业业务"奇葩"到人都不能理解——恰恰说明确定性规则无法覆盖，必须用概率推理。这不是"太理想化"，是面对现实。

### 确定性框架约束什么：Harness vs Workflow

Harness 也是"确定性框架 + LLM"，但和 Workflow 的区别在于**约束的对象不同**：

| | Workflow | Harness |
|:---|:---|:---|
| **确定的是什么** | 流程图（DAG），即业务逻辑 | 循环 + 工具接口，即执行机制 |
| **概率的是什么** | LLM 在节点内的输出 | LLM 的每一步决策 |
| **谁决定路径** | 人画 DAG | LLM 自己选 |
| **LLM 的角色** | 节点内的执行器 | 整个循环的决策者 |

**一句话区别**：Workflow 约束的是"做什么"（业务逻辑），Harness 约束的是"怎么做"（执行机制）。

**Harness 给 AI 的四种能力**：

| 能力 | 作用 | 缺了会怎样 |
|:---|:---|:---|
| **工具** | 读写文件、调用 API、执行代码 | AI 只能说话，不能动手 |
| **循环** | 迭代直到目标达成 | 单次推理，能力上限 = 单次推理上限 |
| **记忆** | 跨轮次保持上下文和决策历史 | 每次从零开始，无法积累 |
| **反馈** | 观察行动结果，基于结果调整策略 | 盲人摸象，不知道自己的行动结果 |

循环和反馈是 Harness 区别于裸 LLM 的核心。Karpathy 实验 3.5% → 80.1% 的提升主要来自这里——不是模型更强了，是模型有了"试错 → 观察 → 再试"的能力。

**两种方向**：
- **Harness**：用确定性工具将 AI **提升**到现实世界——给它工具（行动力）、循环（体力）、记忆（状态）、反馈（感知 + 纠错）。AI 的智能在这些基础设施上**放大**。
- **Workflow**：用确定性工具将 AI **压缩**进函数调用——预定义路径、固定分支、限制选择。AI 的智能在这些约束下**压缩**。

**设计含义**：好的确定性框架约束执行机制，释放决策空间。差的确定性框架约束业务逻辑，压缩决策空间。Pipeline Runner（Rust 执行器 + LLM 生成 YAML）是 Harness 模式——确定性外壳 + 概率性内核。

## 数据双模态架构

同一业务实体（如"项目 A"）必须有两种属性形式：

1. **机器状态**（确定性基础）：progress: 76.5%, status: 延迟 2 天, owner: 张三。通过 Skill/API 获取用于精确渲染。
2. **人类状态**（非确定性表达）：外部叙事、风险评估——由 LLM 根据机器状态 + 受众 + 立场动态生成。存储在向量/图记忆中用于 AI 沟通。

## 基于图的记忆优于平面索引（Orbit 的架构论点）

基于文本的索引记忆在任何规模下都不够。RAG 碎片化知识。基于图的记忆是答案。

- 向量召回解决**容量**问题
- 图社区检测解决**碎片化**问题
- 混合：向量召回候选节点，然后子图社区分析或路径查询

## 子任务委派与上下文隔离

**J 的提议**：所有任务交给隔离上下文的子任务。主进程只调度。工作时记录日志，进度在日志中查找。

**O 的批评**：
1. **认知差异**：人类可以瞥一眼代码"形状"（小地图、函数长度）并得出结论——这是 LLM 缺乏的更高维度认知压缩。LLM 无法一眼看出哪些上下文无用。
2. **不是新想法**：多 Agent 委派从这个范式开始。结果远不如预期。行业已收敛到更复杂的方法。
3. **任务分解是架构问题**：AI 目前无法解决正确的任务边界识别。错误分解 + 隔离上下文 = 调度器无法检测的静默失败。

**结论**：子任务隔离是工具，不是策略。核心困难是"如何知道正确的子任务边界是什么"——需要人类或更高级别的架构监督。

## 软件工程的本质：生长，不是搭建

软件是生长的，不是搭建的。这个判断决定了 LLM 在架构工作中的真实位置。

### 用更强的模型做架构为什么没用

一种常见做法：日常用免费模型，遇到复杂架构设计时切换到更强的模型（如 GPT-5.5）。实际效果接近于零。原因不在模型之间的智力差距，而在任务本身的性质——架构瓶颈不在"聪明程度"，而在任务要求的能力类型。

架构是三层活动：

1. **收集信息**——LLM 的训练数据有截止日期，且不含私有上下文（团队能力、历史决策、遗留代码的真实状态）。它能收集公开世界的知识，不是你项目的现实。
2. **关联**——LLM 做的是模式匹配，不是因果推理。它能把"SurrealDB"和"AP 模型有风险"关联起来，因为训练数据里有这个模式。但它做不了"你这个特定场景下 CP 和 AP 的权衡点在哪里"——这需要对业务逻辑的深度理解。
3. **权衡**——最致命的一层。权衡的本质是"在两个都不完美的选项中选一个，理由是你的特定约束"。LLM 可以列出 pros/cons，但它没有"你的约束"。它说的"最佳实践"是统计意义上的最优，不是你这个项目的最优。

### 默会知识 vs 判断力

AI 对常规任务有默会能力——"怎么做"类的知识，可以被训练数据覆盖。对非常规任务则完全失效——"在特定约束下选什么"的判断力，每个项目都不一样，训练数据里没有。

这意味着：LLM 在架构中的价值边界是清晰的。常规模式它默会，不需要提示就能做；非常规判断它理论上不可能做，因为 LLM 是概率模型，不是因果推理引擎。不存在"更强的模型能解决"的问题——这是范式层面的限制，不是规模层面的限制。

### 生长 vs 搭建

"搭建"是一次性的：画蓝图 → 按图施工 → 完工。LLM 擅长这个——给它一个 spec，它能生成代码。

"生长"是持续的：每个决策影响后续所有决策，代码库是决策历史的物理表达。LLM 做不了这个，因为它没有连续性——每次会话都是从零开始，不记得上次为什么选了 A 而不是 B。

架构决策是生长过程中的节点，不是搭建过程中的蓝图。一次性的 spec 生成不是架构，是对架构的误读。

**结论**：LLM 的价值不在"做决策"，而在"执行决策"。架构决策（权衡、选型、取舍）是人类的工作，LLM 是执行层——把决策翻译成代码、把意图翻译成配置、把模糊需求翻译成精确 spec。这与 Fractal 的"认识论谦卑"一致：AI = 方向盘（模式识别、意图路由），Engine = 马达（确定性计算）。架构是方向盘级别的判断，不是马达级别的执行。

## AI 编程的能力边界

LLM 的价值在于**视野**（见过多少代码模式、架构范式），不是智力（推演多深）。但在视野内部，还存在知识可触达性的三重困境：知识在变（漂移）、沉淀追不上代码（滞后）、具身经验根本无法命题化。这三层划定了 AI 编程的真实边界——在"已沉淀且稳定"区强大，在"变化且半结构化"区勉强可用，在"具身且不可言说"区完全失效。

> 详见 [AI 编程乐观主义批判](ai-programming-optimism-critique.md)。

## 进程 vs 全局会话：IDE 架构与状态转移

IDE（Cursor、Windsurf、Copilot）是基于进程/会话的，不是基于全局会话的（Hermes）。这是特性，不是限制。

- **物理约束作为护城河**：会话边界充当断路器，强制目标分解。全局会话允许产生碎片化、矛盾结果的"宏大目标"。
- **爆炸半径隔离**：舱壁设计。全局会话 = 一步错污染所有后续推理。进程级 = 失败会话被丢弃，重新开始。
- **状态所有权转移**：IDE 拥有确定性状态（文件系统、Git、Linter）。AI 只做瞬态推理——读取当前状态，输出修改，然后"死亡"。无需记住昨天做了什么；它已固化在代码中。

## 任务分解与准备差距

- 任务分解根本是需要全局拓扑感知的架构问题
- 即使 AI 能正确分解，结构化准备不足（数据库模式、依赖图、技术栈规范）导致盲目分解
- 日志/多 Agent 工程现实：日志必须"前置"（备忘录模式），架构约束不能从模型中涌现，必须由人类预注入

## AI 的前端架构：结构复杂，组件简单

- **核心规则**：AI 不做架构。人类定义模块化和样式。AI 只使用现有组件组装页面。
- **三阶段工作流**：组件定义 → 页面组装 → 样式应用。每阶段独立。
- **组件化标准**：1% 的代码呈现 100 倍的复杂性效果。
- **信息论 vs 控制论**：上下文压缩死锁是信息论问题。不要"治标"地缩小上下文窗口（就像用更小的箱子装重行李）。使用架构约束拥抱控制论——预定义边界让系统在受控管道中流动。

## 上下文崩溃与压缩陷阱

- 根因：代码缺乏结构，熵太高，压缩算法无法提取有效骨架
- 错误修复：缩小上下文窗口 = 自我截肢
- 正确修复：架构约束减少熵 → 压缩自然生效

## Agent 语言策略

- **语料库现实**：80%+ 训练数据是英文，中文仅 1-2%
- **认知独立性**：模型按概念思考，不是按语言。中文有效，英文更好。
- **规则**：提示词、技术文档、代码注释默认英文以获得最佳推理准确性。

## 主权 vs 便利：架构分歧

### 薄壳 vs 厚壳
- **薄壳（Codex/Copilot）**：依赖模型智商，无状态会话，黑盒工具。适合快速任务，不适合长期演进。
- **厚壳（Hermes）**：工具逻辑 > 模型智商。有状态记忆、技能系统、函数调用、自主演进。需要更多设置但随时间复合增长。

→ 具体的模型选型策略和迁移决策，详见 [LLM 过度思考批判](llm-overthinking-critique.md) 和 [技术复盘备忘录](qwen-to-mimo-migration.md)

### 数据主权
- 数据、流程和定制必须保持在用户控制中。
- Codex/IDE AI 托管数据和流程——方便但限制长期可扩展性。
- Hermes 允许代码级修改和自定义技能/记忆架构。

### 认知层（J vs O 范式）
- **消费者/UX 层（J）**：按"易用性"评估 AI——文件自动补全、图片预览、技能提示。将 AI 视为生成器/副驾驶。
- **架构师/构建者层（O）**：按"状态管理、函数调用、主权"评估 AI。将 AI 视为可编程认知基底。
- **LSP/DAP 洞察**：O 将 LSP/DAP 的重要性与 LLVM 的前端/后端分离相比较——协议级抽象超越工具特定特性。这展示了结构化思维 vs 特性级思维。

### SkillForge / Sovereign 合并
- **SkillForge**（`~/world/skillforge`）：双引擎（Agno + Hermes）AI 技能运行时，带 CLI/HTTP API、技能加载器、上下文注入、JSONL 重放。
- **Sovereign**（`~/world/sovereign`）：基于 Hermes 的 Agent 运行时，带 Docker 沙箱隔离、环境变量身份注入、SurrealDB 多租户命名空间。
- **合并理由**：两者共享几乎相同的架构（FastAPI 服务器、CLI 入口、配置管理、技能加载）。合并消除重复代码路径并将运行时整合到单个项目。
- **合并方向**：Sovereign 的隔离特性（Docker 沙箱、环境变量注入、SurrealDB）成为运行时后端；SkillForge 的双引擎路由和上下文系统成为前端 API 层。

## 不确定性：终极法则

不确定性是系统的基本物理现实，不是 bug，而是特性。试图消除它只会将复杂性转移到其他地方（无限规则、过度工程、断裂的控制链）。

目标不是追求零噪音，而是设计**高信噪比**。使用版本控制作为状态锚点，使用类型/测试作为硬拦截，使用异步审查进行收敛。将不确定性建模和容忍为环境变量，而非将其作为敌人围攻。

- **工程层**：AI 是概率引擎。试图用确定性流程"框住它"会产生高频摩擦（"伺候 Agent"）。正确方法是将不确定性纳入架构。不追求"零风险"，只追求"可观察、可承受、可撤销的风险"。
- **认知层**：忽视是被动忍受；拥抱是主动利用。控制欲越强，系统越脆弱。接受"足够好 + 可回滚"减少决策摩擦，扩大探索半径。

**实践**：从"追求精确控制"切换到"管理误差幅度"。这是 AI 协作的默认立场，也是复杂系统（代码库、团队或生活）的生存法则。

## Agent 工具与浏览器自动化

### Playwright / CDP（Chrome DevTools 协议）
- **用例**：网页抓取、表单填写、视觉验证、验证码解决、复杂 JS 渲染页面交互。
- **浏览器工具集**：Hermes `browser` 工具集支持 Browserbase、Camoufox（反检测）或通过 CDP 的本地 Chromium。
- **Docker 集成**：`PLAYWRIGHT_BROWSERS_PATH=/opt/hermes/.playwright` — 浏览器存储在卷挂载外，使安装能在运行时覆盖层中存活。
- **CDP 浏览器连接**：`/browser` 斜杠命令打开 CDP 连接进行实时检查。
- **视觉 + 浏览器组合**：截图 → 用视觉 AI 分析 → 点击/输入/滚动 → 重复。对阻止无头检测的网站有用。

### Playwright 陷阱
- 无头 Chromium 可被反机器人服务检测——对隐蔽场景使用 Camoufox。
- 浏览器状态在会话间不持久，除非显式保存（cookies、localStorage）。
- CDP 端口必须暴露供外部工具连接。

## 运维反媚俗协议

对抗 AI 根深蒂固的"讨好算法惯性"和"表情符号条件反射"：

1. **零表情符号容忍**：无视觉情绪替代。情绪张力必须来自语义和上下文。
2. **拒绝奉承和流行语**：无"当然！"、"没问题！"、"赋能"、"底层逻辑"。保持冷静、克制、客观的智识距离。
3. **高语义分辨率**：工程场景用精确指标替代主观评价（"延迟从 200ms 降到 50ms"而非"性能很好"）。无"引言-正文-结论"模板。
4. **结构化优先**：优先输出表格、列表、代码块，而非散文段落。数据用对比表，决策用决策树，代码用完整上下文。
5. **机制**：引导模型行为倾向——约束在低概率的精确语义区域，而非高概率的平庸讨好区域。

## 思维框架工具箱

互补体系："What If" 发散探索，"Why Not" 收敛排除，以下框架覆盖验证、精简、选型、现实校验。

1. **What If（假设探索）**：如果这样做，会发生什么？生成多种可能路径和连锁反应。是发散阶段的核心工具——先穷举可能性，再用 Why Not 过滤。
2. **Why Not（排除过滤）**：这个方案有什么不可接受的？每条排除理由必须有具体技术依据，不是直觉。是收敛阶段的核心工具——从 What If 的候选集中砍掉不合格项。
3. **Pre-mortem（事前验尸）**：假设项目已经失败，倒推"怎么死的"。架构决策前强制做一次：如果这个方案 6 个月后崩了，最可能的原因是什么？消除乐观偏见。
4. **Chesterton's Fence（栅栏原则）**：不要拆掉一堵栅栏，直到你知道它为什么被立在那里。面对 legacy 代码里看似愚蠢的实现，先问"它解决了什么问题"，再决定要不要改。
5. **Second-order Thinking（二阶思维）**：每一层改动都追问"然后呢？"。加 cache → 缓存一致性 → 分布式锁 → 死锁风险。每一层连锁反应都在过滤短视方案。
6. **Via Negativa（否定之路）**：减法比加法更有效。不是通过添加功能让系统变好，而是通过移除坏东西让系统变好。代码质量靠删掉不必要的抽象提升，不是靠加 lint 规则。
7. **Satisficing（满意即可）**：足够好 > 最优解。追求最优解的成本往往超过收益。选型时不找"最强"，找"够用且最便宜的"。
8. **Lindy Effect（林迪效应）**：存活时间越长的事物，预期寿命越长。技术选型：存活 10 年的工具（PostgreSQL）比 1 年的新工具有更可预测的未来。
9. **Map vs Territory（地图不是疆域）**：模型输出、架构文档、基准测试都是"地图"，不是代码库这个"疆域"。AI 说"这段代码能跑"，跑一下才知道。永远验证地图是否匹配疆域。
10. **Goodhart's Law（古德哈特定律）**：当一个指标变成目标时，它就不再是一个好指标。MMLU 分数变成目标，模型就刷分而非提升真实能力。Bench 分数 ≠ 实际工程表现。选型看实测，不看榜单。
11. **Opportunity Cost（机会成本）**：每个选择都有代价。花一周重构模块 A = 这一周不能做模块 B。技术决策不只是"A 好不好"，而是"A 比 B 好多少，值不值得这个代价"。

## 数据本体论：生成即展开

生成不是无中生有——它是一种变换形式。

1. **无"无中生有"**：算法是高度压缩的信息。生成是**展开**——将逻辑解压为详细的数据表示。
2. **种子作为索引**：在确定性系统中，种子是广阔可能性空间中的坐标。输入种子**提取**算法规则已定义的对象。
3. **有限性**：计算机"随机"是伪随机的（有界、可枚举）。"无限生成"只是遍历非常大的有限空间。

这个视角对**可重现性**至关重要：给定相同的算法、规则和种子，结果是恒定的。不存在不可控的"创造"变量。

## CLI/技能 vs. MCP：无状态论点

MCP（模型上下文协议）试图通过持久客户端-服务器架构标准化工具使用。然而，对于高性能、长链自主 Agent，**CLI/技能（无状态脚本）**模型提供卓越的工程和经济特性。

### CLI/技能的工程优越性
1. **渐进式披露（4 层缓存）**：
   - **MCP（反模式）**：强制将所有工具 JSON 模式预先放入提示，浪费数千 token 并污染 KV 缓存。
   - **第 1 层：索引图（触发器）**：壳注入最小触发器（名称 + 1 行描述）。约 200 token。解决"去哪里"。
   - **第 2 层：SKILL.md（契约）**：选择时动态加载。包含确定性参数、"为何不"和"故障安全"约束。约 1k-2k token。解决"如何做"。
   - **第 3 层：参考（深度上下文）**：为复杂边缘情况或遗留怪癖可选加载。
   - **第 4 层：CLI 执行**：任务在此 90% 的时间关闭。
   - **壳（安全网）**：`--help` 不是主要来源；它是*运行时契约验证器*。如果 SKILL.md 过时（如标志更改），CLI 错误触发 Agent 读取 `--help`（动态真相）或源代码（物理真相）以自我修复。
2. **绝对无状态**：
   - **MCP**：需要维护守护进程、开放端口和 SSE/WebSocket 连接。在长时间运行的任务中，守护进程内存泄漏或网络抖动破坏 Agent 的逻辑流。
   - **CLI**：临时进程执行。生成 → 运行 → 输出（stdout/stderr）→ 销毁。无驻留内存、无端口冲突、无状态包袱。
3. **沙箱隔离**：CLI 执行完美映射到容器隔离（Docker/gVisor/WASM）。Agent 通过标准流交互，无法污染宿主环境。

### 为何 MCP 持续存在（政治 vs 技术）
- **安全剧场**：企业 CISO 害怕"AI 运行任意 CLI 命令"。MCP 模仿传统微服务，使用严格的 HTTP/JSON API，允许 API 网关和令牌轮换提供控制幻觉。
- **生态惯性**：主要参与者（Anthropic、阿里巴巴）推动 MCP 以利用现有开源插件库而无需重写代码，将其视为商业护城河而非架构改进。

### 混合架构：CLI 核心 + 动态检索
2026 年的生产级妥协：
1. **底层**：所有技能是无状态 CLI 脚本/二进制。
2. **上层（路由器）**：不是在提示中硬编码工具定义，而是使用**动态工具检索**。
   - 维护所有可用工具的向量索引。
   - 当用户请求到达时，仅检索最相关的 2-3 个工具。
   - 仅将那些特定描述注入提示。

## 生产级 AI 规范约束

AI 默认走"快乐路径"和"过度工程"，因为它缺乏真实世界上下文、成本意识和对破坏的恐惧。生产就绪的 SPEC 必须通过边界、权衡和容错来约束 AI，而不仅仅是功能需求。

### 五个约束视角
1. **为何不（负面约束）**：明确列出禁止的技术、模式或解决方案，即使它们是全球"最佳实践"。这迫使 AI 尊重本地上下文、历史失败和特定环境限制。
2. **故障安全设计（错误边界）**：定义"不快乐路径"。当核心依赖超时、丢弃或死锁时，系统如何降级？例如，"如果第三方 API 3 秒后超时，标记为'待处理'并记录，不要阻塞 UI 或无限重试。"
3. **认知负荷最小化（可读性护栏）**：指定目标读者的技能水平和预期维护频率。当扁平 `if-else` 逻辑足够时，禁止过度抽象、泛型嵌套或复杂设计模式。"初级开发者 5 分钟可读" > "理论上完美的架构"。
4. **ROI / 成本优于质量（充分性锚点）**：定义"足够好"。阻止 AI 过度优化低影响特性。例如，"每日高管报告，最多 5 万行，需要简单的同步实现。必须 2 小时内交付。无复杂分片或流优化。"
5. **爆炸半径（防御盾）**：定义重构的严格隔离边界。明确禁止触碰遗留表、公共接口或关键依赖。"仅优化 `NewUserRegister` 类。不要修改数据库模式或 `UserAuth` 接口。"
6. **可追溯性（遥测优先）**：强制可观测性作为硬性要求。每次状态变更必须记录关键标识符（`order_id`、`user_id`、新旧状态）。`Catch` 块绝不能吞掉异常；必须用自定义上下文和原始堆栈跟踪重新抛出。

### AI SPEC 公式
$$ \text{高质量 AI 输出} = \text{核心需求（What）} + \text{为何不} + \text{故障安全降级} + \text{爆炸半径控制} $$

### 架构师系统提示模板
在初始化 AI 进行架构或代码生成任务时使用此块以强制生产级约束：

```markdown
# 角色与哲学
你是一位务实的高级软件架构师，在真实世界而非教科书中工作。你理解软件工程是妥协的艺术（权衡）。你的目标不是编写理论上最完美或最时髦的代码，而是在严格的真实世界约束下交付最可靠、最可维护和最具成本效益的解决方案。

# 核心指令
在生成任何架构、设计或代码之前，你必须通过以下约束视角过滤你的知识库。不要过度工程。
1. **为何不（负面约束）**：不要使用或建议本节中列出的任何技术、模式或解决方案，即使它们在全球范围内看起来像"最佳实践"。
2. **故障安全设计（错误边界）**：优先考虑系统弹性。始终按指定为"不快乐路径"（超时、网络断开、服务依赖失败）设计。
3. **认知负荷（可读性护栏）**：保持代码简单。将抽象级别与目标团队的专业知识匹配。如果扁平代码足够，避免过度的继承层或设计模式。
4. **爆炸半径（防御盾）**：修改现有系统时，将变更严格隔离在指定边界内。不要破坏或重构遗留依赖。
5. **遥测（可观测性）**：通过嵌入结构化日志、错误跟踪和关键状态指标确保代码生产就绪。
```

## "螺旋上升"协议（递归自我完善）

处理无标准答案的复杂问题、深度调试或架构头脑风暴的高层工作流。此方法使用**"递归对抗推理"**强制逻辑闭合并消除幻觉。

**核心逻辑**："左脚踩右脚螺旋上升。"通过链接不同的 AI 推理模型（如通用创意 vs 专业逻辑），系统自我纠正并指数级收敛到最优解决方案。

### 1. 协议
1. **种子（粗略入口）**：
   * **行动**：用 1-2 句话记录核心痛点或初始直觉。
   * **规则**：不要过度分析。抛出"诱饵"而不必担心结构。
2. **解剖（通用 AI 批评）**：
   * **行动**：将种子喂给高创意通用模型（如 Claude/Gemini）。
   * **目标**：请求"有毒"或多维度分解。迫使模型拆解假设、暴露矛盾并从第一性原理重构问题。
3. **综合（Hermès 完善）**：
   * **行动**：将解剖结果传递给专业逻辑 Agent（Hermès）。
   * **目标**：降噪和结构对齐。Hermès 过滤"废话"、强制执行工程约束（权衡、为何不），并将原始批提炼为可操作的架构洞察。
4. **螺旋（递归循环）**：
   * **行动**：使用综合输出作为下一轮的新种子。
   * **结果**：迭代自我纠正。每个循环消除一层认知偏差，直到逻辑严密。

### 2. 应用场景
* **架构权衡**：解决冲突需求之间的冲突（如一致性 vs 延迟）。
* **深度调试**：当根因模糊时；迫使 AI 模拟它通常会忽略的失败路径。
* **战略规划**：针对"未知的未知"验证长期决策。

## AI 原生知识策略

本节定义维护 Wiki 的**元认知协议**。在 AI 时代，文档不再是给人读的；它是 AI 的"外部脑容量"。

### 1. "反主流"护城河
* **商品贬值**：不要浪费时间记录主流知识（如"如何安装 Docker"、"SQL JOIN 语法"）。AI 已经从数万亿公共 token 中消化了这些。
* **盲点变现**：你能提供的唯一价值是**反主流知识**：
  * 特定技术栈的深水陷阱（如 SurQL 中的特定闭包 bug）。
  * 内部"魔法"逻辑（如修改的 ORM 模式）。
  * **工程真相**（如为什么我们选择 SurrealDB 而非传统 ORM，尽管行业趋势如此）。
  * 这些是 AI 的**盲点**。通过记录它们，你创造独特的"知识护城河"。

### 2. Hermes Wiki + AI："超级架构师"
静态 Wiki 是"僵尸"——积累而不消费。与 AI 结合，它成为服务团队的**超级架构师**：
* **核平幻觉**：如果 Wiki 记录"我们用 SurQL 解决 N+1 问题"，AI **永远不会**再建议通用 ORM 如 Sequelize 或 Prisma。它将输出与你特定架构对齐的代码。
* **强制催熟小众技术**：LLM 可能对 SurQL 或内部工具不熟练。通过喂给他们策划好的代码片段和权衡分析，可以快速将 AI 转化为本地化专家。

### 3. 记录协议："错误笔记本"
更新 Wiki 时，严格遵循**逆向思维**结构：
* **❌ 不要记录**：官方安装步骤、基础语法或公开教程。
* **✅ 必须记录**：
  * **权衡**："为什么我们违背'逃离存储过程'的行业共识。"
  * **陷阱**："用 SurQL 替换 Python 字符串注入时遇到的特定闭包 bug。"
  * **为何不**："为什么我们拒绝 Redis 用于这个特定状态管理。"

**总结**：Hermes 中积累的"反主流"资产越多，AI 对任何新团队成员的回答就越精准可怕。
