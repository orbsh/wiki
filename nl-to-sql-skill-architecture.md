# NL-to-SQL 技能架构

**状态**：已接受  
**日期**：2026-05-21  
**上下文**：企业级 AI 数据分析技能——无状态、可移植、多租户。

---

## 问题陈述

业务用户希望通过自然语言查询数据（"本月产品销量排名"）。初始方法：将原始数据转储给 LLM → 分析。

**失败**：
1. **速度**：大数据集延迟不可接受
2. **Token 成本**：OOM 级上下文消耗
3. **准确性**：LLM 对原始数据聚合产生幻觉

**认知**：这本质上是 **NL-to-SQL** 任务。以结构化形式存在的数据应该通过 SQL 查询，而非被 AI "阅读"。源和结果之间零信息损失。

---

## 决策链

### 决策 1：数据持久化在哪里？

| 选项 | 结论 | 原因 |
|------|------|------|
| 本地技能路径（Parquet）| ❌ 拒绝 | 技能必须无状态且可移植。本地持久化在实例重启时中断。|
| 直接业务数据库 | ❌ 拒绝 | 在生产数据库上执行 AI 生成的 SQL 是不可控风险。注入、全表扫描、数据损坏。|
| 新数据库 + 同步 | ⚠️ 可行但重 | 需要专用数据库运维、连接池、备份、迁移。|
| **湖仓一体（Delta Lake）** | ✅ **选择** | 需要同步但无数据库维护。自描述文件 + 事务日志。|
| 湖仓一体（Iceberg）| ❌ 拒绝 | Catalog 服务（Hive Metastore/Nessie）维护开销超过问题本身。|

**选择**：**S3/对象存储上的 Delta Lake**。工具成熟，无需 catalog 服务（基于文件的事务日志），原生分区裁剪。

### 决策 2：多技能交叉引用与并发写入

**初始假设**：单技能、单表、纯 Parquet 即可。

**纠正**：企业技能是多租户和多技能的。跨技能业务关联需要跨表 JOIN。纯 Parquet 无法维护关系完整性。

| 关注点 | Parquet | Delta Lake |
|--------|---------|------------|
| 多技能 JOIN | ❌ 无模式注册表 | ✅ 模式演进、事务日志 |
| 并发写入 | ❌ 文件损坏 | ✅ 通过 `_delta_log/` 实现 ACID |
| 碎片化 | ❌ 小文件累积 | ✅ 内置 `OPTIMIZE` 压缩 |
| 点更新/删除 | ❌ 重写整个文件 | ✅ 支持 MERGE INTO / DELETE |

**结论**：Parquet 被拒绝。Delta Lake 对企业技能架构是强制的。

### 决策 3：工具选择

| 工具 | 结论 | 原因 |
|------|------|------|
| Spark/Flink（Java/Scala）| ❌ 拒绝 | 启动慢、JVM 预热、重集群维护。技能是非持久进程。|
| Polars | ⚠️ 部分 | 数据操作优秀，但 SQL 支持有限。AI 生成 SQL 比 Polars DSL 更好。|
| DuckDB | ✅ 读层 | 原生 SQL、快速 OLAP、Python 绑定成熟。完美适合 AI 生成的 SELECT 查询。|
| `deltalake` Python | ✅ 写层 | 基于 Rust、经过验证、处理 Delta Lake 写入。AI 永远不触碰写入路径。|

**选择**：**`deltalake`（Rust/Python）用于写入 + DuckDB 用于读取**。职责分离：技能的 `run.py` 拥有写入逻辑；AI 只生成 SELECT 查询。

### 决策 4：同步机制

| 方法 | 结论 | 原因 |
|------|------|------|
| RisingWave 外部 ETL | ❌ 拒绝 | 违反技能自包含。外部维护、独立生命周期。如果外部 ETL 可接受，单独 RisingWave 更简单（物化视图）。|
| Cron + 同步脚本 | ❌ 拒绝 | 同上——外部维护负担。30 分钟同步节奏是任意且无意义的。|
| **查询时同步（技能内）** | ✅ **选择** | 逻辑内聚且可移植。重用现有 API（已返回宽表）。冷启动延迟被内部测试者的首次请求吸收。|

**查询时同步流程**（硬编码在技能脚本中）：
1. 查询 `_delta_log` 获取最后记录的 `update_time`
2. 从 API 获取：`GET /data?since={update_time}`（要求所有记录有 `update_time`）
3. 通过 `deltalake` 将新数据写入 Delta Lake
4. 通过 DuckDB 在 Delta 表上执行 AI 生成的 SQL

脚本只接受 SQL 文本。同步 + 查询在技能内是原子的。

### 决策 5：数据安全与租户隔离

**需求**：多租户企业系统。用户只能看到自己组织的数据。

**解决方案**：`org_path` 作为**分区列**，带路径枚举前缀（如 `org/42/dept/7/`）。

- **同步阶段**：所有同步记录从源 API 携带 `org_path`。
- **查询阶段**：技能注入 `WHERE org_path LIKE '<user_prefix>%'` 作为硬前缀。
- **AI 生成**：提示指示："SQL 将附加 `WHERE org_path LIKE '<prefix>%'` 执行。你生成此子句之前和之后的逻辑。"
- **性能**：Delta Lake/DuckDB 原生分区裁剪 `LIKE 'prefix%'`——避免全表扫描。

这实现了**行级安全（RLS）**而无需数据库级授权。

## 架构总结

```
┌─────────────────────────────────────────────────┐
│                   SKILL（无状态）                 │
│                                                  │
│  ┌──────────┐    ┌───────────┐    ┌───────────┐ │
│  │   用户    │───►│   AI LLM  │───►│  SQL 生成  │ │
│  │   查询    │    │ (NL→SQL)  │    │           │ │
│  └──────────┘    └───────────┘    └─────┬─────┘ │
│                                         │        │
│  ┌──────────────────────────────────────▼──────┐│
│  │           技能脚本 (run.py)                  ││
│  │  1. 检查 _delta_log 中最后 update_time       ││
│  │  2. GET /api/data?since={update_time}       ││
│  │  3. deltalake.write(new_records)            ││
│  │  4. duckdb.sql(AI_SQL + org_path_filter)    ││
│  └──────────────────────┬──────────────────────┘│
│                         │                        │
└─────────────────────────┼────────────────────────┘
                          │
              ┌───────────▼───────────┐
              │   S3 / 对象存储        │
              │  s3://bucket/org/42/  │
              │    _delta_log/        │
              │    data/*.parquet     │
              └───────────────────────┘
```

## 关键原则

1. **AI 设计，引擎执行**：AI 生成 SQL；DuckDB/deltalake 处理重计算。永远不让 AI 直接处理数据。
2. **技能自包含**：无外部 ETL、无 cron、无 catalog 服务。同步逻辑存在于技能脚本内。
3. **无状态设计**：Delta Lake 存储在 S3 上。本地路径仅为临时缓存，重启时丢弃。
4. **强类型优于测试**：AI SQL 生成受模式上下文 + 强制 `org_path` 过滤器 + 只读 DuckDB + `LIMIT 1000` + 超时约束。
5. **查询时同步可接受**：冷启动延迟可接受。首次请求（内部测试者）吸收同步成本。后续请求是增量且快速的。
