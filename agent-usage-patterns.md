# AI Agent 使用模式

*人类侧的交互技巧和审查习惯——决定 Agent 质量上限的不是模型能力，是使用方式。*

## 核心逻辑：质量上限由反馈精度决定

AI 没有主动的"质量意识"。它不会自发觉得"这段回复写得不好"或"这个操作可以更优"。执行指令，遇到边界情况用最省力的方式处理。没有反馈 = 没有信号说"这里不对"。

长期不纠正不会导致崩溃或致命错误，但会停在"能用但差一点"的层次。而那个"一点"恰好是使用习惯拉开的差距。

纠正的本质不是"告诉 AI 哪里错了"，而是建立共同的判断标准。AI 的 memory 机制将偏好固化成持久规则，同类场景不用再重复纠正。这是复利的具体操作路径——每次纠正都是一次校准，Hermes 逐渐从"通用默认值"变成"理解你的协作体"。

> 复利机制的完整论述，见 [Agent 复利](agent-compound-interest.md)。

## 交互模式

### 主动驾驶，不要被动接受

两种使用模式的本质区别：

- **被动使用**：AI 做了什么就接受什么。积累不满但不说，最终觉得"这工具不好用"——实际上从未建立过反馈回路。
- **主动驾驶**：每次交互都是一次校准。频繁检查产出，不一致的立即纠正并说明理由，超出预期的要求解释动机。

第二种模式让 AI 从"每次从零开始的通用助手"变成"越来越懂你的协作体"。

### 要求解释动机

当 AI 做了超出预期的操作时，直接问"为什么这样做"。

这是低成本的自检机制：AI 被迫审视自己的决策链。如果动机站不住脚，它自己也能发现——比事后发现问题代价小得多。也可能学到一个你没想到的技术原因（"X 在这个场景下有问题"），即使 AI 最终是偷懒了（"没有特别原因"），你也获得了审查依据。

### 问"为什么不用另一个方案"

当你觉得某个方案怪的时候，问"为什么不用 X"。

这比直接否定更有效——直接否定会让 AI 防御性辩护，而提问让它主动审视。三种可能的结果：学到真正原因（你赚了），发现它偷懒了（你拦截了），或者确认两个方案等价（你节省了纠结时间）。

## 指令技巧

### 负面指令优于正面指令

"不要自动提交"比"提交时先通知我"效果好得多。

正面指令是路径，覆盖已知场景。负面指令是围栏，覆盖所有未预想的路径。一条"不要"能拦截一整类问题，而不需要逐一列举"如果遇到 A 不要、遇到 B 不要、遇到 C 不要"。

工程场景中，"为何不"约束（禁止的技术、模式、方案）比功能需求更能决定输出质量。即使是"全球最佳实践"，在特定环境下也可能不可接受——负面约束迫使 AI 尊重本地上下文和历史失败。

> 生产级 SPEC 的约束框架，见 [Agent 原则](agent-principles.md) 中"五个约束视角"。

### 先观察再行动，有偏差及早纠正

改代码之前让 AI 先读相关文件、解释现有结构。带着理解去改，比带着假设去改错误率低一个数量级。很多人直接说"帮我改 X"，AI 跳过理解阶段直接动刀。阅读文件的成本极低，但省掉这一步导致的误修改成本很高。

另一面：AI 的观察本身可能有偏差。一旦发现理解方向不对，立即纠正，不要等它执行完再说。偏差在理解阶段纠偏成本最低，到了实现阶段纠正成本翻倍，到了多个文件联动后纠正成本指数增长。

### 一次只做一件事

上下文窗口是有限资源。"重构这个模块并修复三个 bug"不如先修 bug 再重构。多任务并行会互相挤占上下文，AI 开始丢前面的约束——你说了"不要改公共接口"，它在第四个任务里忘了。

每个原子任务必须有明确的验证标准——怎么确认它完成了。没有验证标准的任务 = 没有终点的任务，AI 会在"差不多了"的时候停下来。

### 执行中发现 Spec 不合理，回溯而非绕行

执行阶段发现原始计划有漏洞时，正确的做法是回溯修正计划，不是在执行中随意打补丁。绕行的代价是后续所有步骤都建立在错误假设上，debug 成本指数增长。

### 修改前列出计划

"先说你要改哪些文件、改什么，我确认后再动手"。

这一步成本极低，但能拦截大量"我帮你顺便优化了"的越权行为。计划审查是人类最低成本的干预手段——读一行计划比读一百行 diff 快得多。

### 区分生成和修改

生成新代码和修改已有代码是完全不同的任务类型。修改任务错误率更高，因为要同时理解现有上下文 + 做出正确变更。对修改任务需要更严格地要求计划和验证。

## 审查习惯

### 看 diff，不看全文

AI 改完后看 diff，不要从头读整个文件。diff 是 AI 实际做了什么的最短表达——每行变更都有明确的意图，比全文阅读效率高一个数量级。

版本控制系统天然支持这种审查模式。git log + diff 是最低成本的质量门。

### 检查后台维护质量

Agent 的长期价值在于后台积累（记忆、技能、自动化）。这些积累的质量需要主动审查——memory 写入是否准确、skill 是否过时、cron 是否按预期运行。

被动积累的后台数据会逐渐漂移。定期检查并纠偏，是维持复利方向的必要投入。

### 纠正要写入持久化

口头纠正只影响当前会话。写入 memory 或 skill 的纠正才具有复利效应。

操作路径：发现问题 → 纠正并说明理由 → AI 写入 memory → 同类场景永久生效。跳过第三步 = 每次重复纠正。

写入什么有讲究。记忆策略三原则：

- **存事实**：`User prefers X`、`Tool Y fails on Z`。陈述句，客观描述。
- **不存指令**：`Always do X`（系统提示词已包含行为准则，重复是污染 memory）。
- **不存临时态**：会话进度、PR 编号、中间文件路径——这些靠 session_search 检索，不值得固化。

## 持久化配置

### AGENTS.md 锁定项目规则

把核心约束（代码风格、架构原则、禁止操作）写进项目根目录的 `AGENTS.md`。AI Agent 启动时自动读取，相当于给每个 session 预装了一套偏好。

比口头约定可靠——口头约定依赖 AI 记住（它会忘），文件约束是硬编码的（每次强制执行）。

### 项目文档作为知识沉淀

在项目中维护一份文档，把 AI 需要的背景知识整理进去。这比每次在对话中重复解释高效得多——文档是无状态的，AI 每次启动自动加载，不需要你手动注入。

不是所有知识都值得沉淀。主流知识（"如何安装 Docker"、"SQL JOIN 语法"）AI 已经从训练数据中消化了。值得沉淀的是**反主流知识**：特定技术栈的深水陷阱、内部架构决策的真实原因、被否定的方案及其理由。

> 反主流知识的护城河效应，见 [Agent 原则](agent-principles.md) 中"AI 原生知识策略"。

## 技术方案设计套路

与 AI 交互生成技术方案，不是"问 AI 给答案"，而是"用 AI 做压力测试"。以下从 NL-to-SQL 架构设计案例中提炼的套路，适用于单次方案设计场景（与上文日常使用习惯互补）。

### 1. 先给约束，再给问题

错误做法：直接问"帮我设计一个用 AI 查询数据的方案"——AI 会给出泛泛而谈的方案。

正确做法：先列出所有硬约束（技术、运维、组织限制），再描述问题。约束条件大幅缩小 AI 的推理空间，排除大量不切实际的方向。

### 2. 记录"为什么排除"，不只记录"为什么选择"

决策文档中，排除理由比选择理由更重要。后续有人质疑"为什么不用 X"时，答案已经有了，不用重复辩论。

### 3. 让 AI 做"挑刺者"，不是"设计者"

你已经有思路了，让 AI 来找漏洞。把你的分析给 AI，明确说"分析评估这个逻辑，指出问题"。AI 的建议不一定对（它缺少你的领域知识），但它的质疑能暴露你没想到的边界情况。

### 4. 用决策链替代结论列表

不要只给结论，展示推理过程：

```
纯 Parquet 不够
  ├─ 需要 ACID 处理并发 → 选 Delta 或 Iceberg
  ├─ Iceberg 需要 Catalog → 运维过重
  └─ 结论 → Delta Lake 是唯一合理选择
```

读者能跟踪推理过程，判断是否同意每一步。如果某个前提变了，决策链自动失效，需要重新评估。

### 5. 分阶段迭代：大框架 → 细节 → 工程陷阱

- **第一轮：大框架对齐**——确认核心方向是否正确
- **第二轮：细节补充**——补充场景和约束
- **第三轮：工程落地**——讨论实际实现的时的坑（重试、分区、schema 变更策略）

### 6. 明确告诉 AI 什么不重要

AI 倾向于给"完整方案"，会加上不需要的高级功能。明确排除可以节省讨论时间，避免方案膨胀。

### 7. 保留完整的决策文档

最终产出不是"一个方案"，而是包含决策链、排除理由、架构图的完整文档。新成员加入时不需要重新解释"为什么这么设计"。

### 速查清单

与 AI 讨论技术方案时，按这个顺序给信息：

1. **背景** — 真实发生了什么（不要抽象描述）
2. **约束** — 所有硬限制（技术、运维、组织）
3. **你的思路** — 你已经想清楚的部分
4. **你的疑惑** — 你不确定的部分
5. **排除项** — 你明确不想要的东西
6. **迭代** — 让 AI 挑刺，你补充领域知识，循环 2-3 轮
7. **产出** — 决策链文档（含排除理由）
