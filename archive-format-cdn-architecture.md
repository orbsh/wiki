# 归档格式与 CDN 架构：从物理结构到生态妥协

## 一、核心矛盾：随机访问 vs 流式传输

文件格式的物理骨架，直接决定了它在网络上的命运。这两者在底层逻辑上存在天然的对抗：

**中央索引在尾部**（Zip, SquashFS, Seekable Zstd）
- 优势：支持完美的部分解压/随机访问。客户端可以通过 HTTP Range Request 先读尾部索引，再精准拉取目标字节段。
- 代价：无法流式解压（边下载边解压）。在读到最后一个字节前，解压器在前端是盲目的。

**索引散落在头部/无全局索引**（Tar 变体）
- 优势：支持完美的流式传输。解压状态机前后连续，适合网络管道一次性拉取并原子化安装。
- 代价：无法随机访问。即使只想看一眼目录或提取一个小文件，也必须从头扫描/解压到目标位置。

## 二、为什么不把中央索引放开头？

将索引放开头看似能兼顾两者（先读索引，再流式下载），但在实际落地时会引发严重的工程灾难：

1. **破坏一趟流式写入（Single-pass Write）**：压缩前无法预知文件大小，若索引在开头，服务器必须「回头补写（Seek Back）」或走两趟处理，这与 AWS S3 等不可变对象存储（Immutable）的「不可修改」特性完全冲突。

2. **CDN 缓存雪崩**：追加新文件会导致开头索引变大，后续所有数据的字节偏移量整体发生「平移」，导致 CDN 上的所有 Range 缓存瞬间全部失效。

3. **无法真正解决网络流式点播**：在连续的 HTTP 单向流中，即使知道了偏移量也无法「跳跃」，依然需要等待前方无用数据传输完毕。

## 三、主流格式在不同场景下的工程权衡

| 格式组合 | 索引位置 | 部分解压（Range Request） | 流式解压（Pipeline） | 核心适用场景与权衡 |
|---|---|---|---|---|
| tar.gz / tar.zst | 无（散落于文件头） | ❌ 不行（必须从头扫描） | ✅ 可以（原生状态机） | Linux 包管理器/全量安装：利用「固实压缩」追求极限压缩率，且安装属于原子操作，无部分解压需求。 |
| Zip（传统） | 文件末尾 | ✅ 可以（按字节寻址） | ❌ 不行（必须读到尾部） | 浏览器端源码预览：生态极其成熟，配合 CDN Range Request 实现接近零负载的按需点击。 |
| SquashFS + Zstd | 文件末尾（按块寻址） | ✅ 可以（按块寻址） | ❌ 不行（必须拿到尾部） | 虚拟文件系统/容器镜像：块内支持 Zstd 高效压缩，支持免解压直接挂载（Mount），随机 I/O 性能极佳。 |
| Seekable Zstd | 文件末尾（跳帧） | ✅ 可以（按块寻址） | ✅ 可以（标准解压忽略跳帧） | 现代折中方案：牺牲了一定的全局压缩率（块边界重置字典）来换取双向支持。 |

## 四、终极结论：生态的「懒惰」战胜了技术的「先进」

crates.io 之所以最终选择「Tar.gz → Zip 离线转换 + 外部 JSON 站岗 + CDN Range Request」的架构，其本质在于：

- **CDN 的盲视性**：商业 CDN 不需要懂任何复杂的文件格式，它只需要三十年前的 HTTP/1.1 标准——`Accept-Ranges: bytes`。

- **生态的万能钥匙**：Zip 在浏览器前端（JS/WASM）的解析生态极其成熟。如果采用 CAS（内容寻址）会带来恐怖的 CDN 请求数（HTTPS Count）计费刺客与小文件 LRU 缓存失效；如果采用 SQLite 容器则会引发 B-Tree 的网络读放大。

- **架构师的智慧**：不试图去发明一种逆天改命的「完美格式」，而是顺应物理规律，在业务层做补偿。用一个外部并排的 `manifest.json` 帮前端省去第一次读 Zip 尾部的 RTT，将计算负载完美转化为 CDN 最擅长的静态字节切片分发。

## 五、Zip 尾部读取的工程细节：64KB 的物理赌注

前端浏览器解析 Zip 的第一步，是从文件末尾读取 EOCD（End of Central Directory，中央目录终端记录）。这里有一个精巧的工程把戏：

**64KB 的来源**：Zip 规范规定 EOCD 末尾有一个变长的 Comment 字段，最大长度由 2 字节无符号整数决定，即 $2^{16} - 1 = 65535$ 字节。EOCD 本体 22 字节，加上最大 Comment，总共 $22 + 65535 = 65557$ 字节（约 64KB）。浏览器请求「最后 64KB」，就能物理上 100% 覆盖 EOCD。

**倒着扫描**：前端拿到 64KB 后，从最后一个字节往前搜索 EOCD 的魔数 `0x06054b50`。一旦匹配，就能从 EOCD 结构中读出两个关键数字：中央目录的大小和起始偏移量。

**两种情况**：

- **小 Crate**：中央目录只有几 KB，紧挨着 EOCD 放在末尾。64KB 的缓存已经包含了整个中央目录。1 次网络请求，前端直接在内存里解析出文件树。
- **大 Crate**：中央目录长达数 MB，64KB 装不下。前端通过 EOCD 查出中央目录的位置和大小后，发起第 2 次精准的 Range 请求拉取完整中央目录。总共 2 次请求，但相比动辄几十 MB 的完整 Zip 文件，仍然只下载了微不足道的索引。

**crates.io 的优化**：如果让前端走「猜 64KB → 没装下 → 补读中央目录 → 用户点击 → 再 Range 读单个文件」的流程，弱网下 RTT 会累积。crates.io 的做法是后端预提取中央目录为 JSON，前端第一步连「猜 64KB」都省了——直接拿 JSON，一步到位发起针对目标文件的 Range 请求。

## 六、SquashFS 运维核心速查

在实际运维或构建镜像时，SquashFS 提供了强大的本地操纵能力：

- **打包**（Zstd 高效模式）：`mksquashfs source/ out.sqsh -comp zstd`
- **极限压缩**（发布用）：`mksquashfs source/ out.sqsh -comp zstd -Xcompression-level 22`
- **部分提取**（解压单个文件）：`unsquashfs out.sqsh src/main.rs`
- **免解压直接挂载**（调试）：`mount -t squashfs -o loop out.sqsh /mnt/dir`
