# 循环工程（Loop Engineering）分析

**状态**：分析完成
**日期**：2026-06-23
**来源**：对 Addy Osmani（2026-06-07）canonical essay 的批判性评估

---

## 一、原始定义

Loop Engineering 由 Addy Osmani（Google Chrome 团队）提出，Boris Cherny（Anthropic Claude Code 负责人）和 Peter Steinberger 推动。核心定义：

> 用你设计的「系统」来代替「你」去向 Agent 发送提示词。

五个构建块：Automations（cron/webhook）、Worktrees（git worktree 隔离）、Skills（SKILL.md 持久化）、MCP（外部连接器）、Sub-agents（Maker/Checker 分离）+ Memory（状态持久化）。

参考实现：`cobusgreyling/loop-engineering`（723 stars），理论基础：`wquguru/harness-books`（2548 stars）。

---

## 二、本质判断

### 不是新范式，是已有实践的重命名

Loop Engineering 的五个构建块**全部已有实现**：

| 构建块 | Hermes 实现 | Codex/Claude Code 实现 |
|--------|------------|----------------------|
| Automations | cron job、webhook | /loop、/goal、Automations tab |
| Worktrees | delegate_task 隔离上下文 | git worktree、--worktree |
| Skills | Hermes skills（SKILL.md） | Agent Skills（SKILL.md） |
| MCP | 原生 MCP 客户端 | MCP servers |
| Sub-agents | delegate_task | Task subagents、agent teams |
| Memory | session DB + 长期记忆 | Markdown/Linear 状态文件 |

和"Context Engineering"（Prompt Engineering 的重命名）、"Harness Engineering"（Agent 框架的重命名）一样，Loop Engineering 是 2026 年的品牌事件，不是技术事件。

### 核心价值：一次定义、反复执行

Loop Engineering 的真实价值是**减少人的参与**——在特定场景下，一次定义、反复执行，参与总量确实降低了。典型案例：用户提工单 → AI 分类 → AI 直接开发修复，中间的客服/分派环节整个消失。

但这个收益**高度场景依赖**——只有任务来源可定义、执行可验证、结果可机械判断的场景（CI 修复、依赖升级、简单 bug、工单自动处理）才适用。设计新系统、决定技术选型等场景不适用，因为没有可自动验证的终止条件。

## 三、任务来源：Loop 的真正瓶颈

### 任务不会凭空产生

无论 loop 的输入源是什么，都需要人先定义"什么是问题"：

| 输入源 | 人的参与 |
|--------|----------|
| 直接告诉 Hermes | 每次都要开口 |
| 工单系统 | 需要人写工单 |
| K8s 监控 | 需要人安装监控、定义告警规则 |
| CI 失败 | 需要人写测试、设阈值 |
| Git push event | 需要人定义什么算"问题" |

Loop 不能凭空产生任务，但在适用场景下确实能减少人的参与（如工单自动处理消除了客服分派环节）。关键区别：参与减少只发生在高频、机械、可验证的场景。

### 事件驱动优于轮询

Cron polling 是没有 webhook 能力时的备选方案。事件驱动优先：

| 场景 | 轮询（备选） | 事件驱动（优先） |
|------|-------------|-----------------|
| K8s 故障 | cron 每小时检查 pod 状态 | Watch API / Event 推送 |
| 新工单 | cron 每 10 分钟轮询 | Webhook POST 到 Hermes |
| CI 失败 | cron 检查构建状态 | GitHub Actions webhook → Hermes |
| 代码变更 | cron 扫描 git log | Git hook / push event |

Hermes 已有 webhook 能力（`hermes webhook subscribe`）。能用事件就别轮询。

### 侧面指定也不能解决任务来源

工单系统（Linear、GitHub Issues）是常见的侧面任务来源，但它只是把"你直接告诉 Hermes"变成了"你写工单，Hermes 读工单"。任务仍然需要人来创建和定义。工单系统解决的是**任务管理**问题，不是**任务来源**问题。

## 四、验证天花板：Loop 的硬边界

Loop 能做的只有**验证**（通过/不通过），不能做的**判断**（好不好、对不对、值不值得）。

| 能自动验证 | 不能自动验证 |
|-----------|-------------|
| 测试通过 / 失败 | 架构是否合理 |
| Lint 干净 / 有警告 | 业务逻辑是否正确 |
| 类型检查通过 | 代码风格是否符合团队约定 |
| 安全漏洞扫描 | 这是不是用户真正想要的 |
| diff 大小 / 变更范围 | 技术方案选型是否恰当 |

四个概念正交，不在同一维度：

| 概念 | 职责 | 层次 |
|------|------|------|
| **Harness** | 给 Agent 正确的环境、工具、上下文，让它能干活。解决"能不能做" | 技术架构（固定方案，跨场景通用） |
| **内循环** | 单任务内的自我验证（修 → 测 → 修 → 测 → 过） | Agent 的能力 |
| **Loop Engineering** | 系统级调度：触发、分派、推进、跨会话状态管理。解决"做没做对" | 系统编排能力（场景相关，每个业务不同） |
| **事件驱动** | 决定什么时候触发，替代 cron 轮询。解决"什么时候做" | 触发机制 |

Harness 解决的是机械性修复本身（lint 报错修 lint，一次就够），内循环是 Agent 在单任务内反复验证直到通过，Loop Engineering 是系统级编排（触发、分派、跨会话状态），事件驱动决定什么时候触发。Loop Engineering 文档强调的"明确终止条件"是内循环的事，不是 Loop Engineering 本身的贡献。

验证是二元的，判断是多维的。一旦涉及"这个功能该不该这么做"，必须停下来等人。

## 五、实操指南：五个步骤

以下是从批判性分析中存活下来的实操方法论——不是 Loop Engineering 的品牌叙事，而是真正有用的工程实践。

### 步骤一：四条件测试（与 Karpathy/Niklaus 实验共识）

不是所有活儿都适合塞进 loop。在动手之前，先过四关：

| 条件 | 含义 | 不满足的后果 |
|:---|:---|:---|
| **任务重复发生吗？** | 频率决定 ROI | 一年一次的任务不值得设计 loop |
| **有自动化验收手段吗？** | 没有验证就没有终止条件。注意：只有二元可验证（通过/不通过）才算，需要多维判断的（架构是否合理、业务逻辑是否正确）不在范围内（参见第四章验证天花板） | loop 永远不知道自己做对了没有 |
| **Token 预算扛得住吗？** | 每次运行都有成本 | 烧钱比人工贵 |
| **Agent 有"高级工程师"的工具吗？** | 能访问代码库、CI、测试 | 巧妇难为无米之炊 |

四关全过才值得建 loop。大部分"我想自动化这个"的冲动过不了这四关。

这四项条件与 Karpathy（AutoResearch）和 Niklaus（Hugging Face）的实验结论一致——他们的实验也证明：只有在高频、可验证、有真实环境访问的场景下，Loop 自动化才有正 ROI。详见 [LLM 过度思考批判 §7.8](llm-overthinking-critique.md#78-harness-优先的实证karpathy--niklaus)。

### 步骤二：最小可行 loop

第一次别搞花活，就建四件套：

1. **触发器**（Automation）：定时跑、事件触发跑都行
2. **技能**（Skill）：把项目上下文写进 STATE.md，每次运行不用重新解释
3. **状态文件**（State File）：Markdown 记"做到哪了、什么成了、什么挂了"，下次接着跑
4. **门禁**（Gate）：测试、类型检查、构建——能自动拦住坏结果的东西

**顺序很关键**：先手动跑通一次 → 写成 Skill → 包进 loop → 最后才上定时。

### 步骤三：拆卷子 vs 判卷子

整个 loop 设计里最重要的一条原则：**写代码的和验代码的，必须分开**。

用一个模型（或子 Agent）负责写，用另一个独立的模型（或子 Agent）负责验收。验收的那个**不能看到**写代码的那个的推理过程。

为什么？模型给自己写的代码打分时"标准过低"——这是元认知缺失的直接体现（参见 [LLM 基础认知](llm-fundamentals.md) 5.3.5）。所有"看起来不错"的代码，在独立验收器面前大概率能挑出大量问题。Maker/Checker 分离不是架构偏好，是对 LLM 无元认知这一事实的工程适配。

### 步骤四：避坑指南

| 坑 | 后果 | 防法 |
|:---|:---|:---|
| 没有硬停止条件 | 跑到账单爆炸或被限流 | 设 Token 上限、迭代次数上限、时间限制 |
| 状态不落地 | 今天学到的明天就忘 | 写进 STATE.md，每次运行接着读 |
| 让 loop 碰"需要判断"的活 | 架构重写、鉴权、支付逻辑出错 | loop 只干"对错清晰、机器可验证"的活 |
| 不读 Diff | 合入越来越快，理解越来越浅 | 读 Diff，哪怕只是扫一眼——否则产生"理解力债务" |

"理解力债务"是最隐蔽的坑：真正的代价不是 Token 账单，而是某天你要调试一个团队里没人读过的系统。

### 步骤五：唯一的衡量指标

别管烧了多少 Token、开了多少 PR、跑了多少次任务。

**唯一有用的指标：每个被接受的改动，平均成本是多少。**

如果"被接受率"低于 50%，说明你在做 loop 本该替你省掉的评审工作——loop 在亏钱。

## 六、补充方案：AI 主持人 + Loop 执行

> Loop Engineering 解决不了沟通问题——它只能自动化机械任务，不能替代需求讨论中的多维判断。以下是 Loop 之外的补充架构：在沟通层用 AI 主持人降低共识达成成本，在执行层用 Loop 自动化机械任务，在判断层保留人类拍板。

### 需求讨论的现实

IM 是中国职场的主流沟通工具，但它的结构性问题是把异步沟通同步化——要求所有人实时在场，级联中断碎片化深度思考。根因是等级制度和对深思熟虑的不尊重，不是工具本身（详见 [职场沟通](workplace-communication.md)）。

沟通量**无法被减少**，它是组织复杂度的函数（10 个决策者 = 45 条通道）。AI 主持人的价值不是消除沟通，而是让不可避免的沟通运转得更顺畅。

### 三层架构

试图用 loop 消除沟通是错误的方向。正确方向是三层各司其职：

| 层 | 职责 | 工具 |
|:---|:---|:---|
| **AI 主持人** | 整理混乱对话、标注冲突、同步共识锚点 | IM 频道 + AI 持续分析 |
| **Loop 执行** | 代码修改、测试、PR、CI/CD | Gitea + CI + 容器 |
| **人类判断** | 拍板需求、审核 PR、验收结果 | 频道讨论 + PR Review |

AI 主持人的价值不是"减少沟通"，而是让不可避免的 45 条通道运转得更顺畅：
- **降低信息不对称**：AI 整理的结果所有人都看到，不用靠"谁参加了哪个会"来同步
- **减少重复沟通**：共识锚点不断更新，不用车轱辘话
- **标注分歧**：AI 发现意见冲突直接拉到台面上，不用私下拉扯

### 需求讨论的完整流程

```
IM 频道（混乱对话）
  → AI 持续分析、整理、标注冲突
  → 人基于 AI 整理的结果在频道里讨论确认
  → 共识达成 → Gitea Issue
  → Loop：代码 → 测试 → PR
  → 人类审核 PR → 合并 → 发版
```

关键细节：
- **提取不是自动化的**：AI 整理 + 人在频道里审核修正，不是 AI 直接提交 Issue
- **确认在频道里完成**：提需求的人同时看到 AI 的整理（包括与目标/现有路径的冲突），现场沟通解决，透明度更高
- **AI 是主持人不是决策者**：不断更新进展和共识，人基于新的共识锚点讨论
- **最终判断是人下的**：AI 整理 + 标注冲突，但拍板是人说"对，就这样"或"不对，改这个"

### "平等对待"问题的解法

LLM 对所有发言平等对待，但实际对话中不同发言的权重不同：

| 解法 | 适用场景 |
|:---|:---|
| **时间衰减** | 最近的发言权重更高（需求在演化） |
| **共识检测** | 多人说了类似的话 → 高置信度；只有一个人说 → 低置信度 |
| **冲突检测** | A 说"要快"，B 说"要稳" → 标红让相关人确认 |
| **发言者级别排序** | 决策者的发言权重更高 |
| **确认投票** | 关键决策发起相关人投票确认 |

### 风险点

**AI 整理的准确性**：如果 AI 遗漏关键信息或误读对话意图，后续所有讨论都基于错误的整理结果。但审核 AI 的整理结果比审核代码 diff 简单——每个人只关心"自己的主张有没有被正确传达"，不需要理解全局上下文。diff 反而更难，因为要理解变更前后的完整逻辑。解法：给 AI 的整理结果加置信度标注（哪些是明确拍板的，哪些是 AI 不确定的需要人确认）。

**共识到 Issue 的转化**：频道里达成的共识是口语化的（"那就用方案 B 吧"）。转成 Issue 需要结构化（目标、验收标准、边界条件）。这一步 AI 可以做，但需要人确认——"AI 理解的方案 B 是不是你说的方案 B"。

**理解力债务**：如果跳过沟通直接让 AI 做需求，团队成员变成"审批机器人"——只看结果不理解过程。真正的代价不是 Token 账单，而是某天调试一个没人理解的系统。AI 主持人模式通过让所有人看到 AI 整理 + 讨论过程，部分缓解了这个问题。

## 七、场景分析：Loop 的适用范围

### 两种 Loop

Loop Engineering 适用于两类场景，核心逻辑相同（观察 → 分类 → 决策 → 执行），但循环特性不同：

| | 维护型 Loop | 创造型 Loop |
|:---|:---|:---|
| **目标** | 保持系统状态 | 从 A 到 B（开发、写作） |
| **循环次数** | 无限 | 有限 |
| **模式重复** | 高（同类问题反复出现） | 低（每次都是新问题） |
| **自动化潜力** | 随时间增长 | 固定（机械部分有限） |
| **人的角色** | 分类 + 拍板 | 规划 + 判断方向 |
| **典型场景** | K8s 巡检、库存管理、监控告警 | 需求开发、架构设计、创作 |

**维护型 Loop** 的核心是已知类别 + 预定义策略。以 K8s 卫士为例：

| 类别 | 紧急 | 危险 | 动作 |
|:---|:---|:---|:---|
| 已知（修复） | ✓ | ✗ | 自动修复 + 记录 |
| 已知（修复） | 其他 | - | 等人批准 |
| 已知（建议） | ✓ | ✗ | 自动建议 + 记录 |
| 已知（建议） | 其他 | - | 等人批准 |
| 已知（上报） | - | - | 直接上报 |
| 未知 | - | - | 等人分类 |

随着类别库增长，紧急且低风险的类别自动化比例上升。人只在非紧急或新类别出现时介入判断。

**创造型 Loop** 是 Karpathy 的"思考→执行→验证"闭环：从模糊需求出发，收敛后停止。Loop 只能接管验证环节（跑测试、格式检查），不能接管思考本身。文档驱动和闭环验证是把创造型 Loop 的模糊度降下来，让 Loop 能接管更多机械部分。

### 运营场景的适用性

库存管理等运营场景同样适用，但阈值更模糊：

- 低库存 → 自动补货（紧急 + 低风险）
- 库存积压 → 建议促销（非紧急 + 低风险）
- 大额采购 → 上报审批（高风险）
- 新品/季节性需求波动 → 未知类别，等人判断

和技术场景的区别：K8s 的阈值是硬的（pod 重启 3 次 = 异常），库存的阈值是软的（多少算"低"取决于季节、供应商周期、资金状况）。运营场景中人需要介入更多，但机械部分（库存扫描、已知模式匹配、阈值检查）仍然可自动化。

### 关键洞察

**适合 Loop 的场景有一个共同特征**：判断标准可定义，验证可机械化。

- K8s 巡检：阈值清晰，已知类别可自动化，人只处理新情况
- 库存管理：阈值模糊但模式可识别，自动化比例低于 K8s 但仍有价值
- 需求开发：判断标准模糊，Loop 只能接管验证，人主导全过程

**不适合 Loop 的场景**：没有可自动验证的终止条件（架构设计、产品方向、创意工作）。这些场景中人的判断是核心价值，Loop 无法替代。

## 八、Hermes 的现状

Hermes 已经具备 Loop Engineering 的全部组件：

- **Automations**：cron job + webhook ✓
- **Worktrees**：delegate_task 隔离上下文 ✓
- **Skills**：SKILL.md + /curator 生命周期管理 ✓
- **MCP**：原生 MCP 客户端 ✓
- **Sub-agents**：delegate_task + kanban worker ✓
- **Memory**：session DB + 长期记忆 ✓
- **Kanban**：任务分解 + dispatcher + worker 工具集 ✓

Hermes 具备 Loop Engineering 的全部组件，但具备组件≠已实现场景化编排。Loop Engineering 只是给 Hermes 已有的能力起了个新名字——名字是新的，场景化连接还缺。

### 缺的不是组件，是连接

Hermes 的 cron、kanban、skills 各自独立运行。完整的事件驱动产线应该是：

```
外部事件（工单/CI/K8s/webhook）
  → webhook 触发 Hermes
  → Hermes 创建 kanban 卡片
  → dispatcher spawn worker
  → worker 执行、验证
  → 完成后 webhook 回写外部系统
  → 经验写入 skill
```

这个"胶水"不是新范式，是一个配置好的工作流。

## 九、渐进式信任模型（唯一真正的新意）

Loop Engineering 提出的 L1→L2→L3 分级值得记录：

| 层级 | 模式 | 人的参与 |
|------|------|----------|
| L1 | Agent 只生成报告，不修改代码 | 审阅报告 |
| L2 | Agent 辅助修复，需要人确认 | 审阅 diff |
| L3 | Agent 完全自主，只通知结果 | 抽查 |

这不是技术问题，是**组织采纳策略**——让团队从"完全不信任 Agent"逐步过渡到"Agent 自主工作"。对于企业环境（如 skillforge），这是实际可用的渐进部署路径。

---

## 十、结论

1. Loop Engineering 是已有实践的重命名，不是新范式
2. 核心价值是减少人的参与，但只在高频、机械、可验证的场景下成立
3. 任务来源始终需要人定义，loop 不能凭空产生任务
4. 事件驱动（webhook）优于轮询（cron）
5. Harness、内循环、Loop Engineering、事件驱动——四者正交，不在同一维度
6. 验证天花板：只能验证机械性结果，不能判断业务合理性
7. Hermes 已具备全部组件，缺的不是能力而是场景化连接
8. 唯一真正的新意是渐进式信任模型（L1→L2→L3）

### 命名问题

"Loop" 这个词含义模糊、无所不包——代码 loop、Agent loop、业务 loop、学习 loop 都能套进去。"Loop Engineering" 中 "Loop" 的限定力不足：任何重复性工程实践都能被套进这个术语，它没有划出一个区别于"Engineering"本身的子集。

正因为含义模糊，每个人都可以把自己已有的实践套进去说"这就是 Loop Engineering"。一个术语如果能被所有场景同时认领，它就没有区分力。没有区分力的术语是营销工具，不是工程概念。

### 分类错误

Addy 的文章把 Loop Engineering 放在技术演进的第四阶段（Prompt → Context → Harness → Loop），这个分类是错的。Harness 是技术架构（固定方案，跨场景通用），Loop Engineering 是业务建模方法论（场景相关，每个业务不同）。两者不在同一个维度。类比：DDD（领域驱动设计）是业务建模方法论，不是技术架构的下一个台阶。

### 与 Harness 的根本区别

**Harness** 是一次设计、所有场景通用的固定方案。**Loop Engineering** 是每个场景需要单独设计的定制方案——CI 修复是一套 loop（触发条件、验证标准、终止条件），依赖升级是另一套，工单处理是又一套。Harness 是工程投入一次、收益无限次；Loop Engineering 是工程投入 N 次、每次收益有限。

成本回收的主要变量是**频率**，但不是唯一变量——错误代价同样关键。同一场景每天发生 100 次值得设计 loop，一年发生 1 次不值得；但如果错误的代价极高（支付逻辑、安全漏洞），即使频率低也值得投入自动化防护。

### 方法论深度

Loop Engineering 作为方法论本身太简单——看到重复劳动就想到自动化，这是工程师的本能反应。"设计触发条件 → 定义验证标准 → 设置终止条件"就是写 if-else 的思路，没有什么可展开的。Addy 文章的真正价值不在于提出"Loop Engineering"这个概念，而在于系统性对比了 Claude Code 和 Codex 的相同原语——这部分有信息量。但"Loop Engineering"作为概念本身，就是一个正确但平庸的工程直觉被包装成了"范式"。
