# 反应式架构：从被动拉取到主动推送

**核心命题**：传统 Web 架构是请求驱动的被动模式——浏览器刷新、轮询、回源拉取。反应式架构反过来：数据变更时主动推送到边缘和客户端，用户请求只命中已就绪的数据。流计算和消息队列不是辅助组件，而是核心基础设施。

## 被动模式的代价

传统架构的默认交互是**浏览器刷新**：用户点刷新 → HTTP 请求 → 源站查询 → 响应。这带来三个问题：

1. **重复请求**：数据未变也重复拉取，浪费带宽和源站资源
2. **延迟不可控**：每次请求都经过完整链路（DNS → TCP → TLS → 源站 → 数据库），用户感知延迟是链路总和
3. **缓存失效复杂性**：为了缓解重复请求，引入 Redis 做中间缓存，又带来 TTL 管理、缓存击穿、雪崩降级等连锁问题

Redis 缓存本质上是被动模式的补丁——它不改变"请求驱动"的本质，只是在回源路径上加了一层内存缓冲。真正的解法是改变模式本身。

## 三层推送策略

反应式架构按数据特性分层，每层用不同的推送机制：

### 静态层：index.json 资源树 → CDN

能静态化的数据（商品目录、配置、文档）由流计算预计算为 JSON 资源树，推送到 CDN 边缘。客户端通过版本号接口判断是否需要拉取新版本——`if (remote_version > local_version) { download_from_cdn(); }`。

这类似 NixOS 的理念：一个不可变的索引（如 NixOS 的 derivation）描述完整的依赖树，客户端按需拉取。index.json 不是每次请求都查，而是版本号变更后才拉取——大幅减少不必要的流量。

**缓存失效**：CDN 不做主动失效（无法通知全球边缘节点）。解法极简——一个接口返回当前版本号，版本号存在共享内存中读取纳秒级。客户端自己判断是否需要更新，CDN 只做静态文件托管。不需要 Redis 的 TTL 管理。这与 [Lakehouse Catalog](lakehouse-research.md) 同构——都是把版本治理从存储层提升到独立的元数据/控制面，用最简的查询接口替代存储层内部的缓存失效机制。

### 动态层：WebSocket/WebTransport 主动推送

不能静态化的数据（实时库存、用户状态、协作编辑）走 WebSocket/WebTransport 通道，数据变更时主动推送到客户端。客户端不需要刷新，数据自动到达。

**与 HTTP 的真实差异**：现代 HTTP 默认 keep-alive，连接复用层面和 WS 差别不大。但 WS 的核心优势是**连接语义**——HTTP 是请求-响应（客户端发起），WS 是双向流（服务端可随时推送）。没有 keep-alive 的 HTTP 每次要握手，开销更大；WS 一次握手后持续复用，实际开销小得多。

极端情况下（比如一天不刷新），被动模式看似"不请求就没成本"，但现代浏览器的复杂性（广告插件、后台请求、Service Worker）使得"不请求"是理想假设。WS 通道则始终有一条可控的连接，流量可预测。

**WS 的代价**：弱网环境下 WS 长连接不稳定——跨网络切换断线、中间设备超时断连、连接状态管理增加服务端内存开销。WebTransport 解决了这些问题——基于 QUIC，天然支持连接迁移（IP 切换不断连）、多路复用（一个连接承载多个流）、0-RTT 恢复。在移动端和高延迟场景下，WebTransport 是 WS 的演进方向。

**内部延迟的诚实评估**：反应式架构的推送链路是 `CDC 事件 → 流计算处理 → 消息队列 → WS 网关 → 客户端`。流计算和消息队列都会增加内部延迟，可能比直接 HTTP 请求更大。但关键差异在**最后一跳**——WS → 客户端是服务端主动推送，用户体感快（数据变更后立即到达，不需要等用户发起请求）。内部延迟和微服务调用链一样是架构内聚的开销，真正的优势在客户端侧的体感。消息队列的延迟换取的是天然解耦——生产者和消费者不需要互相感知。

### 混合策略

实际系统通常是混合的：能静态化的走 index.json + CDN（低频变更、可接受陈旧度），动态的走 WS 推送（高频变更、需要实时性）。混合策略带来更高的自由度——不需要把所有数据都静态化（过快的版本轮换浪费 CDN 流量），也不需要把所有数据都走 WS（低频数据不值得维持长连接）。

关键判断标准：**数据变更频率 × 客户端数量**。高频 × 多客户端 = WS 推送；低频 × 多客户端 = CDN 静态化；高频 × 单客户端 = 进程内缓存；低频 × 单客户端 = 直接读取，无需缓存。

## 流计算与消息队列：从辅助到核心

在传统架构中，流计算（Flink/RisingWave）和消息队列（Kafka/NATS）是可选的辅助组件——用于实时数仓或异步任务。在反应式架构中，它们成为**核心基础设施**：

- **流计算引擎**（以 [RisingWave](https://risingwave.dev) 为主）：监听数据库变更，预计算聚合结果，推送到 CDN（静态层）或通过消息队列推送到 WS 网关（动态层）。它决定什么数据该推送、何时推送、推送什么形态。注意：流计算始终有运行成本（即使不推送，监听 CDC 也消耗资源），这是反应式架构的固有代价——被动模式在用户不请求时零成本，反应式架构用持续的流计算换取用户请求时的即时响应。
- **消息队列**（Kafka/NATS/Iggy）：连接流计算引擎和 WS 网关的管道。流计算产出变更事件，消息队列保证有序投递到网关，网关再推送到连接的客户端。延迟略增，但换取天然解耦——生产者和消费者不需要互相感知。

**Dapr 的诱惑与拒绝**：Dapr 通过"构建块"抽象屏蔽不同消息队列实现的差异，以 RESTful 方式提供订阅-发布，切换底层实现无需改业务代码。听起来诱人，但代价是：性能惩罚（每个调用经过 Dapr sidecar 的一跳）、抽象层过重（每个构建块是一层 REST API 包装）、以及最关键的——各领域已有事实标准协议（Kafka 的协议、Postgres 的协议、S3 的协议），不需要再在标准之上阉割一遍。Dapr 的价值在于跨云迁移和厂商锁定规避，但在技术栈可控的系统中，直接使用原生协议始终优于抽象层。

移除流计算和消息队列，反应式架构退化为传统被动模式——没有事件源就没有推送的触发。

### 流计算引擎选型

选型不是单选——不同引擎服务不同业务线，可能共存。

**RisingWave（主选）**：流数据库，存储与计算一体。物化视图是可查询的表，`SELECT * FROM mv_xxx` 直接看中间结果，调试体验和普通数据库一样。运维链路短：Flink 方案需要"Flink 计算 + 外部存储存中间状态 + Kafka 做管道"三个组件，RW 方案只需要"RW 计算+存储 + 消息队列做投递管道"两个。

**Flink（重型补充）**：纯计算引擎——中间状态是 opaque 的（TaskManager 内存或 RocksDB），不存储、不可查询，调试几乎是黑盒。但 Flink 的生态成熟度和超大规模处理能力是 RW 尚未达到的。即便选了 Aura/Arroyo/Fluvio 这些更轻量的方案，当业务线扩展到跨系统聚合分析、全局报表、超大规模实时数仓时，Flink 仍可能作为补充引入。这是一条不同的业务线——不是选型替代，而是按规模分层。

**Fluvio（Rust 原生轻量选型）**：纯 Rust 流处理平台，更轻量，适合对二进制体积敏感的嵌入式场景。与 Aura 架构的技术审美一致（Rust + 进程内），但生态和成熟度不如 RW。

**Arroyo（Rust 原生流计算）**：纯 Rust 编写，支持 Rust UDF，Serverless 架构。UDF 能力与 Aura 多语言哲学一致。在传统架构视角下，Arroyo 的短板是连接器不足（无 Postgres 原生 CDC，需走 Debezium；MySQL 支持不成熟；不支持 Pulsar）。但在 Aura/Fluxora 架构上这些短板不存在——事件源是 Fjall 进程内写入（不需要外部数据库 CDC 连接器），消息队列是 Kafka（不需要 Pulsar）。Arroyo 在这个架构下是 RW 的直接替代候选。

### 集成度 → 业务层简化

RW 集成存储后，带来的不只是运维简化，是**业务层架构的简化**。传统架构的业务层做四件事：查数据库 + 查 Redis 缓存 + 管缓存失效 + 序列化/反序列化。RW 集成存储后，业务层直接订阅物化视图变更，不需要中间缓存层——缓存失效、缓存击穿、雪崩这些问题从根上不存在。

这条路线有两个先例：
- **[Fluxora](projects/fluxora-architecture.md)**：UI 直接订阅事件流，无 API URL、无 HTTP 动词。业务层退化为 Kafka 消费者，只处理逻辑不做数据搬运。
- **Meteor 的 DDP 协议**：客户端直接订阅数据库查询的变更。RW 的物化视图 + sink 就是同一思路——客户端订阅的是"查询结果的变更流"，不是"查询请求的响应"。

当集成度足够高时，业务层甚至可以像 Meteor 那样**直接捅到数据库层**——RW 本身就是数据库，物化视图的变更直接作为推送源，不需要中间的业务服务翻译。这和传统三层架构（Web 容器 → 业务层 → 数据库）的"业务层做数据搬运"形成鲜明对比。

**成本的辩证**：流计算的持续运行成本看似高于被动模式，但它消除了 Redis 这个更昂贵的缓存层——Redis 的内存成本比流计算的 CPU 成本高一个数量级（内存 $16.56/GB/月 vs CPU 按核时计费）。整体更划算。更重要的是，**当业务复杂到一定程度，总要上流计算**——不是"要不要"的问题，是"主动用还是被迫用"的问题。不用流计算做预计算和变更检测，就得在应用层手写缓存失效逻辑、定时任务、跨服务状态同步——这些用更昂贵、更不可靠的方式实现了流计算本来要做的事，还少了流计算引擎的容错、重放、精确一次语义。

**响应速度和架构可靠性**：反应式架构在这两个维度上吊打被动模式。响应速度——数据变更后毫秒级推送到客户端，不需要等用户发起请求。架构可靠性——流计算引擎有 checkpoint/replay 机制，消息队列有持久化和至少一次投递，整条链路从 CDC 到客户端都有容错保障。被动模式的 Redis 缓存是单点故障，缓存击穿和雪崩是常态风险。

## 替代路径：Aura 的进程内事件处理

上述架构以 RisingWave 为中心——流计算引擎监听 CDC、维护物化视图、产出变更事件。但还有一条更激进的路径：不依赖外部流处理引擎，直接在分布式存储（Fjall + Openraft）上架设事件处理逻辑。

[Aura 架构](aura-architecture.md) 就走这条路。核心思路：

- **分布式存储即事件源**：Fjall + Openraft 本身就是分布式强一致存储，每次写入都经过 Raft 共识——写入即事件，WAL 即事件流。不需要外部 CDC 管道去监听另一个数据库的变更。
- **嵌入式脚本即事件处理器**：框架内置多种语言运行时，每个 Actor 选一种脚本语言。语言用途无硬限制，以下仅为建议：Steel Lisp 适合确定性策略逻辑（权限、路由、状态转换），Python（PyO3）适合业务逻辑，WASM 适合引入其他语言的重型插件（沙箱隔离）。语言接入也无硬限制，任何能通过 PyO3 或 WASM 接入的语言都可以。RPC 调用外部服务不内置支持也不阻止——用户可以在 Python 侧用 httpx 等库自行发起，但框架不提供基础设施。详见 [Aura 架构](aura-architecture.md)。
- **零网络跳数**：RisingWave 方案的链路是 `PostgreSQL CDC → RW 计算 → 消息队列 → WS 网关 → 客户端`（四跳）。Aura 方案是 `Fjall 写入 → Steel Lisp 处理 → WS 网关 → 客户端`（三跳），且前三跳全在进程内，无网络开销。

**与 RW 路线的分工**：

| 维度 | RisingWave 路线 | Aura 路线 |
|:--|:--|:--|
| **事件源** | 外部数据库 CDC | Fjall 写入即事件（内建） |
| **事件处理** | RW 物化视图（SQL 声明式） | 多语言分层（Lisp 策略 + Python 业务 + WASM 插件） |
| **中间状态** | RW 存储（可查询） | Actor 进程内（0 网络） |
| **网络跳数** | 四跳（含两次跨进程） | 三跳（仅最后一跳跨网络） |
| **适用规模** | 大规模（多数据源、跨团队） | 单系统内（Actor 自治） |
| **表达能力** | SQL（声明式，受限但可审计） | Lisp（图灵完备，灵活但需约束） |

**Aura 路线的约束**：事件处理逻辑在 Steel Lisp 沙箱内运行，确定性要求高——同一输入必须产出同一输出（类似智能合约的确定性约束）。这限制了不能在脚本内做网络调用、随机数、时间戳等非确定性操作。RW 路线没有这个约束，因为 SQL 物化视图本身就是声明式的。

**设计原则：全事件，无指令**：架构中持久化和重放的只有事件（已发生的事实），没有指令（要求执行的操作）。指令不可安全重放——"转账 100 元"重放三次是灾难；事件可安全重放——"账户余额变为 X"重放一百次结果一样。Raft 共识日志本身就这么做：log 里存的是 committed entries（已被多数派确认的事实），不是"请执行"的指令。这与 Actor 模型的 let it crash 哲学一致——Actor 崩溃后从持久化的事件日志重放恢复状态，未完成的操作由客户端超时重试，服务端从最后一个一致状态恢复。不需要指令的持久化，也不需要分布式事务的回滚机制。

**混合可能**：两路线不互斥。Aura 处理进程内的高频状态事件（购物车、会话、实时库存），RW 处理跨系统的聚合分析（全局报表、跨服务物化视图）。Aura 产生的 WAL 也可以作为 RW 的 CDC 源——Aura 负责快路径，RW 负责慢路径。

## Fluxora：反应式架构的激进实践

[Fluxora](projects/fluxora-architecture.md) 是事件驱动 UI 框架（Rust/Dioxus），在 UI 和协议层比上述架构更激进：

- **UI 层**：组件绑定到事件名称，无 API URL、无 HTTP 动词。UI 不是"请求-渲染"的循环，而是"订阅-接收-渲染"的事件流。
- **协议层**：WebSocket 作为唯一通信通道，Kafka/Iggy 消息队列连接业务服务和网关。所有通信都是事件流，无 REST 轮询。
- **架构拓扑**：`UI (Dioxus/WASM) ↔ Gateway (Axum/WS) ↔ 业务服务 (Kafka 消费者)`。业务服务彼此透明——analysis 可以拦截 chat 流而 chat 不知。

Fluxora 的激进之处在于它**从一开始就拒绝 HTTP 请求-响应模型**。不是在传统架构上加 WS 通道，而是把 WS 作为唯一通道，从协议层到 UI 层完全事件驱动。但 Fluxora 的 Brick JSON 树本身仍是静态资源——框架的静态部分（组件定义、类型枚举）走文件加载，业务数据走事件流。这不是否定混合策略，而是把混合策略的边界推到了协议层。

**Fluxora vs 哑网关**：传统的 WS 推送方案（如 nchan）是哑网关——只做消息透传，没有定制化逻辑。问题是：消息队列不能直接怼到推送网关上，中间还需要一层消费者做分发逻辑，这又抵消了消息队列的可靠性保证。Fluxora 的网关就是这层消费者——它在网关层实现了消息分发、频道路由、事件类型解析。在 Aura 架构上，这条路径可以更激进：数据变更直接从 Fjall 写入触发 Steel Lisp 处理，经 WS 网关推送到客户端，类似 Meteor 的 DDP 协议——数据库到 UI 直通，中间不需要独立的业务服务层。前提是 Aura 的进程内事件处理足够强大，能承担 Fluxora 网关的角色。

## 架构对比

| 维度 | 被动模式（传统） | 反应式架构 |
|:--|:--|:--|
| **数据到达方式** | 客户端拉取 | 服务端推送 |
| **无变更时流量** | 重复请求（浪费） | 零推送流量（但流计算持续运行） |
| **延迟** | 完整链路总和 | 内部链路略长但体感快（最后一跳即时到达） |
| **缓存层** | Redis 中间缓存（补丁） | CDN + WS（原生） |
| **失效机制** | TTL + 手动清除（复杂） | 版本号接口 + 事件驱动（简单） |
| **流计算角色** | 可选辅助 | 核心基础设施 |
| **消息队列角色** | 可选辅助 | 核心基础设施 |
| **客户端体验** | 需要刷新/轮询 | 数据自动到达 |
| **可靠性** | Redis 单点故障，缓存击穿/雪崩风险 | 全链路容错（checkpoint/replay/持久化） |
| **复杂业务适应性** | 手写缓存失效/定时任务/状态同步 | 流计算原生支持，复杂度内聚在引擎层 |
| **集成度** | Web 容器 + 业务层 + Redis + 数据库（四层） | RW（计算+存储）+ 消息队列 + WS 网关（两层） |
| **调试** | Redis 缓存命中/失效难追踪 | RW 物化视图直接 SELECT 查询中间结果 |
| **业务层角色** | 数据搬运（查库+查缓存+管失效+序列化） | 逻辑处理（订阅物化视图变更，无缓存层） |
